## Next.js 2.0 以降でのルーティング構造の変革
Next.js 2.0 以降では、従来の `pages/` ベースのルーティング（Pages Router）が非推奨となり、`app/` ベースのルーティング（App Router）が標準化されました。この変更により、初心者はルーティングの基本概念を再学習する必要が生まれます。特に、以下の点が混乱の原因になります：
- ファイルベースのルート定義方法の変化
- ネストされたルートの表現方法
- ルート同士の関係性の表現

## App Router の基本構造と利点
App Router では、ルートは `app/` 内のファイル名によって自動的に定義されます。以下の例は、基本的なルート構造を示しています：
```jsx
// app/page.js
export default function Home() {
  return <h1>Home Page</h1>;
}

// app/about/page.js
export default function About() {
  return <h1>About Page</h1>;
}
```
この構造の利点は、以下の通りです：
1. ファイル名から直感的にルートを理解できる
2. ルートのネストが自然に表現可能（例: `app/products/[id]/page.js`）
3. ステート管理やデータフェッチングがコンポーネント内で統一可能

## サーバーコンポーネント vs クライアントコンポーネントの役割分担
Next.js 13 以降では、コンポーネントの実行環境（サーバーかクライアント）を指定できるようになりました。この区別に混乱する点は、以下の3つです：

### サーバーコンポーネентの特徴
```jsx
// server component
export default function ServerComponent() {
  // サーバーで実行される処理（例：データベースクエリ）
  const data = fetchDataFromDB();
  return <div>{data}</div>;
}
```
特徴：
- 最初のレンダリングでサーバーで処理される
- クライアント側の状態管理は必要ない
- SEOやパフォーマンス向上に有効

### クライアントコンポーネントの特徴
```jsx
// client component
export default function ClientComponent() {
  // クライアントで実行される処理（例：フォーム操作）
  const [count, setCount] = useState(0);
  return (
    <div>
      <button onClick={() => setCount(count + 1)}>Count: {count}</button>
    </div>
  );
}
```
特徴：
- 初期レンダリングはサーバーで行われるが、サブsequentな更新はクライアントで処理される
- UIのインタラクティブ性を実現するのに適している
- サーバー負荷を軽減できる

## データフェッチングのルーティングとの連携
App Router では、データフェッチングはコンポーネント内で直接行うか、または `load` 関数を使用します。以下は具体的な例です：

### load 関数を使ったデータフェッチ
```jsx
// app/products/load.js
export async function load() {
  const res = await fetch('https://api.example.com/products');
  return res.json();
}

// app/products/page.js
export default function Products({ products }) {
  return <ul>{products.map(p => <li>{p.name}</li>)}</ul>;
}
```
このパターンでは：
- `load` 関数がサーバーで実行され、クライアントにデータを渡す
- 複雑なロジックは `load` で統一するのが推奨される

## よくある混乱ポイントと対策
### 1. Pages Router との混同
既存の `pages/` プロジェクトを App Router に変換する際、以下の点に注意：
- ファイル構造を `app/` に移行する必要がある
- `getServerSideProps` は `load` に置き換える
- ルートのネスト表現方法が変更される

### 2. サーバーコンポーネントの制限
サーバーコンポーネントでは以下ができない：
- `useEffect` などのクライアント専用ハookの使用
- クライアント側の状態管理（React Hooks による状態管理）

### 3. ルートの再利用性
App Router では、コンポーネントを複数のルートで再利用する場合、以下のパターンが推奨されます：
```jsx
// app/layout.js
export default function Layout({ children }) {
  return <div>{children}</div>;
}

// app/products/page.js
export default function Products() {
  return <Layout><div>Products Content</div></Layout>;
}
```

## まとめ
Next.js の App Router は、ルーティング構造やコンポーネントの振る舞いを一新したものです。初期導入時には以下の流れを意識してください：
1. ファイル名からルートを定義する構造を理解
2. サーバーコンポーネントとクライアントコンポーネントの役割を明確化
3. データフェッチングは `load` 関数やコンポーネント内で統一する

特に、既存の Projects Router プロジェクトを移行する場合や、サーバーコンポーネントの制限に注意することが重要です。今後の Next.js の進化に備え、これらのルールをしっかり押さえておくことで、スムーズな開発が可能になります。
