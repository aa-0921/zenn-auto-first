### 指摘の受け止め方：「問題」を「学び」に変える技術

最初に大切なのは、指摘を「自分への攻撃」と捉えないことです。ある時、先輩から「この実装だとパフォーマンスが悪いよ」と指摘された時があります。最初は「そんなの知らなかったし、私の実力が足りないのかな」と嘆いていましたが、その後自分で調べてみると、確かにその指摘は正しかったことがわかりました。

指摘をうまく受け止めるために私が実践している方法を紹介します：

1. **指摘をそのまま受け入れる**: まずは「そのとおりです」と素直に受け入れる
2. **質問して理解を深める**: 「どういう場合にパフォーマンスが問題になるのか教えてもらえますか？」
3. **自分で調べてアクションを起こす**: 指摘された内容について勉強し、次のレビューで改善を見せる

```javascript
// 悪い例: 指摘されただけで終わる
const processData = (data) => {
  return data.filter(item => item.active).map(item => item.name);
};

// 良い例: 指摘を受けて改善を適用
const processData = (data) => {
  // 指摘事項: 配列のメソッドチェーンを避ける
  const result = [];
  for (const item of data) {
    if (item.active) {
      result.push(item.name);
    }
  }
  return result;
};
```

### 質問の仕方：建設的な対話を促すコツ

質問は単なる「わからない」を相づちするだけでなく、チーム全体の理解を深めるツールです。レビューでの質問は次のようなパターンが効果的です：

- **背景を知る**: 「なぜこのアプローチを選んだのか教えてもらえますか？」
- **他の選択肢を探る**: 「他に実装方法はないのかな？」
- **学びを共有する**: 「このパターンは他でも使えるのかな？」

実際のSlackでのやりとりを例にすると：

```
レビュワー: 「このクエリのN+1問題解決策として、preloadを使った方がいいですよ」
自分: 「preloadの方が良いんですか？ どういうメリットがあるのでしょうか？ 
       また、他のテーブルでも同様の問題が起きそうなのですが、
       そちらも同じ対応でいいのでしょうか？」
```

このように質問をすると、単純な指摘だけでなく、その理由や応用まで学ぶことができます。

### レビューを成長の機会にするコツ

コードレビューを成長の機会にするために、私が実践している3つの習慣があります：

**1. 毎回のレビューで必ず1つはメモを取る**

レビューで指摘された内容や新しい発見をノートに記録しています。後で自分のコードで活用できるかもしれませんし、チーム内でのナレッジとして共有することもできます。

**2. 指摘されたパターンを自分のプロジェクトに適用してみる**

レビューで学んだことは、必ず自分の別のプロジェクトで試してみます。たとえそのプロジェクトで問題がなかったとしても、「こんなやり方があるのか」という発見が成長につながります。

**3. レビューを振り返ってフィードバックをする**

月に1度程度、先月にレビューで学んだことをまとめて、チームにフィードバックしています。「先月レビューで学んだ〜というパターンを実践した結果、〜という改善が見られました」という具合です。

## メンタルを保つための心構え

コードレビューは技術的な学びだけでなく、メンタル管理も重要な要素です。指摘を受けるたびに落ち込むのではなく、次のような心構えを持つように努めています：

- **指摘は個人攻撃ではない**: コードの品質を上げるためのものであることを意識する
- **完璧を目指すのではない**: 誰も完璧なコードを書けないので、改善の過程を楽しむ
- **成長の証拠を残す**: レビューでの指摘を記録し、時間が経つにつれて成長を実感する

## まとめ

コードレビューはエンジニアとして成長する上で欠かせないプロセスです。指摘をうまく受け止め、建設的な質問をし、学びを実践に移すことで、レビューをただの「指摘の場」から「成長の機会」に変えることができます。

また、メンタルを保つためには、指摘を攻撃と捉えず、常に学びの心持で臨むことが大切です。次回レビューがあれば、「この指摘は何を教えてくれるのか」「どうやって成長につなげられるのか」を考えながら臨んでみてください。
