## はじめに：なぜ今 Git の基本を見直すのか

「Git はなんとなく使えているけど、コンフリクトが起きると毎回焦る」「`pull` と `fetch` の違いをうまく説明できない」——そんな駆け出しエンジニアの声を現場でよく耳にします。モダンな開発現場では Git がインフラ同然に存在していますが、GUI ツールや IDE のボタンに頼りきりで、裏側で何が起きているか理解せずに運用しているケースも少なくありません。

本記事では、**「現場で最低限これだけは押さえておきたい」**という観点で、リポジトリのクローンからプッシュまでの標準的なワークフロー、ブランチ運用の勘所、そして避けて通れないコンフリクト発生時の冷静な対処法までを、実務で使うコマンド例とともに体系的に解説します。この記事を読めば、明日すぐに自信を持って `git commit` できるようになるはずです。

## 前提知識と環境設定：まずは「自分」を Git に教える

コマンドを打つ前に、絶対にやっておくべき初期設定があります。コミットログに残る「作者情報」の設定です。これをサボると、後で `git log` を見たときに `Author: unknown` になってしまい、チームメンバーから「誰これ？」と思われます。

```bash
# グローバル設定（自分の PC 全体で有効）
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# エディタの設定（VS Code を使う場合の例）
git config --global core.editor "code --wait"

# デフォルトブランチ名を main に統一（最近の標準）
git config --global init.defaultBranch main
```

また、日本語ファイル名が文字化けしないよう、以下の設定も入れておくと安心です。

```bash
git config --global core.quotepath false
```

これらは `~/.gitconfig` に書き込まれます。`git config --list` で確認できますので、最初に一度だけ実行しておきましょう。

## 実践ワークフロー：clone → branch → commit → push の「型」を身につける

現場での標準的な開発サイクルは、**「最新の main を取り込む → 作業用ブランチを切る → 小さくコミットする → リモートに上げる → プルリクエストを出す」**という流れです。これをコマンドレベルで確実に回せるようになりましょう。

### 1. リポジトリをローカルに持ってくる：`git clone`

```bash
# HTTPS の場合（初心者はこちらから）
git clone https://github.com/owner/repo.git

# SSH の場合（鍵設定済みならこちらが楽）
git clone git@github.com:owner/repo.git
```

クローン直後は `main` ブランチ（または `master`）にいます。まずはリモートの最新状態を確認する癖をつけましょう。

```bash
cd repo
git branch -a  # リモートブランチも含めて確認
```

### 2. 作業を始める前に：`git pull` で最新化する

「自分のローカルが古いまま作業を始めて、後で大量のコンフリクトを解消する羽目になった」はあるある失敗です。作業開始前には必ず `main` を最新にします。

```bash
git switch main        # main ブランチへ移動（checkout でも可）
git pull origin main   # リモートの main を取り込む
```

> **ポイント**: `git pull` は内部で `git fetch` + `git merge` をやっています。`fetch` だけで「リモートの状況だけ確認したい」ときは `git fetch origin` と打ちます。慣れてきたら使い分けると安全です。

### 3. 作業用ブランチを切る：`git switch -c` または `git checkout -b`

機能追加やバグ修正ごとにブランチを切るのが鉄則です。ブランチ名はチームの命名規則に従いましょう（例: `feature/login-form`, `fix/typo-readme`, `refactor/api-client`）。

```bash
# 新しいブランチを作成して即座に切り替え
git switch -c feature/add-user-profile
# 古い書き方: git checkout -b feature/add-user-profile
```

`git switch` はブランチ切り替え専用コマンドとして Git 2.23 で導入されました。`checkout` はファイル復元も兼ねており誤操作のリスクがあるため、ブランチ操作には `switch` を使うのがモダンで安全です。

### 4. 変更をステージングしてコミットする：`add` → `commit`

「なんとなく `git add .` で全部入れて `git commit -m "fix"`」は**NG**です。コミットは**「論理的にひとまとまりの変更」**単位で行い、メッセージで意図を伝えます。

```bash
# 変更されたファイルを確認
git status

# 差分を確認（何が変わったか目で見て確認）
git diff

# 関連するファイルだけをステージング
git add src/user/profile.ts tests/user/profile.test.ts

# コミット（エディタが開くので、件名・本文を書く）
git commit
```

#### 良いコミットメッセージの書き方（Conventional Commits 風）

```
feat(profile): ユーザープロフィール画面の表示機能を追加

- API からユーザー情報を取得して表示するロジックを実装
- ローディング状態とエラー状態のハンドリングを追加
- 既存の UserCard コンポーネントを再利用

Refs: #123
```

- **件名**: `タイプ(スコープ): 短い説明`（50文字以内、命令形、末尾にピリオドなし）
- **本文**: なぜこの変更が必要か、何をどう変えたかを補足
- **フッター**: 関連 Issue 番号や Breaking Change 情報

### 5. リモートにプッシュしてプルリクエストを作る：`git push`

```bash
# 初回プッシュ時は -u で上流ブランチを設定
git push -u origin feature/add-user-profile

# 2回目以降
git push
```

これで GitHub/GitLab 上で **Pull Request (Merge Request)** を作成できます。レビューを受け、CI が通れば `main` へマージです。

## 避けて通れない壁：コンフリクト（競合）を冷静に解消する

複数人が同じファイルを編集すると、マージやリベース時に**コンフリクト**が発生します。慌てず、以下の手順で対処すれば怖くありません。

### コンフリクトが発生する典型的なシナリオ

1. `main` が更新された後、自分のブランチで `git merge main` または `git rebase main` をした
2. 同僚が同じファイルの近い行を編集して先にマージしていた
3. `git pull` した際にリモートとローカルで同じ箇所が変わっていた

### 解消手順：ステップバイステップ

#### 1. 状況を把握する

```bash
git status
# Unmerged paths と赤字で表示されるファイルがコンフリクト中
```

#### 2. コンフリクト箇所を確認・編集する

対象ファイルを開くと、以下のようなマーカーが挿入されています。

```typescript
// 例: src/user/profile.ts
function fetchUser(id: string) {
<<<<<<< HEAD
  // あなたのローカルでの変更（現在のブランチ）
  return api.get(`/users/${id}?include=posts`);
=======
  // リモート（main など）での変更
  return api.get(`/users/${id}?include=posts,comments`);
>>>>>>> main
}
```

**`<<<<<<< HEAD` 〜 `=======`** が自分の変更、**`=======` 〜 `>>>>>>> main`** が取り込み側の変更です。どちらを採用するか、あるいは両方を融合させるか判断し、マーカーを削除して正しいコードに書き換えます。

```typescript
// 解消後：両方の要件を満たすように統合
function fetchUser(id: string) {
  return api.get(`/users/${id}?include=posts,comments,likes`);
}
```

#### 3. 解消したらステージングしてコミット（マージの場合）または続行（リベースの場合）

**マージ (`git merge`) でコンフリクトした場合:**

```bash
git add src/user/profile.ts
git commit
# エディタが開く。デフォルトのマージコミットメッセージで OK なら保存して閉じる
```

**リベース (`git rebase`) でコンフリクトした場合:**

```bash
git add src/user/profile.ts
git rebase --continue
# 次のコミットでもコンフリクトすれば繰り返し。全部解消したら完了
```

> **現場のヒント**: VS Code などのエディタを使っているなら、「マージ エディタ」や「インライン解決」UI を活用すると、左右の差分を見ながらボタン一つで「採用/却下/統合」ができ、マーカーの手動削除ミスを防げます。

#### 4. どうしても無理なら中止する

「やっぱり今のブランチでやり直したい」「間違えて変なマージを始めた」というときは、潔く中止しましょう。

```bash
# マージ中止
git merge --abort

# リベース中止
git rebase --abort
```

これでコンフリクト発生直前の状態に完全に戻れます。

## 現場で差がつく「もう一歩進んだ」Tips

基本フローをマスターしたら、以下のコマンドも武器に加えてください。

### `git stash`：一時避難で手を空ける

「今の作業を中断して `main` でホットフィックスを当てたい」というとき、中途半端なコミットを作らずに作業内容を隠せます。

```bash
git stash push -m "WIP: プロフィール画面のスタイル調整中"
# 作業ディレクトリが綺麗になる

git switch main
git pull origin main
# ホットフィックス作業...

git switch feature/add-user-profile
git stash pop  # 一時避難した変更を戻す
```

### `git commit --amend`：直前のコミットをやり直す

「コミットメッセージにタイポがあった」「ファイル追加し忘れた」とき、履歴を汚さず修正できます（**リモートに push していないローカルコミット限定**）。

```bash
# ファイルを追加してから
git add forgotten-file.ts

# 直前のコミットに上書き（メッセージも編集可能）
git commit --amend
```

### `git log --oneline --graph --all`：ブランチ構造を可視化

今、どのブランチがどこまで進んでいて、どう分岐・合流しているかを一目で把握できます。エイリアスを登録しておくと便利です。

```bash
git config --global alias.graph "log --oneline --graph --all --decorate"
# 以後 git graph で実行可能
```

## まとめ：コマンドの「意味」を理解し、恐れずに使いこなす

本記事では、以下のポイントを押さえました。

1.  **初期設定** で「誰がやったか」を明確にする
2.  **`clone` → `pull` → `switch -c` → `add`/`commit` → `push`** の標準フローを体で覚える
3.  **コミット粒度とメッセージ** でレビューアー（未来の自分含む）へ思いやりを持つ
4.  **コンフリクト** は `status` → ファイル編集 → `add` → `commit/continue` の定石で冷静に解消する
5.  **`stash`, `amend`, `graph`** などの便利コマンドで作業効率を上げる

Git は「取り返しがつかない」と思われがちですが、`reflog` がある限り、ほぼあらゆる操作は元に戻せます。まずはローカルリポジトリで意図的にコンフリクトを起こして解消する練習をしてみてください。失敗を恐れず手を動かし、「なぜこのコマンドを打つのか」を言語化できるようになれば、Git はあなたの開発を支える最強の相棒になります。

明日からの開発で、ぜひ自信を持って `git commit` を打ってください。
