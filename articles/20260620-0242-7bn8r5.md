## Dockerとは何か ― イメージとコンテナの違い
### イメージ
- **不変（immutable）**: 一度作成したイメージは変更されない。  
- **レイヤー構造**: Dockerfile の各命令 (`FROM`, `RUN`, `COPY` など) がレイヤーとして積み重ねられ、重複レイヤーはキャッシュされる。  
- **再現性**: 同じ Dockerfile ならどこでも同じイメージが生成できる。  
- **イメージの保存場所**: Docker Hub、Google Container Registry、GitHub Container Registry などにプッシュ可能。

### コンテナ
- **実行中のプロセス**: イメージを起動した状態。  
- **一時的**: コンテナは停止するとデータは失われる（永続化はボリュームで行う）。  
- **軽量**: 同じホスト上で複数のコンテナを同一イメージから起動できる。  
- **ネットワーク**: デフォルトでブリッジネットワークに接続され、他のコンテナと通信可能。

> **ポイント**  
> - **イメージ**は「設計図」、**コンテナ**は「設計図から作られた実体」です。  
> - コンテナはイメージのスナップショットを実行したもの。  
> - 開発時はイメージを頻繁にビルドし、コンテナでローカル環境を再現します。

## Dockerfile 基礎
まずは簡単な Node.js アプリ用の Dockerfile を作成します。  
```Dockerfile
# 1. ベースイメージ
FROM node:18-alpine

# 2. 作業ディレクトリを作成
WORKDIR /app

# 3. 依存パッケージのインストール
COPY package*.json ./
RUN npm ci --only=production

# 4. アプリケーションコードをコピー
COPY . .

# 5. ポートを公開
EXPOSE 3000

# 6. アプリ起動
CMD ["node", "app.js"]
```
- **ベースイメージ**は軽量な Alpine Linux をベースにしているため、イメージサイズが小さくなります。  
- `WORKDIR` を設定することで、以後の命令はそのディレクトリ内で実行されます。  
- `npm ci` は `node_modules` をクリーンにインストールし、ビルド時間を短縮します。

## docker-compose で開発環境を一括管理
実務では複数のサービス（Web、DB、キャッシュなど）を同時に起動するケースが多いです。`docker-compose` を使うと、`docker-compose.yml` で一括管理できます。

### 例：Node.js + PostgreSQL
```yaml
version: "3.9"

services:
  app:
    build: .
    ports:
      - "3000:3000"
    volumes:
      # ソースコードをホストと同期
      - .:/app
      - /app/node_modules
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgres://postgres:password@db:5432/appdb
    depends_on:
      - db

  db:
    image: postgres:15-alpine
    restart: always
    environment:
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: password
      POSTGRES_DB: appdb
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
```
#### ポイント解説
- **`build: .`** で Dockerfile をビルド。  
- **`volumes`** にホストのソースコードをマウント。ソース変更を即座に反映。  
- **`depends_on`** で起動順序を保証。  
- **永続ボリューム** `db-data` で DB データを永続化。

#### 起動コマンド
```bash
# 1. 依存イメージ取得 + ビルド
docker compose build

# 2. コンテナ起動
docker compose up
```
`docker compose up -d` でデタッチモード（バックグラウンド）起動も可能です。

## 開発時に Docker を使うメリット
1. **環境差異の解消**  
   - 本番と同じイメージをローカルで走らせることで、環境差異（OSバージョン、ライブラリバージョン）を減らせる。  
2. **依存関係の一元管理**  
   - Dockerfile で全ての依存を宣言するため、`npm install` を忘れたりパッケージのバージョンズレが起こりにくい。  
3. **チーム全体の共通環境**  
   - `docker-compose.yml` を共有すれば、同じコマンドで同一環境を再現できる。  
4. **CI/CD との連携が容易**  
   - CI パイプラインで同じイメージをビルド・テストに使える。  
5. **リソースの節約**  
   - 仮想マシンより軽量で、複数コンテナを同時に実行してもメモリ・CPU が効率的。

> **注意点**  
> - デバッグ時はホストのデバッグツールとコンテナのツールを併用することが多い。  
> - ボリュームマッピングは開発速度を上げるが、本番環境では慎重に設定する。

## 実務での落とし穴と対策
| 見出し | 内容 | 対策 |
|--------|------|------|
| `COPY` vs `ADD` | `ADD` は URL から自動解凍もできるが、意図しない挙動がある。 | 基本的には `COPY` を使う。 |
| キャッシュの失敗 | 依存ファイル変更で `RUN npm ci` が再実行されない場合がある。 | `COPY package*.json ./` の後に `RUN npm ci` を配置し、依存ファイル変更時だけ再実行させる。 |
| ボリューム競合 | 同じボリュームを複数コンテナで共有すると競合が起こる。 | 必要に応じて `named volume` を分ける。 |
| ネットワーク設定 | `depends_on` だけでは起動順序保証。 | `healthcheck` を設定して、サービスが利用可能になるまで待機させる。 |

## まとめ
- **イメージ**は変更不可の設計図、**コンテナ**はその実行体。  
- Dockerfile でイメージを作成し、`docker-compose` で複数サービスを一括管理することで、ローカル開発環境を本番に近い状態で再現できる。  
- 開発時に Docker を使うことで環境差異が減り、チームの開発効率と品質が向上する。  
- 実務では依存管理、ボリューム設計、ネットワーク設定に注意し、継続的に改善していくことが重要です。

これで Docker を使った開発環境構築の基本が掴めたはずです。次のステップは、実際に自分のプロジェクトに Docker を組み込み、CI/CD パイプラインへ落とし込むことです。ぜひ挑戦してみてください！
