## 背景・前提
- **対象**: 駆け出し〜中級エンジニア
- **言語**: 主にJavaScript/TypeScriptを想定
- **前提知識**: Gitの基本操作、CI/CDの流れ、基本的なテストフレームワーク（Jest, Mocha）を理解していること
- **レビュー環境**: GitHub Pull Request（PR）ベースのコードレビューを想定

### 私のレビュー体験
- **最初のレビュー**: 10行程度のユーティリティ関数に対して「変数名が不明瞭」「コメント不足」「テストが無い」指摘
- **中盤のレビュー**: 複数ファイルにまたがるリファクタリングで「依存関係が多すぎる」「関数が長い」指摘
- **現在**: コードレビューを受けるたびに「何をどう改善すればいいか」「レビューを通じて何を学べるか」を意識している

## 本文

### 1. 指摘を「攻撃」ではなく「情報源」と捉える
最初の指摘は「自分のコードがダメだ」と感じやすいですが、実際は「改善点を共有するための情報」です。以下のようにフレーム化してみましょう。

```text
指摘 = 改善点 + 具体例 + 期待される結果
```

#### 例
- 指摘: 「変数名が不明瞭です」
- 具体例: `const a = 10;`
- 期待される結果: `const userAge = 10;`

このように指摘を分解することで、攻撃的に感じずに「どうすればよいか」を明確にできます。

### 2. 質問の仕方を「疑問→仮説→提案」に変える
指摘を受けたときにすぐに「何が悪いの？」と聞くのではなく、疑問を自分で整理し、仮説を立ててから質問すると、レビューアも答えやすく、議論がスムーズになります。

| ステップ | 例 |
|----------|----|
| 疑問 | 「この変数名が不明瞭だと指摘されましたが、具体的に何を想定しているのでしょうか？」 |
| 仮説 | 「ユーザーの年齢を保持している想定ですか？」 |
| 提案 | 「もし年齢であれば `userAge` という名前に変更し、コメントを追加します。」 |

#### コード例
```js
// 変更前
const a = getUserInfo().age;

// 変更後
/**
 * ユーザーの年齢を取得します。
 * @returns {number} ユーザーの年齢
 */
const userAge = getUserInfo().age;
```

### 3. レビューを「学習の機会」として管理する
レビューで指摘された箇所を「改善タスク」としてタスク管理ツールに追加し、次のレビュー時に進捗を確認します。これにより、レビューを単なる修正作業ではなく、継続的な学習プロセスに変えられます。

#### 実践例
1. PRコメントで指摘された箇所をタスク化
2. タスクに「学習目標」を設定（例: 「ESLintのルールを理解し、徹底的に適用する」）
3. 次回PRで「学習結果」を記載し、レビューを受ける

### 4. 事前準備でレビューの質を上げる
- **リファクタリング**: コードが長くなる前に小さな単位でリファクタリング
- **テスト**: 変更前後でテストを書き、動作保証
- **ドキュメント**: 関数の入力・出力をコメントで明確化

こうした準備を行うことで、レビューアは「コードの意図」を理解しやすくなり、フィードバックが具体的かつ建設的になります。

### 5. メンタルを保つためのセルフケア
- **レビュー後の振り返り**: 1日終わったら「何がうまくいったか」「何が難しかったか」をメモ
- **ポジティブフィードバック**: 自分が改善した点を自分で評価し、次のレビューで自信を持つ
- **休息**: 長時間のレビュー作業は集中力を低下させるため、30分ごとに短い休憩を取る

## まとめ
コードレビューは「指摘される」だけでなく、「学び、成長する」機会です。指摘を情報源と捉え、質問を構造化し、レビューを学習タスクとして管理することで、コード品質と個人スキルを同時に向上させることができます。また、事前準備とセルフケアを徹底すれば、メンタル面での負担も軽減できます。次にコードレビューに臨むときは、この記事のポイントを意識して「レビューを成長のチャンス」に変えてみてください。
