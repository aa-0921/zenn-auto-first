## 背景・前提
### Pages Router
- `pages` ディレクトリ以下にファイルを置くだけでルーティングが自動生成されます。
- `getStaticProps` や `getServerSideProps` でデータ取得が可能。
- クライアントコンポーネントは `useEffect` や `useState` で自前管理。

### App Router
- `app` ディレクトリ以下にファイルを置き、**セグメント**（`(group)` や `[[...slug]]`）でルーティングを制御します。
- データ取得はコンポーネント自体が **Server Component** として動作し、`fetch` や `generateStaticParams` が標準。
- クライアント側が必要な部分は **Client Component** として明示的にマークします。

## 本文
### 1. ディレクトリ構造の違い
```tsx
// Pages Router (pages/index.tsx)
export default function Home() {
  return <h1>Pages Router</h1>;
}
```

```tsx
// App Router (app/page.tsx)
export default function Home() {
  return <h1>App Router</h1>;
}
```
- **Pages Router** は `pages` ディレクトリ直下のファイル名がルートとなります。
- **App Router** は `app` ディレクトリ内に `page.tsx` が入り口となり、ネストしたルートはフォルダ構造で表現します。  
  例: `app/blog/post/[slug]/page.tsx` は `/blog/post/:slug` のパスを処理。

### 2. Server Component と Client Component
Server Component は **サーバー側で描画**され、HTML に直接組み込まれます。データ取得は非同期でも問題なく行えます。

```tsx
// Server Component (app/user/page.tsx)
export default async function UserPage({ params }) {
  const res = await fetch(`https://api.example.com/users/${params.id}`);
  const user = await res.json();

  return (
    <div>
      <h2>{user.name}</h2>
      <p>{user.bio}</p>
    </div>
  );
}
```

一方、クライアント側の状態やブラウザ API を使う場合は **Client Component** としてマークします。

```tsx
// Client Component (app/counter/counter.tsx)
'use client';

import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);
  return (
    <button onClick={() => setCount(c => c + 1)}>
      Clicked {count} times
    </button>
  );
}
```

### 3. データ取得のベストプラクティス- **App Router** では、`fetch` をトップレベルで行うだけで自動的に **Incremental Static Regeneration (ISR)** が有効になります。  
  ```tsx
  export async function generateStaticParams() {
    const res = await fetch('https://api.example.com/posts');
    const posts = await res.json();
    return posts.map(p => p.id);
  }
  ```
- **Pages Router** で同様の機能を得るには `getStaticProps` を使用しますが、App Router では `generateStaticParams` がよりシンプルです。

### 4. エラーハンドリングと注意点
| トラブル | 原因 | 対策 |
|----------|------|------|
| `Error: Cannot read property 'params' of undefined` | `params` が動的に取得できていない | `generateStaticParams` が正しく返っているか確認 |
| `Server Component cannot use useState` | クライアント側のフックを Server Component に入れ込んでいる | `use client` ディレクティブを付与、または別ファイルに切り出す |
| `Route not found` | `app` ディレクトリに `page.tsx` がない、または名前が `index.tsx` でない | ルートファイルを `page.tsx` として配置、または `export const dynamic = 'force-dynamic'` で動的化 |

## まとめ
- **App Router** はディレクトリベースのネストが強みですが、**Pages Router** との混同が初心者の足かせになりがちです。  
- Server Component と Client Component の境界を明確にし、**'use client'** を適切に付与することが、予期しないエラー回避の鍵です。  
- データ取得はコンポーネントレベルで非同期処理を実装し、**generateStaticParams** や **fetch** の使い方をマスターすれば、ISR や SSR の恩恵を最大限に活かせます。  
- 最後に、プロジェクトを始めるときは **`next dev`** でローカル環境を確認し、**`next build`** で本番ビルド時にエラーが出ないか必ずチェックしましょう。
