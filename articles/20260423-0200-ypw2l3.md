## 2. 背景・前提

### 2.1 REST APIとは？

- **REST**（Representational State Transfer）は、Webの設計原則に基づくアーキテクチャスタイルです。  
- クライアントはリソース（例：`/users/1`）に対してHTTPメソッドを使って操作を行い、サーバはステータスコードとリソース表現（JSONなど）で応答します。

### 2.2 HTTPメソッドと典型的な使い分け

| メソッド | 意味 | 典型的な用途 |
|----------|------|--------------|
| GET | 取得 | `GET /users` |
| POST | 作成 | `POST /users` |
| PUT | 更新（完全） | `PUT /users/1` |
| PATCH | 更新（部分） | `PATCH /users/1` |
| DELETE | 削除 | `DELETE /users/1` |

### 2.3 ステータスコードの見方

| コード | 意味 | 例 |
|--------|------|----|
| 200 OK | 成功 | 取得成功 |
| 201 Created | 作成成功 | POSTで新規作成 |
| 204 No Content | 成功・ボディなし | DELETEの応答 |
| 400 Bad Request | クライアント側エラー | パラメータ不足 |
| 401 Unauthorized | 認証必要 | ログイン必須 |
| 404 Not Found | リソースが無い | `GET /users/9999` |
| 500 Internal Server Error | サーバ側エラー | システム障害 |

### 2.4 JSONとは？

- **JSON**（JavaScript Object Notation）は、データをテキストで表現する軽量フォーマットです。  
- JavaScriptでは`JSON.parse()`でオブジェクトに、`JSON.stringify()`で文字列に変換できます。  
- REST APIでは、ほとんどの場合リクエスト/レスポンスのボディがJSONでやり取りされます。

## 3. 実際にAPIを叩いてみる

### 3.1 テスト用APIを用意する

実際に動くAPIを試すため、JSONPlaceholder（<https://jsonplaceholder.typicode.com/>）という無料のモックAPIを使います。  
例：`https://jsonplaceholder.typicode.com/users`

### 3.2 Reactプロジェクトの作成

```bash
npx create-react-app rest-api-demo
cd rest-api-demo
npm start
```

`src/App.js`を以下のように書き換えます。

### 3.3 fetchを使ったデータ取得

```jsx
import React, { useEffect, useState } from 'react';

function App() {
  const [users, setUsers] = useState([]);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    fetch('https://jsonplaceholder.typicode.com/users')
      .then((res) => {
        if (!res.ok) {
          throw new Error(`HTTP error! status: ${res.status}`);
        }
        return res.json();
      })
      .then((data) => {
        setUsers(data);
        setLoading(false);
      })
      .catch((err) => {
        setError(err.message);
        setLoading(false);
      });
  }, []);

  if (loading) return <p>Loading...</p>;
  if (error) return <p>Error: {error}</p>;

  return (
    <div>
      <h1>ユーザー一覧</h1>
      <ul>
        {users.map((user) => (
          <li key={user.id}>{user.name} ({user.email})</li>
        ))}
      </ul>
    </div>
  );
}

export default App;
```

#### ポイント解説

- `useEffect`でコンポーネントがマウントされたタイミングでAPIを呼び出す。  
- `fetch`はPromiseベース。`res.ok`でステータスコードを判定。  
- `res.json()`でJSONをパースし、状態に保存。  
- エラーハンドリングは`catch`で行う。  
- `loading`, `error`でユーザーにフィードバック。

### 3.4 axiosを使ったデータ取得（オプション）

`axios`はPromiseベースで、リクエスト/レスポンスのインターセプトやデフォルトヘッダー設定が便利です。

```bash
npm install axios
```

```jsx
import axios from 'axios';

// useEffect内
useEffect(() => {
  axios
    .get('https://jsonplaceholder.typicode.com/users')
    .then((response) => {
      setUsers(response.data);
      setLoading(false);
    })
    .catch((err) => {
      setError(err.message);
      setLoading(false);
    });
}, []);
```

### 3.5 APIエラー時の対処

- **404 Not Found**  
  - URLが間違っている、リソースが無いケース。  
  - `res.status === 404`で分岐し、ユーザーに「データが見つかりません」と表示するとUXが向上します。

- **401 Unauthorized**  
  - 認証が必要な場合、`axios`のインターセプトでトークンを付与する実装が必要です。

- **500 Internal Server Error**  
  - サーバ側の問題。再試行ロジック（exponential backoff）や管理者への通知を検討します。

## 4. まとめ

- REST APIは「リソースをHTTPメソッドで操作する」シンプルな設計が特徴。  
- フロントエンドでは`fetch`または`axios`でリクエストを送り、JSONをパースして状態管理に組み込む。  
- ステータスコードを見て適切にエラーハンドリングを行うことで、ユーザー体験を損なわずに障害に対処できます。  
- 実務では認証（JWT, OAuth2）やCORS設定、データのバリデーションも重要です。ここでは基本に絞りましたが、これらを踏まえてさらに発展させていきましょう。
