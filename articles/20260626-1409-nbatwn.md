## 背景：AIツール、どう使うの？

### 私の導入シーン

筆者は今、ベンチャーのスタートアップでサーバーサイドエンジニerとして働いています。2年前まではJava製の厳密なコードレビューが送られる現場でしたが、ある日から「Claude Codeでも良いよ」と柔軟になり、だんだんとAIの出力を受け入れるようになりました。

最初はこんなことを考えていました：
- 自分が書いたコードとAIが書いたコード、どっちが良いの？
- レビューする側になったら、AIのコードって指摘しにくいの？
- 勉強になるのか、それともスキルが残らないのか？

こうした疑問に、いろいろなパターンを試してみた結果、つくったのが次の「3つのMake」という考え方です。

## 本文：AIツールを使う3つのMake（まず使ってみる編）

### Make 1：「参考になるもの」をMakeする

最初のステップとして、AIツールに「参考になるコード」を作ってもらうというのがオススメです。

具体的には、次のようなパターンで使います：

```
# 自分が書き始めたら、こんな風に頼む
Claude Code:
- ユーザー情報を取得するAPIを作成してください
- 次の仕 unneeds:
  * 認証は不要
  * PostgreSQLを使用
  * ユーザーIDを引数に取る
  * 結果はJSONで返す
```

こうすると、AIは素直に実装案を提示してくれます。自分が書かなかった部分は、「いわゃあああ」ってな感じで参考になるコードが出てきます。

でも重要なのは、これを**コピペするだけで渡す**ということではありません。

```javascript
// AIが書いてくれたコード
app.get('/api/users/:id', async (req, res) => {
  const user = await db.getUserById(req.params.id);
  res.json(user);
});
```

これをそのまま使わず、次のように自分の手でアレンジします：

```javascript
// 自分がアレンジしたコード
app.get('/api/users/:id', async (req, res) => {
  try {
    const userId = req.params.id;
    
    // バリデーションを追加
    if (!userId || typeof userId !== 'string') {
      return res.status(400).json({ error: 'Invalid user ID' });
    }
    
    const user = await db.getUserById(userId);
    
    if (!user) {
      return res.status(404).json({ error: 'User not found' });
    }
    
    res.json({
      id: user.id,
      name: user.name,
      email: user.email,
      created_at: user.createdAt
    });
  } catch (error) {
    console.error('Error fetching user:', error);
    res.status(500).json({ error: 'Internal server error' });
  }
});
```

このようにすることで、「AIが用意してくれた骸」を自分でAnimations（アニメーション）する感覚で使うことができます。

### Make 2：「自分の疑問」をMakeする

AIツールの最大の利点は、「疑問をすばやく解決できる」ということです。特に、現場で「これどうやるの？」となったとき、AIに聞くのは結構です。

ただ、その前に必ずやるのが、**「なぜこの方法が正しいのか？」をAI自身に聞く**ということです。

例えば、次のようなコードがあったとします：

```python
# ある関数を最適化したい
def calculate_fibonacci(n):
    if n <= 1:
        return n
    return calculate_fibonacci(n-1) + calculate_fibonacci(n-2)
```

これは再帰で書かれているけど、パフォーマンスが問題だとします。

```
Claude Code:
- この再帰関数のパフォーマンス問題を解決する方法を3つ教えてください
- それぞれのメリットとデメリットも教えてください
```

すると、AIはこんな答えを返してくれます：

1. **メモ化（キャッシング）**
   - メリット: 計算回数が減り、O(2^n) → O(n) に
   - デメリット: メモリ使用量が増える

2. ** bottom-up（反復）**
   - メリット: メモリ効率が良い
   - デメリット: コードがやや複雑になる

3. **ライブラリ使用（numpyなど）**
   - メリット: 高速かつ簡潔
   - デメリット: 外部依存が増える

こうしてもらった後、自分は次のように考えます：
- このプロジェクトではメモリ使用量が問題にならなさそう
- 外部ライブラリは最小にしたい
- コードの可読性も大事だ

というわけで、実装しながら考えるのです。

```python
# メモ化を採用した実装
def calculate_fibonacci(n, memo={}):
    if n in memo:
        return memo[n]
    
    if n <= 1:
        return n
    
    result = calculate_fibonacci(n-1, memo) + calculate_fibonacci(n-2, memo)
    memo[n] = result
    return result
```

このパターンでやると、AIからのアイデアを「新しい知識」として取り込むことができますし、自分の中でしっかりと定着します。

### Make 3：「自分のスキル」をMakeする

ここが一番重要です。AIツールを使うと、「なぜか自分のコードが書けなくなっている」という経験はありませんか？

私の場合、特に「インフラ系のコード」（CI/CD、Dockerなど）が多いです。なぜかある日を振って考えたのです：

> AIが書いたコードをそのまま使うより、自分で書き始めてからAIに確認してもらうほうが、自分のムキムキ度が高い

ということに気づきました。

その後、確立したのが次のサイクルです：

1. **自分でコードを書く**（80%まで手書き）
2. **AIにレビューしてもらう**
3. **AIからのフィードバックを元に改善**

具体例を見てみましょう。Docker-compose.ymlを書くとします。

まずはこれだけ書きます：

```yaml
version: '3'
services:
  app:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
```

次に、Claude Codeに聞きます：

```
Claude Code:
- このdocker-compose.yml、セキュリティとパフォーマンスの観点から改善してください
- 足りない環境変数やボリューム設定も教えてください
```

すると、AIはこんな提案をしてくれます：

```yaml
version: '3.8'
services:
  app:
    build: 
      context: .
      target: development
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgresql://user:pass@db:5432/myapp  # 追加
    volumes:
      - ./src:/app/src  # 追加
      - /app/node_modules  # 追加
    depends_on:
      - db  # 追加
    restart: unless-stopped  # 追加

  db:
    image: postgres:15
    environment:
      - POSTGRES_DB=myapp
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
    volumes:
      - postgres_data:/var/lib/postgresql/data
    restart: unless-stopped

volumes:
  postgres_data:
```

これを見て「あ、こんな書き方もあるんだ」と気づくのです。

## 聞き方のコツ：AIにコードを請う時、こうすると捗る

AIツールを使う上で、最も重要なのが「聞き方」です。私が実践している「3つの聞き方」を紹介します。

### 1. コンテキストを与える

悪い例：
```
PythonでWebスクレイピングしてください
```

良い例：
```
次の要件でWebスクレイピングツールを作ってください
- 対象: https://example.com/products
- 取得したいデータ: 商品名、価格、在庫状況
- 出力形式: CSVファイル
- 制限: 1秒ごとにリクエストを送る
- 使用ライブラリ: Python標準のみ
```

### 2. 制約を明示する

特に、次のような場面で制約を話すことが多いです：

```
次の条件でAPIを作成してください
- フレームワーク: FastAPI
- 非同期処理: 必須
- データベース: SQLAlchemyのみ使用
- 認証: JWTトークン
- テスト: pytestを使用
- 型ヒント: 必ず使用
```

### 3. フィードバックを求める

最後のコツ：「なぜこの方法を選んだの？」と聞くことです。

```
Claude Code:
- このコード、なぜExpress instead of Fastifyを選択したのですか？
- パフォーマンスと開発速度の観点で比較してください
```

こうすると、AIが本気で説明してくれるという現象があります（本気で説明してくれない場合は、もう一度別の角度から聞く）。

## 実践編：Claude Codeを使った開発フロー

最後に、私が実際に使っているClaude Codeの開発フローを紹介します。

### フェーズ1：要件定義

```
Claude Code:
- 次の要件を満たすタスク管理APIを設計してください
- ユーザーがタスクを作成/取得/更新/削除できること
- タスクにはタイトル、説明、ステータス、期限があること
- データはメモリ上に保存（永続化不要）
- レスポンスはすべてJSON形式であること
```

### フェーズ2：実装

AIが提示した設計に沿ってコードを書きますが、書き始める前にここで一度立ち止めます。

```
Claude Code:
- この設計、本当にメモリ上にデータを保持できるのですか？
- 複数リクエストがあった時、データが被らないのですか？
```

### フェーズ3：改善

自分が書いたコードをAIで改良してもらいます。

```
Claude Code:
- 次のコード、セキュリティの観点から改善してください
- 特にSQLインジェクションとエラーハンドリングを気にしてください
```

## まとめ：AIツールを使うべきか、使わないべきか

結論を言いますと、AIコーディングツールは「使わない方が良い」ということは決してありません。ですが、「そのまま渡す」ことが問題だということを学びました。

正しい使い方は：

1. **参考として使う** → 自分のコードにアレンジさせる
2. **疑問を解決する** → なぜその方法が正しいのかを学ぶ
3. **スキルを高める** → 80%は自分で書き、20%はAIで確認する

AIはとても賢いですが、それは「人の脳の補完」という位置付けです。逆に、人の脳が錆剤にならないように、AIをいつもそばで、そして適度に使うのです。

最後に、私が今使っているClaude Codeの設定を紹介しておきます。

```json
{
  "model": "claude-sonnet-4-20250514",
  "max_tokens": 4096,
  "temperature": 0.3
}
```

temperatureは低めにして、創造性より正確さを優先しています。

皆さんも、AIコーディングツールを「人の手の延長」として使ってみてください。すると、開発がより楽しくなるはずです。
