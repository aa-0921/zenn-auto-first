## 現場での Git 基本ワークフロー

チーム開発では、メインのコード（`main` や `master` ブランチ）に直接変更を加えることはまずありません。基本的には「機能ごとにブランチを切り、レビューを受けてからマージする」という流れになります。

以下に、一連の標準的なサイクルをステップバイステップで解説します。

### 1. リポジトリをローカルに持ってくる (`clone`)
プロジェクトに参加して最初に行うのがクローンです。リモートリポジトリにある全ての履歴を自分の PC にコピーします。

```bash
git clone <リポジトリのURL>
cd <ディレクトリ名>
```

### 2. 最新の状態に更新する (`pull`)
作業を開始する前に、他のメンバーが更新した内容を反映させる必要があります。これを忘れると、後で大きなコンフリクトが発生しやすくなります。

```bash
git checkout main
git pull origin main
```

### 3. 作業用ブランチを作成する (`branch` / `checkout`)
新機能の開発やバグ修正を行う際は、必ず専用のブランチを作成します。ブランチ名は `feature/add-login-function` や `fix/issue-123` のように、役割が分かる名前にするのが一般的です。

```bash
# ブランチの作成と切り替えを同時に行う
git checkout -b feature/my-task
```

### 4. 変更を記録する (`add` / `commit`)
コードを編集したら、変更内容をステージングエリアに追加し、コミットします。コミットメッセージは「何をしたか」が後から見て分かるように簡潔に書きましょう。

```bash
# 変更したファイルをステージングに追加
git add .

# コミットメッセージと共に記録
git commit -m "feat: ログイン画面のバリデーションを追加"
```

### 5. リモートに送信してレビューを依頼する (`push`)
ローカルで完了した作業をリモートリポジトリにアップロードします。これにより、GitHub 等でプルリクエスト（Pull Request / Merge Request）を作成できるようになります。

```bash
git push origin feature/my-task
```

## コンフリクトの正体と対処法

Git を使っていて最も緊張するのが「コンフリクト（Conflict）」の発生です。

### コンフリクトとは何か？
コンフリクトは、**「同じファイルの同じ箇所を、2人以上の人が同時に変更し、どちらを優先すべきか Git が判断できなくなった状態」**を指します。これはエラーではなく、Git が「人間が判断して正解を決めてください」と教えてくれている状態です。

### コンフリクトが発生するタイミング
主に以下の操作を行った時に発生します。
- `git pull` をした時
- `git merge` をした時
- プルリクエストをマージしようとした時

### 具体的な解消ステップ

例えば、`main` ブランチの最新内容を自分のブランチに取り込もうとしてコンフリクトが発生したとします。

```bash
git checkout feature/my-task
git merge main
# ここで「CONFLICT (content): Merge conflict in index.html」と表示される
```

#### ステップ1：コンフリクト箇所を確認する
エディタで該当ファイルを開くと、以下のような特殊な記号で囲まれた箇所が見つかります。

```html
<<<<<<< HEAD
<p>こんにちは、ゲストさん！</p>
=======
<p>こんにちは、ユーザーさん！</p>
>>>>>>> main
```

- `<<<<<<< HEAD` から `=======` まで： **自分のブランチ**での変更内容
- `=======` から `>>>>>>> main` まで： **取り込もうとしているブランチ**（今回の場合は main）の変更内容

#### ステップ2：コードを修正する
どちらの内容を残すか、あるいは両方を組み合わせて新しいコードにするかを判断し、**記号（`<<<<`, `====`, `>>>>`）をすべて削除して**正しいコードに書き換えます。

（例：ユーザー名を表示させる方が正しいと判断した場合）
```html
<p>こんにちは、ユーザーさん！</p>
```

#### ステップ3：修正を完了させてコミットする
修正が終わったら、Git に「衝突を解決したこと」を伝えます。

```bash
git add index.html
git commit -m "fix: mainブランチとのコンフリクトを解消"
```

これでコンフリクトの解消は完了です。

## 現場で役立つ「お守り」コマンド

基本フロー以外に、実務で「あ、やっちゃった」と思った時に役立つコマンドを紹介します。

### 直前のコミットメッセージを修正したい
`git commit` した直後にタイポに気づいた場合、やり直す必要はありません。

```bash
git commit --amend -m "正しいコミットメッセージ"
```

### 作業途中でブランチを切り替えたいが、コミットしたくない
「まだ機能が完成していないけれど、急ぎのバグ修正でブランチを切り替えなければならない」という時に便利です。一時的に変更を退避させます。

```bash
# 変更を一時退避
git stash

# ブランチを切り替えて作業し、戻ってきたら...
git checkout feature/my-task
# 退避した内容を復元
git stash pop
```

### どのブランチにいて、何が変わったか確認する
迷ったらまずこのコマンドで現状を把握してください。

```bash
# 現在のブランチと履歴をグラフ形式で表示
git log --oneline --graph --all

# 変更されたファイルの一覧を確認
git status
```

## まとめ

Git の操作に慣れるまでは、コマンド一つ打つのに不安がつきものです。しかし、以下の 3 点を意識すれば、致命的なミスは大幅に減らせます。

1. **作業前には必ず `pull` して最新状態にする。**
2. **直接 `main` ブランチで作業せず、必ず `feature` ブランチを切る。**
3. **コンフリクトは「Git が判断できないから助けて」と言っているだけ。落ち着いてエディタで修正すれば大丈夫。**

Git はツールであり、目的ではありません。コマンドを完璧に覚えることよりも、「どのような流れでコードが管理されるか」というフローを理解することに重点を置いて取り組んでみてください。
