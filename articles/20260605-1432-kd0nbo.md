## はじめに ― なぜ今、エンジニア転職が注目されるのか

近年、IT業界は「人手不足」という言葉が日常的に聞かれるほどの拡大を続けています。  
その影響で、**20代後半・30代からのキャリアチェンジ**が以前に比べてハードルが下がってきました。実際に私が30歳でエンジニアに転職したとき、以下のような「良かったこと」を実感しています。

- **年齢が給料に直結しない**：実務で成果を出せば年齢よりもスキルが評価基準になる。  
- **学習コストが予測可能**：未経験からでも体系化された学習ロードマップが多数公開されている。  
- **働き方の選択肢が広がる**：リモート・フリーランス・副業といった多様な働き方が選べるようになる。

本記事では、私が実際に体験した転職プロセス、学習の進め方、そして現場で感じた「手応え」について、具体的なエピソードとともに解説します。これからエンジニアを目指す方が「自分は大丈夫か？」と不安になるポイントをクリアにし、次の一歩を踏み出す手助けになれば幸いです。

## 1. 転職活動の全体像 ― 何をすれば、いつまでに結果が出るのか

### 1-1. 自己分析と市場調査（0〜1ヶ月目）

| 項目 | 具体的なやり方 | 参考ツール |
|------|----------------|------------|
| スキルマトリクス作成 | 現在の業務で培った「論理的思考」「要件定義」などを箇条書きし、ITスキルとマッピング | Notion、Miro |
| 求人市場の把握 | 「未経験歓迎」「経験年数不問」のキーワードで求人サイトを検索 | Indeed、Wantedly、Qiita Jobs |
| 年収・働き方の目安設定 | 同年代・同職種の年収データを参照し、希望年収と働き方を明文化 | 年収データベース（doda、レバテック） |

> **ポイント**：自分の「転職理由」を言語化すると、面接での自己PRが格段に楽になります。たとえば「業務効率化の経験を活かし、開発側で価値を創出したい」など。

### 1-2. 学習計画の立案（1〜2ヶ月目）

未経験からフロントエンドエンジニアを目指す場合、**「基礎→実践→ポートフォリオ」**の3ステップが効果的です。

```text
[基礎]   HTML/CSS → JavaScript 基礎文法
[実践]   小規模アプリ（TODOリスト） → API 連携
[ポートフォリオ] 作品集サイト（GitHub Pages） → デプロイ
```

- **学習時間の確保**：1日2時間、週14時間を目安にスケジュール化。カレンダーに「学習ブロック」を設定し、他の予定と同等に扱う。  
- **アウトプット中心**：毎回の学習で「コードを書いてGitにコミット」することを徹底。アウトプットがないと知識が定着しません。

### 1-3. 書類選考と面接（2〜4ヶ月目）

| フェーズ | 必要書類 | コツ |
|----------|----------|------|
| 書類選考 | 履歴書・職務経歴書・ポートフォリオURL | 「業務での課題解決経験」を箇条書きで具体化 |
| 技術面接 | コーディングテスト（オンライン） | まずはテストケースを全体像で把握し、時間配分を意識 |
| 人事面接 | キャリアビジョン | 「5年後にどんなエンジニアになりたいか」を具体的に語る |

> **実体験**：最初の面接で「未経験でも大丈夫ですか？」と聞かれたとき、**「業務で培った課題抽象化スキルがコードに変換できる」**と答えたところ、好印象を得られました。

## 2. 実務で「手応え」を感じた瞬間 ― 業務を通じて得た3つの収穫

### 2-1. 「コードがプロダクト価値を直接生む」実感

入社後2か月目、社内の社内ツールの UI 改修案件を任されました。**要件定義 → UI デザイン → React 実装**というフルスタック的な流れを経験し、以下のような変化がありました。

- **実装前**：デザイナーが作成したデザインをそのまま受け渡すだけ。  
- **実装後**：自らコンポーネント設計し、アクセシビリティまで配慮したコードを書いた結果、社内アンケートで「使いやすさが向上した」と評価された。

このとき「自分のコードがユーザーの業務効率を上げた」ことが実感でき、エンジニアとしてのやりがいを強く感じました。

### 2-2. 「チームの意思決定に参加できる」ことの価値

30代で入社したチームは、**スクラムマスターが不在**の自律的なチームでした。スプリントプランニングで「**技術的負債**」の議題が出た際、以下のように貢献しました。

1. 現行コードベースのリファクタリング対象を可視化（ESLint + TypeScript の導入）  
2. 見積もりを数値化し、プロダクトオーナーに提示  
3. スプリント内で優先度を再設定し、リファクタリングを実施

結果として、**デプロイ失敗率が30%削減**され、チーム全体の信頼感が向上しました。**「年齢や経験年数ではなく、具体的な提案が評価される」**という風潮が、キャリアの自由度を実感させてくれました。

### 2-3. 「学習が仕事の一部になる」サイクル

エンジニアは**常に学び続ける職種**です。入社後は社内勉強会で**Docker**や**Kubernetes**のハンズオンを行う機会があり、以下のサイクルが自然に回り始めました。

1. 勉強会で新技術を学ぶ → 2. 本番環境で検証 → 3. 成果を社内Wikiにまとめる → 4. 次の勉強会で共有

この循環が「**学習＝成果**」という感覚を生み、**給与交渉やキャリアパスの議論**でも「最新技術の実装経験」が有利に働きました。

## 3. 30代からエンジニアになるための実践的学習ロードマップ

以下は私が実際に走ったロードマップです。**「1日1時間」でも継続できるように設計**しています。

| 期間 | 学習内容 | 推奨教材・リソース | アウトプット例 |
|------|----------|-------------------|----------------|
| 0〜1ヶ月 | HTML/CSS 基礎 | MDN Web Docs、Progate | ポートフォリオ用の静的サイト作成 |
| 1〜2ヶ月 | JavaScript 基本文法 | JavaScript Primer、YouTube（ドットインストール） | Todo アプリ（ローカルストレージ保存） |
| 2〜3ヶ月 | Git & GitHub 基礎 | 「GitHub Flow」公式ドキュメント | 個人リポジトリでコミット履歴管理 |
| 3〜4ヶ月 | React 入門 | create-react-app、公式チュートリアル | 天気 API 連携アプリ |
| 4〜5ヶ月 | TypeScript 導入 | TypeScript Handbook、Udemy | 先ほどの React アプリを TypeScript 化 |
| 5〜6ヶ月 | CI/CD 基礎（GitHub Actions） | GitHub Actions Docs | プルリクエスト時に自動テスト実行 |
| 6〜7ヶ月 | ポートフォリオ完成 & デプロイ | Vercel、Netlify | 完成した作品を公開し、面接で提示 |

### コツ1：**「完走」より「完了」**  
途中で完璧を求めすぎると挫折しやすいです。たとえば React の学習で「Hooks を全部覚える」よりも「useState と useEffect を使える」ことを目標に設定しましょう。

### コツ2：**「学習時間は小刻みに」**  
1日30分でも、**「毎日」**続ける方が「週に5時間」でも不規則にやるより効果が高いです。スマホのタイマーや Pomodoro アプリで時間を区切ると続けやすくなります。

### コツ3：**「実務に近い課題を選ぶ」**  
単なるチュートリアルではなく、**「業務でやりそうな機能」**（例：社内ツールの検索機能）を題材にすると、学習がそのまま実務に転用できます。

## まとめ ― 「遅すぎる」ことはない、でも「行動は早いほど有利」

- **年齢はハンディではなく、経験の棚卸し材料**になる。  
- **学習は計画的に、アウトプット中心で**行うと、書類選考・面接で具体的な実績として提示できる。  
- **実務での手応えは、課題解決をコードで実装した瞬間**に最も感じやすい。  
- **継続的な学習サイクル**がキャリアの自由度と給与交渉力を高める。

30代からエンジニアに転身した私が得た最大の財産は「**自分で価値を創造できるスキル**」です。年齢にとらわれず、まずは**小さなコードを書き続ける**ことから始めてみてください。実務で手応えを感じたとき、きっと「この選択は正しかった」と心から思えるはずです。
