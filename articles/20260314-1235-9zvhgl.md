## はじめにRuby on Rails を使って開発を進めていると、ルーティングやマイグレーション、N+1 クエリといったテーマで頻繁に壁にぶつかります。初心者だけでなく中級者でも「なぜこのエラーが起きるのか？」という疑問が出てくるものです。本記事では、実務でよく遭遇する代表的なエラーと、実際にデバッグした際の具体的なコード例・手順をまとめました。読者が「同じエラーでハマったときにすぐ対処できる」ことを目指します。

## Railsでよくあるエラーとその原因
### ルーティングエラー（ActionController::RoutingError）
ルーティング設定が不足している、またはパラメータの受け取り方がずれると `No route matches [GET] /hoge` などのエラーが出ます。  
```ruby
# config/routes.rb
Rails.application.routes.draw do
  resources :posts do
    collection do
      get 'search' => 'posts#search'   # 検索用ルートを追加
    end
  end
end
```
上記のように `resources` のブロックに `collection` を追加すれば、`/posts/search` が有効になります。エラーが出たときは `rails routes` コマンドで実際に生成されているパス一覧を確認し、足りないパスがないかチェックしてください。

### マイグレーション失敗（ActiveRecord::Migration::IrreversibleMigration）
`rake db:migrate` 実行時に `PG::DuplicateColumn` や `Sequel::DatabaseError` が発生することがあります。これは、既に同名のカラムがある、またはデータ型が不適切な場合が多いです。  
```ruby
class AddBodyToPosts < ActiveRecord::Migration[7.0]
  def change
    add_column :posts, :body, :text
  end
end
```
マイグレーションファイルを修正したら、`db:rollback` で戻してから再度 `db:migrate` を実行します。エラーが消えなければ、`schema.rb` の状態を確認し、手動で `db:structure:load` に切り替えることも検討してください。

### N+1 クエリ問題
1対多リレーションで `includes` を忘れると、N+1 クエリが発生しパフォーマンスが低下します。  
```ruby
# N+1 の例
posts = Post.all                     # 1 クエリ
posts.each do |post|
  puts post.comments.map(&:author)   # 1投稿ごとに別クエリ → 合計 N+1end
```
対策は `includes` を使ってエagerローディングするだけです。  
```ruby
posts = Post.includes(:comments).all   # 1 クエリで全データ取得
posts.each do |post|
  post.comments.each { |c| puts c.author }
end
```
開発環境で `bullet` gem を導入すれば、N+1 が自動で検出できます。`Gemfile` に `gem 'bullet'` を追加し、`bundle install` 後に `rails console` で実行すると、コンソールに警告が出ます。

## エラー別デバッグ手法
### ログの確認
Rails は環境ごとにログ出力レベルが変わります。本番環境でエラーが頻発する場合は `config/environments/production.rb` の `config.log_level = :debug` に設定し、`log/development.log` もしくは `log/production.log` を直接確認します。  
```ruby
# config/environments/production.rb
config.log_level = :debug```
デバッグ情報が大量に出るので、`grep` や `awk` で特定のキーワードだけ抽出すると便利です。

### デバッグgemの活用
`byebug` や `pry-rails` を使うと、コード内で実行中の変数やクラスを即座に確認できます。  
```ruby
# some_controller.rb
def index
  @posts = Post.all
  byebug   # ここでは実行が停止し、コンソールに入る
  render json: @posts
end
```
`byebug` は標準で利用可能ですし、`pry-rails` を入れると `binding.pry` でも同様にデバッグできます。IDE のデバッガと組み合わせても効果的です。

### Rails consoleで試す
実際のモデルやクエリを直接試すのが最速です。`rails console` を起動し、疑わしいコードを対話的に走らせます。  
```ruby
# console で確認
post = Post.first
post.comments.count          # N+1 が発生していないかチェック
post.comments.includes(:author)
```
console では `ActiveRecord::Relation` のメソッドチェーンがそのまま評価されるので、`select` や `where` の条件を細かく検証できます。

## まとめ
- ルーティングは `resources` に `collection` や `member` を足りないパスがないか `rails routes` で確認する。
- マイグレーションはエラー内容を読むだけでなく、`schema.rb` と照合しながらロールバックを繰り返す。
- N+1 は `includes` でエagerローディングし、`bullet` gem で自動検出できる。
- デバッグには `byebug`・`pry-rails`、そして `rails console` を駆使して実行時の状態を可視化する。
- ログレベルを上げて直接ログを見ることで、 production でしか出ないエラーも捕捉しやすくなる。

これらのポイントを意識すれば、日常の開発で遭遇するほとんどのエラーを迅速に切り分け・解決できるはずです。ぜひ実務に取り入れて、開発スピードとコードの安定性を向上させてください。
