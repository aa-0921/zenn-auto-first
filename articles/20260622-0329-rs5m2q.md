## 1. データの抽出とフィルタリング（SELECT / WHERE）

まずは基本中の基本であるデータの抽出です。実務で重要なのは、「必要な列だけを絞り込むこと」と「効率的な条件指定」です。

### SELECT による列の絞り込み
初心者がやりがちなのが `SELECT *` ですが、実務では避けるべき習慣です。不要な列まで取得すると、ネットワーク帯域の消費が増え、データベースの負荷が高まるためです。

```sql
-- ❌ 避けるべき書き方
SELECT * FROM users;

-- ✅ 推奨される書き方
SELECT id, user_name, email FROM users;
```

### WHERE による条件抽出
`WHERE` 句では、論理演算子（`AND`, `OR`, `NOT`）や比較演算子を組み合わせてデータを絞り込みます。

特に実務でよく使うのが、部分一致検索の `LIKE` や、複数値のいずれかに合致させる `IN` です。

```sql
-- ユーザー名に「田中」が含まれる人を抽出（部分一致）
SELECT id, user_name 
FROM users 
WHERE user_name LIKE '%田中%';

-- ステータスが 'active' または 'pending' のユーザーを抽出
SELECT id, user_name, status 
FROM users 
WHERE status IN ('active', 'pending');

-- 2023年1月1日以降に登録したユーザーを抽出
SELECT id, user_name 
FROM users 
WHERE created_at >= '2023-01-01 00:00:00';
```

## 2. テーブルの結合（JOIN）

実務のデータベースは正規化されており、データは複数のテーブルに分かれています。それらを結合して一つの結果セットにするのが `JOIN` です。

### INNER JOIN（内部結合）
両方のテーブルに存在するデータのみを抽出します。例えば、「注文（orders）があるユーザー（users）だけ」を取得したい場合に利用します。

```sql
SELECT 
    u.user_name, 
    o.order_date, 
    o.amount
FROM users u
INNER JOIN orders o ON u.id = o.user_id;
```
※ `u` や `o` はテーブルの別名（エイリアス）です。これを使うことで、どのテーブルの列かを明確にし、記述を簡潔にできます。

### LEFT JOIN（左外部結合）
左側のテーブルの全レコードを保持し、右側のテーブルに一致するデータがあれば結合し、なければ `NULL` を入れます。「注文を一度もしたことがないユーザーも含めて全員抽出したい」場合に利用します。

```sql
SELECT 
    u.user_name, 
    o.order_id
FROM users u
LEFT JOIN orders o ON u.id = o.user_id;
```
この結果、`order_id` が `NULL` の行は「まだ一度も注文していないユーザー」であることがわかります。

## 3. データの集計とグループ化（GROUP BY / HAVING）

統計レポートやダッシュボードの作成で必須となるのが集計関数です。

### 集計関数の基本
よく使う集計関数は以下の通りです。
- `COUNT()`: 行数を数える
- `SUM()`: 合計値を出す
- `AVG()`: 平均値を出す
- `MAX()` / `MIN()`: 最大・最小値を出す

### GROUP BY によるグループ化
特定の列でグループ化して集計します。例えば、「ユーザーごとの合計注文金額」を算出する場合です。

```sql
SELECT 
    user_id, 
    SUM(amount) AS total_amount, 
    COUNT(id) AS order_count
FROM orders
GROUP BY user_id;
```

### HAVING による集計結果のフィルタリング
`WHERE` は「集計前」の行をフィルタリングしますが、`HAVING` は「集計後」の結果に対してフィルタリングを行います。

```sql
-- 合計注文金額が 10,000円以上のユーザーだけを抽出
SELECT 
    user_id, 
    SUM(amount) AS total_amount
FROM orders
GROUP BY user_id
HAVING SUM(amount) >= 10000;
```

## 4. サブクエリと共通テーブル式（CTE）

複雑なクエリを書く際、一度抽出した結果をさらに別のクエリで利用したい場合があります。

### サブクエリ（副問合せ）
クエリの中に別のクエリを記述する方法です。

```sql
-- 平均注文金額よりも高い注文を抽出
SELECT id, amount 
FROM orders 
WHERE amount > (SELECT AVG(amount) FROM orders);
```

### CTE (Common Table Expression)
サブクエリが深くなると可読性が著しく低下します。そこで、`WITH` 句を使った CTE（共通テーブル式）が推奨されます。一時的なビューのようなもので、上から順に定義するため、ロジックが追いやすくなります。

```sql
-- 1. ユーザーごとの合計金額を計算し（tmp_summary）
-- 2. その結果から上位10名だけを抽出する
WITH tmp_summary AS (
    SELECT 
        user_id, 
        SUM(amount) AS total_amount
    FROM orders
    GROUP BY user_id
)
SELECT 
    u.user_name, 
    s.total_amount
FROM tmp_summary s
JOIN users u ON s.user_id = u.id
ORDER BY s.total_amount DESC
LIMIT 10;
```
サブクエリよりも構造が明確になり、チーム開発でのレビューもしやすくなるため、実務では積極的に CTE を使うことが推奨されます。

## 5. 実務で意識すべきパフォーマンスのポイント

クエリが正しく動作しても、データ量が増えると動作が極端に遅くなることがあります。以下の点に注意してください。

### インデックスの活用
`WHERE` 句や `JOIN` の `ON` 句で指定する列には、インデックスが貼られているか確認してください。インデックスがない列で数百万件のデータを検索すると、フルテーブルスキャンが発生し、システム全体が停止するリスクがあります。

### 実行計画の確認
MySQLであれば `EXPLAIN` 文をクエリの先頭につけることで、データベースがどのようにデータを検索しようとしているかを確認できます。
```sql
EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';
```
`type` 列が `ALL` になっている場合はフルスキャンが発生しているため、インデックスの検討が必要です。

## まとめ

本記事では、SQLの基礎から実務で多用するパターンまでを解説しました。

- **SELECT**: 必要な列だけを明示的に指定する。
- **JOIN**: 内部結合（INNER）と外部結合（LEFT）を使い分ける。
- **GROUP BY**: 集計結果を絞り込むときは `HAVING` を使う。
- **CTE**: 複雑なロジックは `WITH` 句で整理して可読性を高める。
- **Performance**: インデックスを意識し、`EXPLAIN` で実行計画を確認する。

SQLは一度身につければ、どの言語やフレームワークを使っても応用が効く汎用的なスキルです。まずは簡単な抽出から始め、徐々に CTE や集計を組み合わせた複雑なクエリに挑戦してみてください。
