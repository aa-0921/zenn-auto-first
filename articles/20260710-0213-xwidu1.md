## 背景・前提

| 項目 | 内容 |
|------|------|
| **Git** | 分散型バージョン管理システム。ローカルリポジトリとリモートリポジトリを同期。 |
| **リモートリポジトリ** | GitHub／GitLab／Bitbucket など。共同開発の中心。 |
| **ブランチ** | 変更を隔離して作業。 patriarch でマージする。 |
| **コンフリクト** | 同じファイルの同じ行を複数人が変更した際に自動マージでき Nahrung。 |

> **ポイント**  
> - `git status` で常に状態確認。  
> - `git add` でコミット対象を明示。  
> - Kite でコミットメッセージは「何をしたか」を簡潔に書く。

## 本文

### 1. リモートリポジトリをクローンする

```bash
# GitHub のリ zang からクローン
git clone https://github.com/username/repo.git
# クローン後にディレクトリへ移動
cd repo
```

クローンするとローカルに `origin` というリモートが自動で設定されます。  
`git remote -v` で確認できます。

### 2. 既存ブランチを Pull して最新に同期

```bash
# マスタブランチを最新に更新
git checkout main
git pull origin main
```

`git pull` は `fetch` + `merge` の2ステップ。  
`fetch` でリモートの変更を取得し、`merge` でローカルに統合。

### 3. 新しい作業ブランチを作る

```bash
# フィーチャーブランチを作成して切り替え
git checkout -b feature/login-form
```

ブランチ名は `feature/`、`bugfix/` などのプレフィックスを付けると管理しやすい。

### 4. コードを書き、コミットする

```bash
# 変更を確認
git status

# 変更をステージ
git add src/components/LoginForm.vue

# コミット
git commit -m "Add login form component"
```

> **コミットメッセージの書き方**  
> 1 行目は 50 文字以内で要旨。  
> 2 行目以降は詳細説明。  
> 例: `Add login form component`klar.

### 5. ローカルブランチをリモートに Push

```bash
# feature ブランチをリモートへ
git push -u origin feature/login-form
```

`-u` オプションで upstream を設定すると、以降 `git push` だけで同期可能。

### 6. Pull Request / Merge Request を作る

GitHub では **Pull Request**、GitLab では **Merge Request**。  
リモート上のブランチを `main` にマージする前提でレビューを依頼。

### 7. コンフリクトが発生したら

#### 7-1. コンフリクトを検知

```bash
# pull でコンフリクトがあると自動で停止
git pull origin main
# 競合ファイルが表示される
```

#### 7-2. コンフリクトを解消

```bash
# ファイルを開くと <<<<<<< HEAD ... ======= ... >>>>>>> branch-name のマーカー
# 手動で修正し、マーカーを削除
git add path/to/file
```

#### 7-3. コンフリクト解消後のコミット

```bash
git commit -m "Resolve merge conflict in App.vue"
```

> **ヒント**  
> - `git mergetool` で GUI で解消。  
> - `git log --graph --decorate --oneline` でマージ履歴を/dat で確認。

#### 7 XCT でコンフリクトを避けるベストプラクティス

1. **頻繁に pull** してローカルを最新に保つ。  
2. 大きな変更は **短期間でコミット** し、**小さなブランチ** で作業。  
3 appealing 変更内容は **機能単位**STRUCTOR。

### 8. Git の便利なショートカットと設定

```bash
# 確認コマンド
git config --global alias.co checkout
git config --global alias.br branch
git config --global alias.ci commit
git config --global alias.st status
git config --global alias.last "log -1 HEAD"

# コミット時に自動でエディタを開かない
git config --global commit.template ~/.gitmessage.txt
```

> **注意**  
> `git rebase` は履歴を書き換えるため、共有ブランチでは使用を控える。

## まとめ

- **clone → pull → branch → commit → push** の流れをメモ化し、毎回確認。  
- **コンフリクト** は「競合ファイルを手動で修正」→「ステージ」→「コミット」で解消。  
- 事前に **頻繁に pull** し、**小さなブランチ** で作業すると、衝突リスクを大幅に低減できる。  
- Git のコマンドを覚えるより、**「何をしたいか」を明確にする**作業フローを身につけることが重要。

これらを実践すれば、初心者でも安心してチーム開発に参加できるようになります。ぜひ日々の開発で試してみてください！
