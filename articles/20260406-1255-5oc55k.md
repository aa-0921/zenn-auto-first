## はじめに：Next.js の「2つの道筋」に戸惑っていませんか？

Next.js は現代のReactアプリケーション開発で最も人気のあるフレームワークの一つですが、その歴史的経緯から「Pages Router」と「App Router」という2つのルーティングシステムが并存しています。さらに「サーバーコンポーネント」と「クライアントコンポーネント」という新しいコンポーネントの概念が加わり、初心者が最初につまずくポイントが多くなっています。

本記事では、これらの違いを明確に整理し、実際のプロジェクトでどう使い分けるべきか、具体的なコード例を交えて解説します。

## 背景：なぜ2つのRouterが生まれたのか

### Pages Router（従来の方式）
Next.js 9以前から存在する古典的なルーティングシステムです。`pages/`ディレクトリ内のファイル構造がそのままURLパスに対応します。`getServerSideProps` や `getStaticProps` といったデータ取得用の特殊関数がコンポーネントファイルに同居するのが特徴です。

### App Router（最新の標準）
Next.js 13で導入され、14で正式安定版となった新しいルーティングシステムです。`app/`ディレクトリをルートとし、ファイルベースのルーティングに加え、レイアウト、ストリーミング、エラーハンドリングなどが統合されています。サーバーコンポーネントをデフォルトとし、より最適化されたユーザー体験を目指しています。

現在、**新規プロジェクトではApp Routerが推奨**されています。ただし、既存のPages Routerプロジェクトは引き続きメンテナンスされます。

## 本文：具体的なコードで比べて理解する

### 1. ルーティング構造の違い

**Pages Router例 (`/pages/about.js`)**
```javascript
// pages/about.js
export default function AboutPage() {
  return <h1>このサイトについて</h1>;
}

// データ取得が必要な場合
export async function getServerSideProps() {
  const data = await fetch('https://api.example.com/data');
  return { props: { data } };
}
```

**App Router例 (`/app/about/page.js`)**
```javascript
// app/about/page.js
// サーバーコンポーネント（デフォルト）
export default async function AboutPage() {
  // サーバーサイドで直接データ取得可能
  const data = await fetch('https://api.example.com/data');
  
  return (
    <main>
      <h1>このサイトについて</h1>
      <div>{JSON.stringify(data)}</div>
    </main>
  );
}
```

**ポイント:**
- App Routerではファイル名が `page.js` であることが必須です。
- データ取得はコンポーネント内のトップレベルで`async/await`を使って直接記述できます（サーバーコンポーネントのため）。

### 2. サーバーコンポーネント vs クライアントコンポーネント

これが最大の理解ポイントです。

**サーバーコンポーネント（デフォルト）**
- サーバーでレンダリングされ、HTMLとしてクライアントへ送信されます。
- バンドルサイズに含まれない（クライアントに送られない）。
- データベースやファイルシステムへの直接アクセスが可能。
- `useState`, `useEffect` などのReact Hooksが**使えない**。
- デフォルトなので、特にディレクティブなしで書けます。

**クライアントコンポーネント（対話型UI用）**
- クライアント（ブラウザ）でJavaScriptとして実行・レンダリングされます。
- `"use client"` ディレクティブをファイルの先頭に記述します。
- `useState`, `useEffect`, イベントハンドラなどが使用可能。
- バンドルサイズに含まれるため、適切な粒度で使用する必要があります。

**例：インタラクティブなカウンター**
```javascript
// app/components/Counter.js
'use client'; // クライアントコンポーネント宣言

import { useState } from 'react';

export default function Counter() {
  const [count, setCount] = useState(0);
  
  return (
    <button onClick={() => setCount(count + 1)}>
      クリック数: {count}
    </button>
  );
}

// app/page.js（サーバーコンポーネント）
import Counter from './components/Counter';

export default function HomePage() {
  return (
    <div>
      <h1>ホーム</h1>
      {/* クライアントコンポーネントを埋め込み可能 */}
      <Counter />
    </div>
  );
}
```

### 3. つまずきがちなポイントと対策

#### ポイントA: `use client` の付け忘れ
`useState` や `onClick` を使っているのに `'use client'` がないと、コンパイルエラーになります。**対話的な処理があるコンポーネントは必ず先頭に記述しましょう。**

#### ポイントB: サーバーコンポーネントからのクライアントコンポーネント呼び出し
サーバーコンポーネントからクライアントコンポーネントを子として呼び出すことは可能です（上記例）。ただし、**クライアントコンポーネントからサーバーコンポーネントを`props`として直接渡すことはできません**（JSONシリアライズ可能なデータのみ）。データが必要な場合は、クライアントコンポーネント内でAPI Routeを呼び出すか、サーバーコンポーネント側で取得済みデータを渡します。

#### ポイントC: データ取得方法の変化
Pages Router: `getServerSideProps` / `getStaticProps` で取得 → `props` で渡す
App Router: コンポーネント内で直接 `fetch`（サーバーコンポーネントのみ）

App Routerでは`fetch`が自動でリクエストの重複を排除（deduplication）するため、効率的です。

#### ポイントD: レイアウトの定義方法
App Routerでは`app/layout.js`がルートレイアウト、`app/about/layout.js`が`/about`用レイアウトというように、ディレクトリごとにレイアウトを定義できます。Pages Routerの`_app.js`や`_document.js`とは発想が異なります。

## まとめ：どう選択し、どう学べばいいか

1.  **新規プロジェクトは迷わずApp Routerを選択**。Next.jsの未来はこちらにあります。
2.  **「サーバーコンポーネントがデフォルト」と覚える**。インタラクションが必要な部分だけを`'use client'`でクライアントコンポーネント化するのが、パフォーマンス的にもバンドル的にも最適なパターンです。
3.  **データ取得は、サーバーコンポーネント内で`fetch`する thinking に切り替える**。これにより、クライアントへの不要なJavaScript送信を減らせます。
4.  **既存のPages Routerプロジェクトは、無理に移行する必要はありません**。ただし、新規機能追加時はApp Routerの導入を検討しても良いでしょう。

最初は「`'use client'` が必要な場面」を体で覚えるのが近道です。ボタンを押したら何か起こる、フォームの入力がある、マウス移動に反応する——そんな部分はほぼクライアントコンポーネントだと考えてください。逆に、ヘッダー、フッター、商品一覧の表示（クリックなし）などはサーバーコンポーネントでまかなえます。

この区別を意識してコードを書くことで、Next.jsアプリケーションはより高速でメンテナンス性の高いものになります。まずは小さなコンポーネントから、サーバー／クライアントの使い分けを試してみてください。
