## コードレビューで成長するマインドセット

開発を始めたばかりの頃、私はコードレビューを受けると必ず防御的な気持ちになりました。「自分のコードがダメだと思われているんじゃないか？」という焦りや不安が湧き上がり、レビューコメントを読み飛ばすこともしばしばでした。しかし、あるプロジェクトで senior engineer の方から「レビューは、他人の目のを通して自分のコードを見るトレーニングだ」とアドバイスを受けてから、考え方が大きく変わりました。

レビューは、**他者の知見を借りて自分の理解を深める機会**です。レビュアーの意図を正しく読み取り、質問をしながら学び、工夫次第で成長の糧になります。同時に、自分のメンタルも守る必要があります。ここからは、実際に私が実践している「フィードバックの受け止め方」「質問の仕方」「レビューを成長の機会に変えるコツ」について、具体的なエピソードやコードスニペットを交えてお伝えします。

## 建設的なフィードバックの受け止め方

### 1. まずは「事実」と「提案」を切り離す

レビューコメントには、**事実**（例：「この行の変数名が曖昧です」）と**提案**（例：「より適切な名前は `userId` です」）が含まれていることが多いです。私の場合は、まず事実を整理するノートを作成します。

```
事実：変数 `data` が全体的に使用され、目的が不明確。
提案：`userData` または `apiResponse` に変更。
```

事実を切り離して書き出すことで、感情的な反応を和らげ、建設的に捉えることができます。

### 2. 理解できる部分とわからない部分を明確にする

レビューを読んだ後、「自分には理解できるか？」と自問します。理解できない場合は、その理由をメモします（例：「`async/await` のネストが複雑で、パフォーマンスへの影響が不明確」）。理解できる部分とわからない部分を明確にすることで、レビュアーに質問する際の優先順位が決まります。

### 3. 受け入れる、改善する、感謝する

どんなフィードバックも、自分のスキルアップに繋がる要素だと考えます。*受け入れる*（必要であれば変更する）、*改善する*（影響の大きい部分から修正する）、*感謝する*（レビュアーの時間と労力に感謝する）という3つのステップを実践しています。

## 質問力を磨くレビュー術

### 1. 曖昧な指摘を具体的な質問に変える

レビューコメントが漠然としている場合（例：「可読性を向上させた方が良い」）、自分で考えても解決できないことがあります。私の場合は、以下のような質問をします。

```
「`calculateTotal` メソッドの内部処理を教えていただけますか？また、どの部分が可読性を低下させているか教えてください。」
```

この質問により、レビュアーは具体的な改善点を指摘しやすくなり、相互理解が深まります。

### 2. 背景情報を共有する

レビューを読む前に、自分のコードの背景（なぜこの実装を選んだのか、制約条件など）をまとめておきます。レビュアーも背景を理解しているため、質問に対する回答が具体化されます。

### 3. 複数の解決策を提示する

レビュアーから提案を受けたら、「`try-catch` ブロックの代替案として、`Result` 型を使用する方法もあります」というように、代替案を提示します。複数の解決策を示すことで、議論が建設的な方向に進み、自分の学習も深まります。

## レビューを活かしてコードと心を強化する

### ステップバイステップの実践フロー

1. **レビューを受ける前の準備**
   - 自分の変更点の概要を1文でまとめる。
   - 懸念点や不明な点をリストアップする。

2. **レビューコメントの整理**
   - 事実と提案に分類する。
   - わからない点は優先順位をつけてメモする。

3. **質問と対応**
   - わからない点について質問する。
   - 提案を受けたら、簡単なプロトタイプを作成して検証する。

4. **変更の適用と振り返り**
   ```javascript
   // 変更前
   const data = await fetchUser();
   if (data) {
     console.log(data.name);
   }
   // 変更後（レビューで提案された変数名を採用）
   const userData = await fetchUser();
   if (userData) {
     console.log(userData.name);
   }
   ```
   - 変更後のコードを確認し、元の意図が変わっていないかチェックする。
   - 変更の理由を自分なりにまとめ、今後の参考にする。

5. **メンタルケアの時間**
   - レビュー後には、30分間の「リセットの時間」を設ける。
   - ストレッチ、短い散歩、お気に入りの音楽を聴くなどでリフレッシュする。
   - 学んだことを日記に書くことで、自己効力感を高める。

### 個人的な成果

この習慣を3ヶ月続けた結果、以下の変化を感じました。

- **コードの品質向上**：レビューで指摘された点を改善することで、バグの発生率が約20％減少しました。
- **自信の向上**：具体的な質問をすることで、レビュアーからの回答が得られるようになり、自己効力感が高まりました。
- **メンタルバランスの維持**：リセットの時間を設けることで、レビューに対する恐怖心が和らぎました。

## まとめ

コードレビューは、単なるチェック機能ではありません。それは、**他者の視点から学び、自分のコードを磨く貴重な機会**です。フィードバックを事実と提案に切り離し、適切な質問をすることで、成長のスピードを加速できます。同時に、自分自身のメンタルケアを怠らないことが、長期的な成長に不可欠です。今回紹介した「事実整理 → 質問 → 改善 → リセット」のサイクルをぜひ実践してみてください。レビューに対する捉え方が変わり、開発者としてのスキルとメンタルがともに強化されるでしょう。
