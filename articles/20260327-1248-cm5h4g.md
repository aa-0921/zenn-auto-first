## 導入
React は宣言的 UI を構築しやすいライブラリですが、初心者から中級者までがつまずきやすい落とし穴がいくつか存在します。特に `useEffect` の依存配列ミス、リストレンダーでの `key` の付け忘れ、そして状態更新による無限レンダーは現場で頻繁に見られます。本記事ではこれらのエラーを具体的なコード例とともに紹介し、実務でそのまま使える対処法を解説します。

## 背景・前提
- React 18 以降（StrictMode が有効な環境）を想定
- フックは関数コンポーネント内でのみ使用
- エラーメッセージは開発コンソールに表示されるものを基に説明

## useEffect の依存配列ミス
### 問題点
依存配列を空にしたり、必要な値を漏らすと、スタールクロージャや古い状態を参照してしまい、期待しない動作が発生します。

```jsx
import { useState, useEffect } from 'react';

function Counter() {
  const [count, setCount] = useState(0);
  // 依存配列が空のため、初回の count = 0 しか参照しない
  useEffect(() => {
    document.title = `Count: ${count}`;
  }, []); // ← ここが間違い

  return <button onClick={() => setCount(c => c + 1)}>{count}</button>;
}
```
この例では、`count` が更新されても `useEffect` 内の `document.title` が変わらず、タイトルが固定されてしまいます。

### 対処法
- 参照しているすべての値を依存配列に入れる
- `eslint-plugin-react-hooks` の `exhaustive-deps` ルールを有効にして自動検知

```jsx
useEffect(() => {
  document.title = `Count: ${count}`;
}, [count]); // count を依存に追加
```
もし依存が多すぎる場合は、オブジェクトや関数を `useMemo` / `useCallback` で安定させるか、必要なプロパティだけを分解して依存に入れます。

## キーの付け忘れによるリストレンダー問題### 問題点
リストの項目を動的に追加・削除するときに `key` を指定しないか、インデックスをキーにすると、DOM の状態がずれたり、コンポーネントの状態が不整合になることがあります。

```jsx
function TodoList({ todos }) {
  return (
    <ul>
      {todos.map((todo, index) => (
        // key がインデックスのみ → 項目の並び替えや削除で問題
        <li key={index}>{todo.text}</li>
      ))}
    </ul>
  );
}
```
たとえば、途中の項目を削除すると、その後の項目の `key` が変わり、React が誤って状態を流用してしまいます。

### 対処法
- 一意で安定した ID（データベースの ID や UUID）をキーに使う
- 一時的に ID がない場合は、`crypto.randomUUID()` やライブラリ（`uuid` など）で生成

```jsx
import { v4 as uuidv4 } from 'uuid';

function TodoList({ todos }) {
  return (
    <ul>
      {todos.map(todo => (
        <li key={todo.id ?? uuidv4()}>{todo.text}</li>
      ))}
    </ul>
  );
}
```
キーが変わらない限り、React は同じ DOM ノードを再利用し、内部状態（フォームの入力値など）を正しく保持します。

## 無限レンダーと状態更新のループ
### 問題点
`useEffect` 内で状態を更新し、その状態が依存配列に含まれていると、更新 → 再レンダー → 再実行 → 再更新 の無限ループが発生します。

```jsx
useEffect(() => {
  // count が変わるたびに setCount を呼び出す → 無限ループ
  setCount(c => c + 1);
}, [count]); // count が依存なのでループ```

### 対処法
- 状態更新は必要なタイミングだけに限定する（例：フェッチ完了後のみ）
- 依存配列から状態を外し、代わりに `ref` や別のフラグで制御する- `useReducer` や状態管理ライブラリで副作用を分離する```jsx
const [count, setCount] = useState(0);
const prevCountRef = useRef(count);

// count が変わったときだけ実行
useEffect(() => {
  if (count !== prevCountRef.current) {
    // ここでは状態を変更せず、副作用のみ実行
    console.log(`count changed to ${count}`);
    prevCountRef.current = count;
  }
}, [count]); // count は依存に残すが、状態更新はしない
```
もし本当に状態を更新したい場合は、条件分岐で無限ループを防ぎます。

```jsx
useEffect(() => {
  if (count < 10) {
    setCount(c => c + 1);
  }
}, [count]); // 10 以上になると更新が止まる
```

## まとめ
React でよく遭うエラーは、主に **依存の見落とし**、**キーの不適切な選択**、**状態更新のループ** の三つに大別できます。それぞれに対して、

1. `useEffect` の依存配列は参照しているすべての値を正確に列挙し、ESLint ルールで検知  
2. リストの `key` は一意で安定した ID を使い、インデックスは避ける  
3. 状態更新は本当に必要なタイミングだけに限定し、無限ループを防ぐ条件分岐や `ref` を活用  これらのポイントを意識してコードを書くだけで、開発時の警告や本番での予期しないバグを大幅に減らすことができます。ぜひ実際のプロジェクトに落とし込んで、より安定した React アプリケーションを構築してください。
