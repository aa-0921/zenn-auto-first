## 1. Docker とは？イメージとコンテナの違い

Docker は、アプリケーションを実行するための軽量な仮想化技術です。**イメージ**は、ファイルシステムのスナップショット、実行すべき処理、設定などがすべて記述された「レシピ」のようなものです。コンテナは、このレシピに基づいて実際に実行される**実行中のインスタンス**です。イメージは共有・再利用が可能なのに対し、コンテナは各マシンで個別に起動され、状態を持つことができます。この分離された単位のおかげで、開発者は他のプロジェクトに影響を与えることなく、安定した実行環境をどこでも再現できます。

## 2. Docker をローカルにインストールする

まず、Docker Desktop（Windows、macOS）または Linux 上の Docker Engine をインストールします。公式ページからダウンロードし、ウィザードに従ってインストールしてください。インストール後、以下のコマンドで動作確認を行います。

```bash
docker --version
docker-compose --version
```

これらのコマンドがバージョン番号を表示すれば、インストールは完了です。

## 3. 単一コンテナの実行

シンプルな例として、公式の `hello-world` イメージを起動してみましょう。

```bash
docker run --rm hello-world
```

`--rm` オプションは、コンテナが終了すると自動的にコンテナを削除します。このコマンドを実行すると、イメージの説明が表示され、コンテナのライフサイクルがどのように動作するかを実際に確認できます。

次に、Node.js アプリケーションを起動する方法を見てみましょう。まず、`Dockerfile` を作成します。

```Dockerfile
# Dockerfile
FROM node:18-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .
EXPOSE 3000
CMD ["node", "index.js"]
```

このファイルは、Node 18 をベースとした軽量な Alpine Linux イメージを使用し、依存関係をインストールし、ソースをコピーし、ポート 3000 を公開し、`index.js` を実行するという手順を定義しています。

この Dockerfile を使用してイメージをビルドします。

```bash
docker build -t my-app .
```

イメージが作成されたら、コンテナを起動します。

```bash
docker run -d -p 3000:3000 --name my-app-container my-app
```

`-d` はデタッチドモードで実行し、`-p` はホストのポート 3000 をコンテナのポートにマッピングします。`docker ps` でコンテナが実行中であることを確認できます。

## 4. docker-compose で複数サービスを設定する

実際のプロジェクトでは、データベース、キャッシュサーバー、メッセージキューなど、複数のサービスが必要になることがよくあります。`docker-compose.yml` を使用すると、YAML ファイルでサービスを宣言し、単一のコマンドですべてを管理できます。

以下は、Node.js API と PostgreSQL データベースを含むシンプルな構成の例です。

```yaml
# docker-compose.yml
version: '3.8'
services:
  api:
    build: .
    ports:
      - "3000:3000"
    environment:
      - DB_HOST=db
      - DB_PORT=5432
      - DB_USER=postgres
      - DB_PASS=secret
    depends_on:
      - db

  db:
    image: postgres:15
    environment:
      - POSTGRES_USER=postgres
      - POSTGRES_PASSWORD=secret
      - POSTGRES_DB=mydb
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

`docker compose up -d` を実行すると、Docker は `api` サービスに対して Dockerfile をビルドし、`db` サービスに対して `postgres:15` イメージをプルし、データを永続化するためのボリューム `pgdata` を作成します。

ダウンストリームでは、以下のコマンドでサービス全体を停止・削除できます。

```bash
docker compose down
```

## 5. ローカル開発で Docker を使うメリット

1. **再現性** – 開発環境と本番環境が同じ設定で動作するため、バグの発生が減少します。
2. **分離** – 各サービスが独自のファイルシステムやネットワークに isolation されるため、競合や環境設定の誤りが起こりにくくなります。
3. **チームメンバー間での共有の容易さ** – `docker-compose.yml` とイメージの Dockerfile は、追加のツールなしでどこでも同じ環境を再現できる「共有設定」となります。
4. **クリーンなマシンでの開発** – ホスト OS に言語やランタイムをインストールする必要がなく、軽量な Docker Desktop で済みます。
5. **ステートレスなコンテナ** – コンテナはステートレスであるため、スクリプトや CI パイプラインを使用して簡単にスケールイン/アウトできます。

## まとめ

Docker は、イメージ（静的なレシピ）とコンテナ（実行中のインスタンス）という2つの主要な概念で、アプリケーション開発に再現性のある隔離された環境を提供します。Docker Desktop のインストール、Dockerfile を使用したイメージのビルド、コンテナの実行を経て、`docker-compose.yml` を使用して複数のサービスを管理できるようになります。シンプルな例では、Node.js API と PostgreSQL データベースを組み合わせ、ポートマッピングとデータ永続化を実現しました。Docker をローカル開発に活用することで、チーム全体で一貫した環境を共有し、ホスト OS に依存せずにクリーンな開発ワークフローを実現できます。
