## はじめに

Git は「分散型バージョン管理システム」の代名詞です。  
チーム開発の現場では、`clone`・`pull`・`branch`・`commit`・`push` といった基本コマンドをスムーズに使いこなすことが最低条件になります。  
本記事では、**駆け出しエンジニアが実務で必ず遭遇するシナリオ**を想定し、コマンドの意味・使い方・ベストプラクティスを具体的な例とともに解説します。さらに、**コンフリクトが起きたときの対処フロー**もステップバイステップで紹介するので、実務で「どうすればいい？」と戸惑うことはなくなります。

## 前提知識と作業環境

- Git がローカルにインストール済み（`git --version` で 2.30 以上が望ましい）  
- GitHub（または GitLab）にリポジトリが作成済み  
- コマンドラインは `bash` / `zsh` / `PowerShell` いずれか  
- 基本的なシェル操作ができること  

> **Tip**  
> 初心者はまず `git config --global user.name "Your Name"` と `git config --global user.email "you@example.com"` を設定しておきましょう。これがコミットの作者情報になります。

## 1. リポジトリを取得する `clone`

リモートリポジトリの内容をローカルに持ってくる最初のステップです。

```bash
# HTTPS でクローンする例
git clone https://github.com/your-org/sample-project.git

# SSH でクローンする例（SSH 鍵が設定済みの場合）
git clone git@github.com:your-org/sample-project.git
```

- **`--depth 1`** を付けると履歴を浅くでき、クローンが高速化します（CI 環境などで便利）。
- クローンすると自動的に **`origin`** というリモートが設定され、`origin/master`（または `main`）が追跡ブランチになります。

---

## 2. 最新状態を取得する `pull`

作業を始める前に、リモートの最新コミットをローカルに取り込みます。

```bash
# 現在のブランチに対してリモートの同名ブランチをマージ
git pull origin main
```

`git pull` は内部で **`fetch` + `merge`** を実行します。  
`fetch` だけしたい場合は `git fetch origin` を使い、マージは手動で `git merge origin/main` と行うことも可能です。

### Fast‑forward とマージコミット

- **Fast‑forward**: ローカルブランチがリモートに対して遅れているだけの場合、履歴が直線的に進みます。
- **マージコミット**: ローカルで独自のコミットがあると、マージコミットが生成され履歴が分岐します。  
  `--no-ff` オプションで必ずマージコミットを作ることもできます。

---

## 3. ブランチで作業領域を分割する `branch`

### ブランチの作成とチェックアウト

```bash
# 新しいブランチを作成し同時にチェックアウト
git checkout -b feature/login
```

- `-b` は「branch」の略で、`git switch -c feature/login` と同等です（Git 2.23 以降推奨）。
- ブランチ名は **目的が分かる** ものを付けるとレビューが楽になります（例: `bugfix/header-align`）。

### ブランチ一覧・削除

```bash
# ローカルブランチ一覧
git branch

# リモート追跡ブランチ一覧
git branch -r

# ローカルブランチ削除（マージ済みが前提）
git branch -d feature/login

# 強制削除（マージされていなくても削除したいとき）
git branch -D feature/login
```

---

## 4. 変更をステージングして `commit`

### 変更の確認

```bash
git status
git diff               # 作業ツリーとインデックスの差分
git diff --staged      # インデックスと HEAD の差分
```

### ステージングとコミット

```bash
# すべての変更をステージング
git add .

# 特定ファイルだけステージング
git add src/login.js

# コミット（メッセージは必ず書く）
git commit -m "feat: ログイン画面の UI を実装"
```

#### コミットメッセージのベストプラクティス

1. 50 文字以内のサマリ行  
2. 空行で区切る  
3. 必要なら本文で「何を」・「なぜ」書く  

例:

```
feat: ログイン画面の UI を実装

- ユーザー名・パスワード入力欄を追加
- バリデーションエラーメッセージを表示
- 画面遷移のテストを追加
```

---

## 5. 変更をリモートへ反映する `push`

```bash
# 現在のブランチをリモートにプッシュ
git push origin feature/login
```

- 初回プッシュでブランチがリモートに存在しない場合は自動で作成されます。
- **`--set-upstream`**（または `-u`）を付けると、次回以降は `git push` だけで同じリモートブランチへプッシュできます。

```bash
git push -u origin feature/login
```

### 強制プッシュは要注意

```bash
# ローカル履歴でリモートを上書きする
git push -f origin feature/login
```

- 強制プッシュは他の開発者の作業を壊すリスクがあるため、**必ず事前に合意**を取るか、`--force-with-lease` を使って安全性を高めましょう。

## 6. コンフリクトが起きたときの対処フロー

### 6.1 コンフリクトの発生シーン

- `git pull` 時にリモートとローカルで同一行が変更された  
- `git merge` / `git rebase` 中に競合が検出された  

### 6.2 コンフリクト解消手順

1. **コンフリクト箇所を確認**  
   ```bash
   git status   # "both modified:" で表示
   git diff     # コンフリクトマーカー <<<<<<<, =======, >>>>>>> が出る
   ```

2. **エディタで手動解消**  
   ```diff
   <<<<<<< HEAD
   const apiUrl = "https://api.example.com/v1";
   =======
   const apiUrl = "https://api.example.com/v2";
   >>>>>>> origin/main
   ```
   - `HEAD` 部分が自分の変更、`origin/main` がリモート側の変更です。  
   - 必要なロジックに合わせて1行に統合し、マーカーは削除します。

3. **ステージング**  
   ```bash
   git add src/config.js
   ```

4. **マージコミットを作成**（マージの場合）  
   ```bash
   git commit   # エディタが開くので、コンフリクト解消の旨を書いて保存
   ```

   **リベースの場合**は `git rebase --continue` を実行。

5. **リモートへプッシュ**（必要に応じて）  
   ```bash
   git push
   ```

### 6.3 コンフリクトを未然に防ぐコツ

| 手法 | 内容 |
|------|------|
| **頻繁に pull** | `git pull` を作業開始前・作業中にこまめに実行し、差分を小さく保つ |
| **小さなコミット** | 1 つの機能を小さなコミットに分割すると、マージ時の衝突が起きにくい |
| **Feature ブランチの短命化** | 完了したらすぐに `main` にマージし、ブランチの寿命を短くする |
| **コードレビューで指摘** | 変更範囲が広い PR はレビューで指摘し、必要ならリベースしてもらう |

## 7. 実務で役立つ便利コマンド集

| コマンド | 用途 |
|----------|------|
| `git log --oneline --graph --decorate` | 視覚的にブランチ構造を把握 |
| `git stash` / `git stash pop` | 作業途中で一時的に変更を退避 |
| `git cherry-pick <hash>` | 任意のコミットだけを別ブランチへ持ってくる |
| `git reflog` | 失われたブランチやコミットを復元 |
| `git shortlog -s -n` | コミッター別の貢献度を集計（チームの見える化） |

---

## まとめ

- **基本フロー**は `clone → pull → branch → commit → push` の順で覚えるだけで、ほとんどの開発シーンに対応できます。  
- **ブランチ運用**は「目的が分かる名前」「短命化」「頻繁な pull」が鍵です。  
- **コンフリクト**は避けられないこともありますが、`git status`・`git diff`・`git add`・`git commit`（または `git rebase --continue`）という一連の手順を正しく踏めば安全に解消できます。  
- さらに **便利コマンド**を日常的に使うことで、Git の操作が格段に速く、ミスも減ります。

この記事で紹介したコマンドとフローを実務にすぐ取り入れ、チーム開発で「Git が怖い」から「Git が得意」へとステップアップしましょう。
