## はじめに

エンジニアとしてバックエンド開発やデータ分析に携わる際、避けて通れないのが SQL（Structured Query Language）です。多くのフレームワークでは ORM（Object-Relational Mapping）が提供されており、直接 SQL を書かなくてもデータの操作は可能です。しかし、複雑な集計やパフォーマンスの最適化が必要になったとき、結局は「生 SQL」が書けるかどうかが開発スピードと品質に直結します。

本記事では、駆け出しから中級エンジニアの方に向けて、実務で頻出する SQL の基本構文から、JOIN によるテーブル結合、集計関数、そして少し踏み込んだサブクエリの使い方までを具体例とともに解説します。

## 1. データの抽出と絞り込み（SELECT / WHERE）

まずは基本中の基本であるデータの抽出です。実務では「全件取得」することは稀で、ほとんどの場合、特定の条件でデータを絞り込みます。

### 基本的な SELECT と WHERE
例えば、`users` テーブルから「20歳以上のユーザーの名前とメールアドレス」を取得したい場合は以下のように記述します。

```sql
SELECT 
  name, 
  email 
FROM 
  users 
WHERE 
  age >= 20;
```

### 実務でよく使う絞り込みパターン
実務では、単一の条件だけでなく、複数の条件を組み合わせたり、部分一致検索を行ったりすることが多いです。

- **AND / OR による複数条件**:
  ```sql
  -- 20歳以上 かつ ステータスが 'active' のユーザー
  SELECT * FROM users WHERE age >= 20 AND status = 'active';
  ```

- **IN 句による複数値の指定**:
  ```sql
  -- 特定のロール（管理者、編集者）を持つユーザーを抽出
  SELECT * FROM users WHERE role IN ('admin', 'editor');
  ```

- **LIKE 句による部分一致（あいまい検索）**:
  ```sql
  -- 名前に「田中」が含まれるユーザーを検索（% がワイルドカード）
  SELECT * FROM users WHERE name LIKE '%田中%';
  ```

## 2. テーブルを繋いで情報を出す（JOIN）

リレーショナルデータベースの最大の利点は、データを正規化して複数のテーブルに分散させ、必要に応じてそれらを結合できることです。ここでは、`users`（ユーザー）テーブルと `orders`（注文）テーブルを例に解説します。

### INNER JOIN（内部結合）
両方のテーブルに共通して存在するデータのみを抽出します。「注文履歴があるユーザーだけを抽出したい」場合に利用します。

```sql
SELECT 
  u.name, 
  o.order_date, 
  o.amount 
FROM 
  users AS u
INNER JOIN 
  orders AS o ON u.id = o.user_id;
```
`AS` を使ってエイリアス（別名）を付けることで、`u.name` のように簡潔に記述でき、可読性が向上します。

### LEFT JOIN（左外部結合）
左側のテーブル（FROM 句に指定したテーブル）の全レコードを保持しつつ、右側のテーブルに一致するデータがあれば結合します。一致しない場合は `NULL` が入ります。「注文を一度もしたことがないユーザーも含めて一覧を出したい」場合に必須の構文です。

```sql
SELECT 
  u.name, 
  o.order_id 
FROM 
  users AS u
LEFT JOIN 
  orders AS o ON u.id = o.user_id;
```
このクエリの結果で `order_id` が `NULL` の行は、「まだ一度も注文していないユーザー」であると判断できます。

## 3. データの集計とグループ化（GROUP BY / HAVING）

レポート作成やダッシュボードの実装で多用するのが集計関数です。`COUNT`, `SUM`, `AVG`, `MAX`, `MIN` などの関数を使い、データをまとめ上げます。

### GROUP BY によるグループ化
「ユーザーごとの合計注文金額」を算出したい場合は、`user_id` でグループ化します。

```sql
SELECT 
  user_id, 
  SUM(amount) AS total_amount, 
  COUNT(*) AS order_count 
FROM 
  orders 
GROUP BY 
  user_id;
```

### HAVING による集計結果へのフィルタリング
よくある間違いが、`WHERE` 句で集計結果を絞り込もうとすることです。`WHERE` は「グループ化する前」にフィルタリングし、`HAVING` は「グループ化した後」にフィルタリングします。

```sql
-- 合計注文金額が 10,000円以上のヘビーユーザーだけを抽出
SELECT 
  user_id, 
  SUM(amount) AS total_amount 
FROM 
  orders 
GROUP BY 
  user_id 
HAVING 
  SUM(amount) >= 10000;
```

## 4. 複雑な抽出を実現するサブクエリ

サブクエリ（副問い合わせ）とは、クエリの中にさらにクエリを書く手法です。

### WHERE 句でのサブクエリ
「平均注文金額よりも高い注文をすべて抽出したい」という場合、まず平均値を出し、その値を条件に指定する必要があります。

```sql
SELECT 
  * 
FROM 
  orders 
WHERE 
  amount > (SELECT AVG(amount) FROM orders);
```

### FROM 句でのサブクエリ（インラインビュー）
サブクエリで一度集計した結果を、あたかも一つのテーブルのように扱い、さらに別の処理を重ねるパターンです。

```sql
-- ユーザーごとの合計金額を算出した後、その合計金額が高い順に並び替えて上位5件を出す
SELECT 
  summary.user_id, 
  summary.total 
FROM (
  SELECT 
    user_id, 
    SUM(amount) AS total 
  FROM 
    orders 
  GROUP BY 
    user_id
) AS summary 
ORDER BY 
  summary.total DESC 
LIMIT 5;
```

## 実務で意識すべきパフォーマンスのポイント

SQL が書けるようになったら、次に意識すべきは「パフォーマンス」です。特にデータ量が増えた際に、以下の点に注意してください。

1. **SELECT * を避ける**: 必要なカラムだけを指定してください。不要なデータの転送を減らすことでメモリ消費を抑えられます。
2. **インデックスの活用**: `WHERE` 句や `JOIN` の結合条件に使用するカラムにはインデックスが貼られているか確認してください。インデックスがない場合、フルテーブルスキャンが発生し、処理時間が激増します。
3. **EXPLAIN の活用**: クエリの前に `EXPLAIN` をつけて実行すると、データベースがどのようにデータを検索したかの実行計画が表示されます。どの部分で時間がかかっているかを分析する習慣をつけましょう。

## まとめ

今回は、実務で頻出する SQL の基本から応用までを解説しました。

- **SELECT / WHERE**: 基本的な抽出と絞り込み。`IN` や `LIKE` を使い分ける。
- **JOIN**: `INNER JOIN` で共通データを、`LEFT JOIN` で片方の全データを抽出。
- **GROUP BY / HAVING**: 集計後のフィルタリングには `HAVING` を使う。
- **サブクエリ**: 複雑な条件や段階的な集計に活用する。

SQL は一度習得すれば、使用するデータベース（MySQL, PostgreSQL, SQLite など）が変わっても基本的な考え方は共通です。まずは小さなテーブルで実際にクエリを叩き、どのような結果が返ってくるかを体感しながら学習してみてください。
