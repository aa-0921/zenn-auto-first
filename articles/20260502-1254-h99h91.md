## 背景・前提

### コールバック地獄とは？

```js
fetchUser(userId, function (err, user) {
  if (err) return handleError(err);
  fetchPosts(user.id, function (err, posts) {
    if (err) return handleError(err);
    fetchComments(posts.id, function (err, comments) {
      if (err) return handleError(err);
      // さらに処理…
    });
  });
});
```

ネストが深くなると「読みづらい」「バグが見つけにくい」「デバッグが面倒」といった問題が発生します。  
さらにエラーハンドリングを一箇所でまとめることが難しく、同じエラーハンドリング関数を何度も書く必要があります。

### Promise と async/await の基本

- **Promise**  
  非同期処理の結果を「約束」するオブジェクト。`then`/`catch`で連鎖させることで直線的に書くことができます。

- **async/await**  
  `Promise`をより直感的に扱う構文。`await`で非同期処理を待ち、`try/catch`でエラーを捕捉します。

両者は相互変換が可能で、プロジェクトのコーディング規約に合わせて使い分けるのが一般的です。

## 本文

### ステップ1: コールバックをPromiseに変換

まずは、コールバックベースのAPIをPromise化してみます。Node.js の `util.promisify` やブラウザの `fetch` が代表例です。

```js
// コールバック版
function getData(url, callback) {
  const xhr = new XMLHttpRequest();
  xhr.open('GET', url);
  xhr.onload = () => callback(null, xhr.responseText);
  xhr.onerror = () => callback(new Error('Network error'));
  xhr.send();
}

// Promise版
function getDataAsync(url) {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest();
    xhr.open('GET', url);
    xhr.onload = () => resolve(xhr.responseText);
    xhr.onerror = () => reject(new Error('Network error'));
    xhr.send();
  });
}
```

### ステップ2: Promiseチェーンでエラーハンドリング

Promiseを連鎖させると、エラーは`catch`で一括で捕捉できます。

```js
getDataAsync('/api/user')
  .then(userStr => JSON.parse(userStr))
  .then(user => getDataAsync(`/api/posts?userId=${user.id}`))
  .then(postsStr => JSON.parse(postsStr))
  .then(posts => {
    console.log('Posts:', posts);
  })
  .catch(err => {
    console.error('エラーが発生しました:', err);
  });
```

ポイントは、**エラーはチェーンの途中で発生しても`catch`でまとめて処理できる**ことです。  
コールバック版ではエラー処理が各レイヤーで重複して書かれるのに対し、Promise版は一箇所に集約できます。

### ステップ3: async/awaitでさらに可読性を向上

同じロジックをasync/awaitで書くと、まるで同期処理のように直線的に記述できます。

```js
async function loadUserAndPosts() {
  try {
    const userStr = await getDataAsync('/api/user');
    const user = JSON.parse(userStr);

    const postsStr = await getDataAsync(`/api/posts?userId=${user.id}`);
    const posts = JSON.parse(postsStr);

    console.log('Posts:', posts);
  } catch (err) {
    console.error('エラーが発生しました:', err);
  }
}

loadUserAndPosts();
```

**メリット**  
- エラーハンドリングが`try/catch`で一箇所に集約  
- 可読性が高く、デバッグも容易  
- `await`はPromiseが解決されるまで処理を一時停止し、結果を直接変数に代入できる

### ステップ4: 複数の非同期処理を同時に実行

非同期処理が複数ある場合、`Promise.all`や`Promise.allSettled`を使うと並列実行が可能です。

```js
async function loadMultiple() {
  try {
    const [userStr, settingsStr] = await Promise.all([
      getDataAsync('/api/user'),
      getDataAsync('/api/settings')
    ]);

    const user = JSON.parse(userStr);
    const settings = JSON.parse(settingsStr);

    console.log(user, settings);
  } catch (err) {
    console.error('並列処理中のエラー:', err);
  }
}
```

`Promise.allSettled`は全ての処理が完了するまで待ち、個別に成功/失敗を返します。  
エラーが起きても他の処理は続行したいケースに便利です。

### ステップ5: 実務でのベストプラクティス

| 項目 | 推奨内容 |
|------|----------|
| **コールバックの排除** | 既存APIがコールバックの場合は`promisify`でラップ |
| **エラーハンドリング** | Promiseチェーンは末尾の`catch`、async/awaitは`try/catch`で一括 |
| **並列実行** | `Promise.all`で同時実行し、待ち時間を最小化 |
| **可読性** | 複雑なネストは関数化し、名前を付ける |
| **型安全** | TypeScriptを併用すると型推論でエラー防止 |

## まとめ

- コールバック地獄はネストが深くなるほどバグが増える原因になる。  
- Promiseを使うとエラー処理をチェーンの末尾でまとめられ、可読性が向上。  
- async/awaitはさらに同期処理に近い形で書け、デバッグも楽になる。  
- `Promise.all`/`Promise.allSettled`で複数の非同期処理を並列実行し、パフォーマンスを最適化。  
- 実務では既存ライブラリのPromise化、エラーハンドリングの統一、並列処理の適切な利用が重要。

これらを意識してコードを書くことで、非同期処理の複雑さを大幅に軽減できます。ぜひ、日々の開発に取り入れてみてください。
