## 1. Pages Router と App Router の決定的な違い

まず、最も根本的な違いである「ルーティングの仕組み」と「レンダリングの考え方」を整理しましょう。

### ファイルシステムベースのルーティング

Next.js にはこれまで、`pages` ディレクトリ配下にファイルを置くことでルーティングを実現する **Pages Router** がありました。

- **Pages Router**: `pages/about.tsx` が `/about` という URL に対応。
- **App Router**: `app/about/page.tsx` が `/about` という URL に対応。

App Router では、ディレクトリ構造がより詳細に制御できるようになりました。`page.tsx` 以外にも、`layout.tsx`（共通レイアウト）、`loading.tsx`（ローディング表示）、`error.tsx`（エラーハンドリング）といった特殊なファイル名が用意されており、これらを組み合わせることで、ページ単位の高度な制御が可能になっています。

### レンダリング戦略の変化

Pages Router では、主に `getStaticProps` (SSG) や `getServerSideProps` (SSR) を使って、データ取得のタイミングを明示的に制御していました。

これに対し、App Router では **React Server Components (RSC)** が中心となります。これにより、「どのコンポーネントをサーバーで実行し、どのコンポーネントをクライアントで実行するか」という、より細粒度な制御が求められるようになりました。

## 2. Server Component と Client Component の使い分け

App Router で最も多くのエンジニアが「つまずく」ポイントが、**Server Component (RSC)** と **Client Component** の境界線です。

### Server Component (デフォルト)
App Router 内のコンポーネントは、デフォルトで **Server Component** として扱われます。

- **メリット**:
  - JavaScript のバンドルサイズを削減できる（クライアントに送られる JS が減る）。
  - サーバー上で直接データベースやファイルシステムにアクセスできる。
  - セキュリティ上、API キーなどの機密情報を安全に扱える。
- **できないこと**:
  - `useState` や `useEffect` などの React Hook の使用。
  - `onClick` や `onChange` などのブラウザイベントリスナーの登録。
  - `window` や `localStorage` などのブラウザ固有 API の使用。

### Client Component
「ブラウザ上でのインタラクション」が必要な場合にのみ、明示的に **Client Component** を指定します。ファイルの先頭に `'use client';` と記述することで宣言します。

- **使いどころ**:
  - ボタンのクリックイベントを扱う。
  - `useState` で状態管理を行う。
  - `useEffect` で副作用（API 通信のトリガーなど）を制御する。

### 実践的な設計パターン：コンポーネントの「境界」をどこに引くか

初心者がやりがちなミスは、「何でもかんでも `'use client';` を付けてしまうこと」です。これでは Next.js のメリットである「高速な初期表示」を享受できません。

**推奨される設計パターン:**
「データ取得や重いロジックは Server Component で行い、インタラクションが必要な最小限のパーツだけを Client Component に切り出す」のがベストプラクティスです。

```tsx
// ❌ 悪い例: ページ全体を Client Component にしてしまう
'use client';

export default function Page() {
  const [data, setData] = useState([]); // ページ全体が JS を抱え込み、重くなる

  useEffect(() => {
    fetch('/api/data').then(res => res.json()).then(setData);
  }, []);

  return (
    <div>
      <h1>商品一覧</h1>
      <ul>
        {data.map(item => <li key={item.id}>{item.name}</li>)}
      </ul>
      <button onClick={() => alert('Clicked!')}>クリック</button>
    </div>
  );
}

// ✅ 良い例: データ取得は Server Component、ボタンだけ Client Component
// (app/products/page.tsx)
import ProductList from './ProductList'; // Server Component
import LikeButton from './LikeButton';   // Client Component

export default async function Page() {
  // サーバーサイドで直接データを取得（API経由ではなくDBに直接アクセスも可能）
  const data = await db.products.findMany(); 

  return (
    <main>
      <h1>商品一覧</h1>
      <ul>
        {data.map(item => (
          <li key={item.id}>
            {item.name}
            <LikeButton /> {/* インタラクションが必要な部分だけを切り出し */}
          </li>
        ))}
      </ul>
    </main>
  );
}

// (app/products/LikeButton.tsx)
'use client';

export default function LikeButton() {
  return <button onClick={() => console.log('Liked!')}>❤️</button>;
}
```

このように、**「Leaf Component（末端のコンポーネント）を Client Component にする」** という意識を持つと、パフォーマンスの高いアプリケーションを構築しやすくなります。

## 3. よくあるエラーと解決策

App Router を触っていると、以下のようなエラーによく遭遇します。

### 「Event handlers cannot be passed to client component props」
これは、Server Component から Client Component へ、関数（`onClick` など）を props として渡そうとしたときに発生します。

- **原因**: 関数はシリアライズ（直列化）できないため、サーバーからクライアントへ渡すことができません。
- **解決策**: 関数を渡すコンポーネント自体を Client Component にするか、データの受け渡しのみに留めるように設計を見直します。

### 「useState is not defined」
これは単純に、`'use client';` の記述を忘れている場合に発生します。

- **解決策**: ファイルの最上行に `'use client';` があるか確認してください。

### 「You're attempting client-side code when it should be server-side」
ブラウザでしか動かない `window` オブジェクトなどにアクセスしようとした際に発生します。

- **解決策**: 
  1. コンポーネントを Client Component にして、`useEffect` の中で実行する。
  2. または、`typeof window!== 'undefined'` でチェックを行う。

## まとめ

Next.js App Router は、これまでの Web 開発の常識を少し書き換える強力なフレームワークです。

- **Pages Router** は「ファイルベースの単純なルーティング」。
- **App Router** は「Server Component による高度なパフォーマンス制御」。
- **設計のコツ** は、「データ取得はサーバーで行い、インタラクションが必要な最小単位だけを Client Component にすること」。

最初は「どこまでがサーバーで、どこからがクライアントか」を判断するのに時間がかかるかもしれません。しかし、この「境界線」を意識できるようになると、Next.js の真の力を引き出し、高速でスケーラブルなアプリケーションを構築できるようになります。

まずは、小さなコンポーネントから `'use client';` を意識して設計する練習を始めてみましょう。
