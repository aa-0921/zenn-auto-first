## 背景・前提Next.js 13 以降では、App Router がデフォルトとなり、従来の Pages Router はオプション扱いに。App Router では **サーバーコンポーネント** と **クライアントコンポーネント** が明示的に区別されます。この違いは、データ取得のタイミングやパフォーマンスに直結するため、理解しておく必要があります。

## 本文### 1. App Router とは
App Router はファイルシステムベースのルーティングを提供し、ディレクトリ構造そのままがルートとなります。`app/` ディレクトリ以下に `page.tsx` や `layout.tsx` を置くだけでルートが定義されます。これに対し、Pages Router は `pages/` ディレクトリに `index.js` などを置く従来方式です。

```tsx
// app/page.tsx
export default function Home() {
  return <h1>Hello from App Router</h1>;
}
```

### 2. サーバーコンポーネントとクライアントコンポーネント- **サーバーコンポーネント** はデフォルトでサーバー側で実行され、データ取得やサーバー側のロジックを書けます。`"use client"` を付けない限り、全コンポーネントはサーバーコンポーネントです。
- **クライアントコンポーネント** はブラウザで描画される必要があるため、`"use client"` をファイルまたはコンポーネントヘ追記します。

```tsx
// app/about/page.tsx
'use client';
import { useState } from 'react';

export default function About() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>count: {count}</button>;
}
```

### 3. データ取得の方法
App Router では `fetch` が自動的に **サーバー側で実行** され、結果は **キャッシュ** 可能です。`getServerSideProps` や `getStaticProps` が不要になった代わりに、**非同期関数** をコンポーネントのトップレベルで書きます。

```tsx
// app/blog/page.tsx
async function getPosts() {
  const res = await fetch('https://api.example.com/posts', { next: { revalidate: 60 } });
  return res.json();
}

export default async function BlogPage() {
  const posts = await getPosts();
  return (
    <ul>
      {posts.map(p => (
        <li key={p.id}>{p.title}</li>
      ))}
    </ul>
  );
}
```

### 4. レイアウトの共有
`layout.tsx` はルート全体で共有されるレイアウトを定義できます。ネストされたディレクトリに配置すれば、部分的に別レイアウトを作れます。Pages Router での `getLayout` の代わりに、**ファイル自体がレイアウト** として機能します。

```tsx
// app/layout.tsx
import './globals.css';
export default function RootLayout({
  children,
}: { children: React.ReactNode }) {
  return (
    <html lang="ja">
      <body>{children}</body>
    </html>
  );
}
```

### 5. 共通のつまずきポイント
1. **「use client」の位置**     - ページ全体をクライアントコンポーネントにしたい場合は、`page.tsx` の最上部に `'use client'` を記述しますが、**子コンポーネント** でも同様に必要です。これを忘れると、React が「サーバーで描画できない」エラーが出ます。
2. **データ取得の非同期処理**  
   - `async` 関数をトップレベルで書くと自動的に **サーバー側実行** ですが、**クライアント側で実行したい** 場合は `useEffect` や `useSWR` などを使う必要があります。
3. **静的生成と動的生成の切り替え**  
   - `export const revalidate = 0;` で **毎回再生成**、`export const revalidate = 60;` で **60 秒ごとに再生成**、といった制御が可能です。設定を誤るとキャッシュが原因で古いデータが表示され、デバッグが困難になることがあります。
4. **エラーハンドリング**     - サーバーエラーが発生した際は、`error.tsx` や `not-found.tsx` を作成してカスタムエラーページを用意できますが、Pages Router と比べてファイル名が変わります。

## まとめ- App Router は **ファイルシステムベース** のルーティング、サーバーコンポーネントとクライアントコンポーネントの明示的分離が中心です。
- データ取得は **非同期関数** をトップレベルで書くだけで簡潔に、キャッシュコントロールも組み込みやすい。
- つまずきやすいポイントは **use client の位置、データ取得のスコープ、キャッシュ設定** の 3 つです。
- これらを意識すれば、初心者でもスムーズに Next.js 13 の新機能を活用できます。
