## はじめに：なぜ今、ローカル開発にDockerが必要なのか

「自分のPCでは動くのに、本番環境や他のメンバーのPCでは動かない」

エンジニアであれば一度は経験したことがある、この**「環境差分地獄」**から脱却するための標準的な解決策が Docker です。近年ではクラウドネイティブな開発だけでなく、ローカル開発環境の構築においても Docker（および docker-compose）を採用するケースが圧倒的に増えています。

本記事では、駆け出しエンジニアの方向けに、**「イメージとコンテナの違い」**という核心概念から、**「docker-compose を使った実践的な開発環境の立ち上げ手順」**までを、具体的なコマンド例とともに解説します。これを読めば、明日から自分のプロジェクトに Docker を導入する第一歩を踏み出せるはずです。

## Dockerの核心概念：イメージとコンテナの違いを正しく理解する

Docker を学ぶ上で最初の関門であり、最も重要なのが **「イメージ（Image）」** と **「コンテナ（Container）」** の関係性です。これを曖昧にしたまま進めると、後で「データが消えた」「設定が反映されない」といったトラブルの原因になります。

### イメージ（Image）＝「設計図」または「テンプレート」

イメージは、アプリケーションを実行するために必要な **OS、ライブラリ、ソースコード、設定ファイル、実行コマンド** などをすべて含んだ **読み取り専用のテンプレート** です。

- **不変（Immutable）**：一度作成したら中身を変更できません（変更するには新しいイメージをビルドする必要があります）。
- **レイヤー構造**：Dockerfile の命令（`FROM`, `RUN`, `COPY` など）ごとにレイヤーとして積み重なり、キャッシュによってビルド高速化が図られます。
- **共有可能**：Docker Hub などのレジストリに push すれば、誰でも同じ環境を `pull` するだけで再現できます。

### コンテナ（Container）＝「設計図から建てられた実際の家（実行インスタンス）」

コンテナは、イメージをベースに **「読み書き可能な薄いレイヤー（コンテナレイヤー）」を追加して起動した、隔離された実行環境** です。

- **可変（Mutable）**：コンテナ内でファイルを作成・編集・削除できます。
- **ライフサイクルがある**：`create` → `start` → `stop` → `remove` という状態遷移があります。
- **削除すれば中身も消える**：コンテナを削除（`docker rm`）すると、コンテナレイヤーに書き込まれたデータは消失します（ボリュームをマウントしていない限り）。

### 図解で理解する関係性

```text
+------------------------+
|   Docker Host (OS)     |
|                        |
|  +------------------+  |
|  |   Image (ReadOnly)  |  <-- docker pull / docker build
|  |  [App Code]       |  |
|  |  [Libs]           |  |
|  |  [Base OS]        |  |
|  +------------------+  |
|           ↑            |
|           | (Layer)    |
|  +------------------+  |
|  | Container (R/W)  |  <-- docker run / docker-compose up
|  | [Container Layer]|  |     (ここでファイル変更などが発生)
|  +------------------+  |
+------------------------+
```

**ポイント：**
> **「イメージはクラス、コンテナはインスタンス」** と捉えるとオブジェクト指向に慣れている方は直感的かもしれません。ただし、インスタンス（コンテナ）を捨てると中身の状態も消える点は通常のオブジェクトとは異なります。

## 実践：docker-compose で「開発用」環境をコード化する

`docker run` コマンドでオプションを長々と書くのは現実的ではありません。複数のコンテナ（アプリ＋DB＋Redisなど）を連携させ、ボリュームマウントやポートフォワーディング、環境変数を管理するために **docker-compose（Compose V2 では `docker compose`）** を使います。

### 前提：プロジェクト構成例

ここでは、一般的な **Node.js (Express) アプリケーション ＋ PostgreSQL** という構成を想定します。

```text
my-project/
├── docker-compose.yml   # 環境定義の要
├── Dockerfile           # アプリ用イメージのレシピ
├── package.json
├── src/
│   └── index.js
└── .env                 # 環境変数（Git管理外推奨）
```

### 1. Dockerfile：アプリの「設計図」を書く

`Dockerfile` はイメージをビルドするための手順書です。開発用と本番用で分ける（マルチステージビルド）のがベストプラクティスですが、まずはシンプルな開発用単体で書きます。

```dockerfile: Dockerfile
# 1. ベースイメージの指定（Node.js 20 の軽量版 Alpine Linux）
FROM node:20-alpine

# 2. 作業ディレクトリの設定
WORKDIR /app

# 3. 依存関係ファイルを先にコピー（キャッシュ活用のため）
COPY package*.json ./

# 4. 依存関係のインストール
# --omit=dev を外し、開発用ツールも入れる
RUN npm install

# 5. ソースコードをコピー
COPY . .

# 6. ポート公開の宣言（ドキュメント的意味合い、実際の公開は compose 側で制御）
EXPOSE 3000

# 7. 起動コマンド（開発時はホットリロードツール等を想定）
CMD ["npm", "run", "dev"]
```

**解説ポイント：**
- `COPY package*.json ./` を `COPY . .` より前に置くことで、**ソースコードだけ変更されたときに `npm install` のキャッシュが効く**ようにしています（レイヤーキャッシュの最適化）。
- `CMD` は `docker-compose.yml` 側の `command` で上書き可能です。

### 2. docker-compose.yml：インフラ構成を宣言的に記述する

これが「インフラのコード化」の本体です。YAML形式で、サービス（コンテナ）、ネットワーク、ボリュームを定義します。

```yaml: docker-compose.yml
version: '3.8' # 現在は version 省略推奨だが、互換性のため記載してもOK

services:
  # --- アプリケーションサービス ---
  app:
    build: 
      context: .
      dockerfile: Dockerfile
    # 開発時はソースコードをホストと同期させる（ホットリロード実現の鍵）
    volumes:
      - .:/app                 # プロジェクトルートを /app にマウント
      - /app/node_modules      # 匿名ボリュームで node_modules をホストから隠蔽（OS差吸収）
    ports:
      - "3000:3000"            # ホスト:コンテナ
    environment:
      - NODE_ENV=development
      - DATABASE_URL=postgres://user:pass@db:5432/myapp # サービス名 'db' で名前解決可能
    depends_on:
      - db                     # 起動順序制御（ヘルスチェックではない点に注意）
    command: npm run dev       # Dockerfile の CMD を上書き

  # --- データベースサービス ---
  db:
    image: postgres:15-alpine  # 既存イメージを利用（Dockerfile不要）
    volumes:
      - postgres_data:/var/lib/postgresql/data # 名前付きボリュームでデータ永続化
    environment:
      - POSTGRES_USER=user
      - POSTGRES_PASSWORD=pass
      - POSTGRES_DB=myapp
    ports:
      - "5432:5432"            # ホストからGUIツール等で接続する場合に公開

# 名前付きボリュームの定義（トップレベル）
volumes:
  postgres_data:
```

#### 開発で重要な3つの記述テクニック

1.  **バインドマウント (`- .:/app`)**
    ホスト側のカレントディレクトリをコンテナの `/app` にマウントします。これにより、**ホストでファイルを保存するだけでコンテナ内のコードが更新され、ホットリロード（nodemon等）が動作します**。

2.  **匿名ボリュームでの `node_modules` 隠蔽 (`- /app/node_modules`)**
    ホスト側の `node_modules`（Windows/Mac固有のバイナリ含む）がコンテナに漏れ込むのを防ぎ、コンテナ内で `npm install` したクリーンなモジュールを使わせます。これが「自分のPCでは動く」を防ぐ最大のコツです。

3.  **サービス名での名前解決 (`DATABASE_URL=...@db:5432`)**
    `depends_on` で指定したサービス名（ここでは `db`）が、コンテナ内部の DNS で自動的に解決されます。IPアドレスをハードコーディングする必要がありません。

### 3. 環境変数の外出し（.env ファイル）

`docker-compose.yml` にパスワードを直書きするのは避けましょう。プロジェクトルートに `.env` ファイルを作成し、`.gitignore` に追加します。

```text: .env
POSTGRES_USER=myuser
POSTGRES_PASSWORD=strong_password_here
POSTGRES_DB=myapp_dev
```

`docker-compose.yml` 側では `${VAR_NAME}` 形式で参照します。

```yaml: docker-compose.yml (抜粋)
  db:
    environment:
      - POSTGRES_USER=${POSTGRES_USER}
      - POSTGRES_PASSWORD=${POSTGRES_PASSWORD}
      - POSTGRES_DB=${POSTGRES_DB}
  app:
    environment:
      - DATABASE_URL=postgres://${POSTGRES_USER}:${POSTGRES_PASSWORD}@db:5432/${POSTGRES_DB}
```

## 開発ワークフロー：日常のコマンドチートシート

環境定義ができたら、あとはこのコマンドセットで日々の開発を回します。

### 初回セットアップ・ビルド

```bash
# イメージのビルドとコンテナの作成・起動（バックグラウンド）
docker compose up -d --build

# ログの確認（-f でフォロー）
docker compose logs -f app
```

### 開発中のよく使うコマンド

| 操作 | コマンド | 補足 |
| :--- | :--- | :--- |
| **コンテナ内でシェルに入る** | `docker compose exec app sh` | Alpineなので `bash` ではなく `sh` |
| **パッケージ追加** | `docker compose exec app npm install <pkg>` | コンテナ内で実行し、ロックファイルをホストに反映 |
| **DBマイグレーション実行** | `docker compose exec app npm run migrate` | アプリコンテナからDBコンテナへ接続して実行 |
| **一時停止（メモリ解放）** | `docker compose stop` | コンテナは残るので再開高速 |
| **完全削除（ボリューム含む）** | `docker compose down -v` | **DBデータも消えるので注意** |
| **不要なイメージ/キャッシュ削除** | `docker system prune -a` | ディスク圧迫時に実行 |

### デバッグのコツ：コンテナの中身を覗く

「コンテナ内ではどうなっているんだ？」と思ったら即座にシェルに入ります。

```bash
# app コンテナの中に入る
docker compose exec app sh

# 中で確認できること
# ls -la /app           # ソースコードがマウントされているか
# cat /etc/hosts        # 'db' が名前解決できるか
# env | grep DATABASE   # 環境変数が正しく入っているか
# npm list              # node_modules の状態
```

## ローカル開発で Docker を使うメリット：導入を決める判断材料

ここまで手順を見てきましたが、「ローカルに直接 Node.js と PostgreSQL を入れればよくない？」と思う方もいるでしょう。あえて Docker を使うメリットを整理します。

### 1. 環境の「完全な再現性」と「オンボーディングコストの激減」
- **新メンバー参加時**: `git clone` → `docker compose up -d` の **2コマンドで開発環境完成** します。README に「Nodeのバージョンは…、Postgresの設定は…」と書く必要がありません。
- **CI/CD との整合性**: ローカルと CI で **全く同じ Dockerfile / compose ファイル** を使い回せます。「ローカルでは通ったのに CI で落ちる」が劇的に減ります。

### 2. ホストOSを汚さない、切り替えが容易
- プロジェクトAは Node 18、プロジェクトBは Node 20、プロジェクトCは Python 3.11…といった時に、**ホストにバージョンマネージャを入れず、コンテナごと切り替えられます**。
- 実験的なライブラリを入れて壊しても `docker compose down -v` 一発で綺麗サッパリ元に戻せます。

### 3. 本番に近いミドルウェア構成をローカルで再現可能
- ローカルに PostgreSQL を入れるとバージョンや設定が本番とズレがちです。Dockerなら `image: postgres:15.4` のように **バージョンをピンポイントで固定** できます。
- Redis, Elasticsearch, MinIO (S3互換), LocalStack (AWSモック) など、**ローカルインストールが面倒なミドルウェアもサービス追加一行で導入**できます。

### 4. ネットワーク分離とポート競合の回避
- ホストの 3000 番ポートを複数プロジェクトで使いたいとき、Dockerならコンテナ側は全て 3000、ホスト側だけ `3001:3000`, `3002:3000` とマッピングを変えれば競合しません。

## よくあるハマりどころと対策（トラブルシューティング）

### 1. ファイル権限問題
**現象**: コンテナ内で作ったファイルが root 所有になり、ホスト側で削除・編集できない。
**対策**: `Dockerfile` で非 root ユーザーを作成し、`USER node` 等で切り替える。または `docker-compose.yml` で `user: "${UID}:${GID}"` とホストのユーザーIDを渡す。

### 2. ボリュームマウントのパフォーマンス
**現象**: Mac/Windows (Docker Desktop) で `node_modules` や巨大なビルド成果物をバインドマウントすると激重になる。
**対策**:
- `node_modules` は前述の **匿名ボリュームで除外** する。
- 次世代のファイルシステム共有 **
