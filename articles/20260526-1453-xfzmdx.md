## はじめに

## 前提：サンプルスキーマとデータ

以下の 3 テーブルを例にします。  

| users | カラム | 型 |
|------|--------|----|
| id   | 主キー | INT |
| name | ユーザー名 | VARCHAR(50) |
| department_id | 所属部署 | INT |

| departments | カラム | 型 |
|------------|--------|----|
| id         | 主キー | INT |
| name       | 部署名 | VARCHAR(50) |

| orders | カラム | 型 |
|-------|--------|----|
| id    | 主キー | INT |
| user_id | 発注者 | INT |
| amount | 金額 | DECIMAL(10,2) |
| created_at | 発注日時 | DATETIME |

```sql
-- テーブル作成（MySQL 例）
CREATE TABLE users (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL,
  department_id INT
);

CREATE TABLE departments (
  id INT PRIMARY KEY AUTO_INCREMENT,
  name VARCHAR(50) NOT NULL
);

CREATE TABLE orders (
  id INT PRIMARY KEY AUTO_INCREMENT,
  user_id INT NOT NULL,
  amount DECIMAL(10,2) NOT NULL,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

## 1. 基本の SELECT と WHERE

### 1-1. カラム指定と全取得

```sql
-- 全カラム取得
SELECT * FROM users;

-- 必要なカラムだけ取得
SELECT id, name FROM users;
```

### 1-2. 条件指定（WHERE）

```sql
-- 部署が「開発」のユーザーだけ取得
SELECT name FROM users
WHERE department_id = 1;

-- 金額が 500 円以上の受注を取得
SELECT * FROM orders
WHERE amount >= 500;
```

**ポイント**  
- `=` の他に `>`, `<`, `>=`, `<=`, `<>`（≠）が使える  
- `IN`, `BETWEEN`, `LIKE` で柔軟な検索が可能  

```sql
-- 複数条件
SELECT name FROM users
WHERE department_id IN (1,2) AND name LIKE '山%';
```

## 2. テーブル結合（JOIN）

### 2-1. INNER JOIN（内部結合）

ユーザーと所属部署名を結合して一覧にする例です。

```sql
SELECT u.id, u.name, d.name AS department
FROM users u
INNER JOIN departments d
  ON u.department_id = d.id;
```

### 2-2. LEFT JOIN（左外部結合）

部署に所属していないユーザーも含めて取得したい場合。

```sql
SELECT u.name, d.name AS department
FROM users u
LEFT JOIN departments d
  ON u.department_id = d.id;
```

### 2-3. 複数テーブルの JOIN

受注とユーザー、部署を同時に取得。

```sql
SELECT o.id AS order_id,
       u.name AS user_name,
       d.name AS department,
       o.amount,
       o.created_at
FROM orders o
JOIN users u   ON o.user_id = u.id
LEFT JOIN departments d ON u.department_id = d.id;
```

**実務での落とし穴**  
- `JOIN` の順序は結果に影響しないが、パフォーマンスはインデックスとテーブルサイズに依存する。  
- 必要なカラムだけを `SELECT` に列挙すると、ネットワーク帯域とメモリ使用量が減る。

## 3. 集計とグルーピング

### 3-1. 基本的な集計関数

| 関数 | 説明 |
|------|------|
| `COUNT(*)` | 行数 |
| `SUM(col)` | 合計 |
| `AVG(col)` | 平均 |
| `MAX(col)` / `MIN(col)` | 最大・最小 |

#### 部署別売上合計

```sql
SELECT d.name AS department,
       SUM(o.amount) AS total_sales,
       COUNT(o.id)   AS order_count
FROM orders o
JOIN users u   ON o.user_id = u.id
JOIN departments d ON u.department_id = d.id
GROUP BY d.name
ORDER BY total_sales DESC;
```

### 3-2. HAVING で集計結果を絞り込む

```sql
-- 売上合計が 2000 円以上の部署だけ
SELECT d.name, SUM(o.amount) AS total_sales
FROM orders o
JOIN users u ON o.user_id = u.id
JOIN departments d ON u.department_id = d.id
GROUP BY d.name
HAVING SUM(o.amount) >= 2000;
```

### 3-3. ウィンドウ関数（MySQL 8.0+, PostgreSQL, SQL Server）

ウィンドウ関数は「行ごとに集計結果を付与」でき、レポートで頻出します。

```sql
-- 各受注に対して、同ユーザーの累計金額を付与
SELECT o.id,
       u.name,
       o.amount,
       SUM(o.amount) OVER (PARTITION BY u.id ORDER BY o.created_at) AS running_total
FROM orders o
JOIN users u ON o.user_id = u.id
ORDER BY u.id, o.created_at;
```

## 4. サブクエリと共通テーブル式（CTE）

### 4-1. サブクエリ（副問い合わせ）

#### 例：直近 30 日間の受注だけを対象にした売上合計

```sql
SELECT SUM(amount) AS last_30_days_sales
FROM (
  SELECT amount
  FROM orders
  WHERE created_at >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
) AS recent;
```

### 4-2. CTE（WITH句）で可読性向上

CTE はサブクエリを名前付きで先に定義でき、複雑なロジックを段階的に書けます。

```sql
WITH recent_orders AS (
  SELECT *
  FROM orders
  WHERE created_at >= DATE_SUB(CURDATE(), INTERVAL 30 DAY)
),
dept_sales AS (
  SELECT d.id, d.name, SUM(o.amount) AS total
  FROM recent_orders o
  JOIN users u   ON o.user_id = u.id
  JOIN departments d ON u.department_id = d.id
  GROUP BY d.id, d.name
)
SELECT name, total
FROM dept_sales
WHERE total > 1000
ORDER BY total DESC;
```

**メリット**  
- 再利用が容易（同じサブクエリを何度も書かない）  
- デバッグがしやすい（`SELECT * FROM recent_orders` だけで中間結果を確認できる）  

### 4-3. EXISTS と IN の使い分け

```sql
-- EXISTS で「ユーザーが 1 件以上受注しているか」だけ確認
SELECT name
FROM users u
WHERE EXISTS (
  SELECT 1 FROM orders o WHERE o.user_id = u.id
);

-- IN で「特定の部署に所属するユーザー」一覧
SELECT name
FROM users
WHERE department_id IN (SELECT id FROM departments WHERE name LIKE '開%');
```

- `EXISTS` は行が存在すればすぐに評価が止まるため、パフォーマンスが高いケースが多い。  
- `IN` は集合が小さいときにシンプルに書ける。

## 5. 実務で役立つクエリパターン集

| パターン | 用途 | 例 |
|----------|------|----|
| **ページング** | 大量データの一部だけ取得 | `SELECT * FROM users ORDER BY id LIMIT 20 OFFSET 40;` |
| **重複除外** | 同一ユーザーの重複受注を除外 | `SELECT DISTINCT user_id FROM orders;` |
| **条件付き集計** | 部署別に「500 円以上」の受注数 | `SUM(CASE WHEN amount >= 500 THEN 1 ELSE 0 END)` |
| **自己結合** | 階層構造（上司-部下）を持つテーブル | `JOIN employees e2 ON e1.manager_id = e2.id` |
| **日付集計** | 月別売上推移 | `GROUP BY DATE_FORMAT(created_at, '%Y-%m')` |

## まとめ

- **SELECT / WHERE / JOIN** はデータ取得の土台。まずはシンプルに書いて、必要なカラム・条件だけを絞り込む習慣をつけましょう。  
- **集計** では `GROUP BY` と集計関数、必要なら `HAVING` を組み合わせ、ウィンドウ関数で「行単位の累積」も実装できます。  
- **サブクエリ / CTE** はロジックを分割し可読性と保守性を向上させます。特に複数段階の集計やフィルタが必要なときは CTE が強力です。  
- 実務では「ページング」「重複除外」「条件付き集計」などのパターンが頻出します。この記事で紹介した例をベースに、テーブル構造に合わせてカスタマイズしてください。

SQL は「書けばすぐに結果が出る」言語です。まずは手元のデータベースで上記クエリを走らせ、結果を確認しながら感覚をつかんでみてください。  
繰り返し書くうちに、**「この情報が欲しいときはどのテーブルをどう結合すればいいか」** が自然に分かるようになります。  

> 次のステップは、**インデックス設計** と **EXPLAIN** でクエリの実行計画を確認し、パフォーマンスチューニングに挑戦することです。
