## AIツールを取り囲む背景と前提

従来のコピペ検索に比べ、Claude Code のようなエージェント型ツールは「ファイルを読み込んで修正まで提案してくれる」ため圧倒的に速いです。しかし、以下の前提を忘れると技術力が伸び悩みます。

- AI は「文脈がある前提」で答えを作る
- AI の出力はあくまで「提案」であり、責任は書いた人（あなた）にある
- 実務では「なぜその実装か」を説明できないコードはレビューで返される

たとえば次のような状況です。

```text
あなた「ログイン機能を作って」
Claude Code「Express + JWT で以下のように実装しました」
```

これで動いたとしても、JWT の有効期限設定やトークン保存先を自分で説明できなければ、本番障害時に何もできません。

## 頼りすぎないための3つの使い分け

### 1. AIに任せてよい範囲

以下は AI に任せてもリスクが低く、むしろ効率が上がります。

- ボイラープレート（CRUD の雛形など）
- 正規表現の作成や単体での変換スクリプト
- エラーメッセージの候補出し
- 既存コードの簡単なリファクタ提案

例えば、日付を `YYYY-MM-DD` に整形する関数を頼む場合：

```javascript
// Claude Code に依頼して生成された例
function formatDate(date) {
  const y = date.getFullYear();
  const m = String(date.getMonth() + 1).padStart(2, '0');
  const d = String(date.getDate()).padStart(2, '0');
  return `${y}-${m}-${d}`;
}
```

この程度であれば、生成結果を読んで「なるほど padStart を使うのか」と学びにつなげられます。

### 2. 自分で書くべき範囲

以下は必ず自分の頭で考え、コードを書くかレビューしてください。

- 認証・認可といったセキュリティ関連
- ビジネスロジックの中核（金額計算など）
- 他システムとの結合部分
- パフォーマンスに直結するクエリ

これらは「AIが書いたから大丈夫」にできず、自分で単体テストを書く習慣をつけましょう。

```python
# 自分で書くべき金額計算の例（消費税込み）
def calc_with_tax(price, rate=0.10):
    # 丸め誤差を意識して整数演算にする等、自分で判断する
    return int(price * (1 + rate))
```

### 3. 聞き方のコツ：目的と制約をセットで伝える

AI に「作って」とだけ言うのは NG です。以下のように伝えます。

- やりたいこと（目的）
- 技術スタック・既存の構成
- 守ってほしい制約（例：外部ライブラリ禁止）

悪い例：

```text
フォームをバリデーションして
```

良い例：

```text
React + TypeScript の既存プロジェクトで、メールアドレスとパスワードの
フォームを作りたい。バリデーションは react-hook-form を使い、
ライブラリは追加せず自前の正規表現で行って。
```

これで出力のブレが減り、自分の学習にもなります。

## 実務での具体的な流れ

日常の開発では次のステップで AI を使うのがおすすめです。

1. 自分で要件を箇条書きにする
2. Claude Code に「草案」を作らせる
3. 自分で読み、不明点をドキュメントで確認する
4. 必要なら自分で書き直してコミットする

こうすることで「AI が書いたけど理解している」状態を保てます。

## まとめ

Claude Code などの AI ツールは強力ですが、駆け出しエンジニアほど「頼りすぎないバランス」が重要です。ボイラープレートや補助的な変換は AI に任せ、セキュリティや中核ロジックは自分で書きましょう。また、目的と制約をセットで伝える「聞き方」を意識すれば、出力の質も自分の理解度も高まります。

AI を「代わりに書く人」ではなく「ペアの相棒」として使い、実務の勘所を自分のものにしていきましょう。
