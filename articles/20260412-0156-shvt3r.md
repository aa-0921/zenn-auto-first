## 背景・前提：なぜエンジニア転職を決めたのか

### 1. 現職の限界感

- **業務の単調さ**：事務系の仕事を 5 年ほど続けてきましたが、毎日同じルーチンに追われ、成長実感が乏しかった。
- **テクノロジーへの興味**：社内ツールの改善提案をした際に、プログラミングで自動化できると知り、実装に挑戦したことが転機に。

### 2. 市場環境の変化

- **リモートワークの普及**：場所に縛られず働ける環境が増え、転職ハードルが下がった。
- **需要の拡大**：Web アプリやクラウドサービスの需要が高く、未経験者歓迎のポジションが多数出てきた。

### 3. 自己分析

| 項目 | 結果 |
|------|------|
| 興味・関心 | Web フロントエンド、API 開発 |
| 強み | 論理的思考、課題発見力 |
| 弱み | コーディング経験がほぼ 0 |
| 目標 | 1 年以内に実務でフルスタックエンジニアになる |

この自己分析をベースに、**「未経験からでも実務に通用するスキルセットを 1 年で構築」**するロードマップを描きました。

## 学習の進め方：実践的にスキルを身につける 3 つのステップ

### Step 1. 基礎は「1 週間 1 機能」スプリントで固める

> **ポイント**：完結した小さな機能を作りながら学習することで、達成感と復習コストを最小化。

| 期間 | 学習テーマ | 主なアウトプット |
|------|------------|-------------------|
| 1 週目 | HTML/CSS 基礎 | 静的な自己紹介ページ |
| 2 週目 | JavaScript 基礎 | 画像スライダー |
| 3 週目 | Git/GitHub 基礎 | リポジトリ作成・プルリク |
| 4 週目 | Node.js/Express 入門 | TODO API（CRUD） |
| 5 週目 | データベース（PostgreSQL） | TODO API に永続化 |
| 6 週目 | フロントフレーム（React） | TODO アプリの UI |

#### コード例：Node.js + Express で簡易 TODO API

```js
// src/app.js
const express = require('express');
const app = express();
app.use(express.json());

let todos = [];

app.get('/todos', (req, res) => res.json(todos));
app.post('/todos', (req, res) => {
  const todo = { id: Date.now(), ...req.body };
  todos.push(todo);
  res.status(201).json(todo);
});

app.listen(3000, () => console.log('Server started on :3000'));
```

### Step 2. 「学んだことをアウトプット」＝ブログ + OSS コントリビューション

- **ブログ**: 学習したことを毎週 1 回は技術ブログにまとめた。検索エンジンからの流入が増え、面接官に「アウトプットできるエンジニア」と評価された。
- **OSS**: `awesome-for-beginners` に掲載されている日本語ドキュメントの翻訳に参加。PR がマージされるたびに GitHub の貢献度が上がり、ポートフォリオとして有効活用。

### Step 3. 「実務に近い課題」を受託案件で経験する

副業プラットフォーム（CrowdWorks）で **小規模の Web アプリ受託** を 2 件受注。  
- **案件 1**: 予約管理システム（React + Firebase） → 認証・リアルタイム DB を実装。  
- **案件 2**: 社内ツールの自動化（Node.js + Slack API） → 定期レポート送信機能を構築。

受託案件は **「成果物を納品する」** という実務感覚を養うと同時に、クライアントとの要件定義・見積もり経験が転職面接で大いに役立ちました。

## 転職活動の実践：書類・面接・条件交渉の具体的手順

### 1. ポートフォリオサイトの作成

- **構成**: `index.html`（自己紹介） → `projects.html`（受託案件・GitHub リポジトリ） → `blog.html`（技術記事一覧）。
- **デプロイ**: Vercel（無料プラン）を利用し、デプロイ自動化（GitHub Actions）を設定。  
  → 「実際に動くサービスが見える」ことで、書類通過率が約 1.5 倍に向上。

### 2. 書類選考の通過率を上げるテクニック

| 書類項目 | 工夫したポイント |
|----------|-----------------|
| 職務経歴書 | 「未経験」ではなく「〇〇 で課題解決した経験」＋「学習成果」＝具体的数値で示す |
| ポートフォリオ | 受託案件の URL と GitHub リポジトリを両方掲載 |
| カバーレター | 「なぜ御社のプロダクトに共感したか」＋「自分のスキルで貢献できる点」 を 3 行以内で簡潔に |

### 3. 面接でのエピソードトーク

- **質問例**: 「未経験者が入社したらどのように成長させますか？」  
- **回答例**:  
  「① 1 週間で小さな機能を実装し、コードレビューを受ける。  
   ② 2 週間ごとにペアプログラミングを実施し、チームの開発フローに慣れる。  
   ③ 1 ヶ月ごとに振り返りミーティングで課題を可視化し、次のスプリントに反映させる」  

このように、**具体的なスケジュールとアウトカム** を示すと、面接官に「即戦力になり得る」印象を与えられます。

### 4. 条件交渉のコツ

- **年収**: 市場平均（約 600 万円）をリサーチし、*「スキルと経験に見合う」* と根拠を示す。  
- **リモート**: 「週 3 日はリモート希望」→ 会社側の制度と合わせて「生産性向上の根拠」も添える。  
- **研修制度**: 「入社後 3 ヶ月は社内ハッカソンに参加させて欲しい」など、成長意欲をアピール。

## 現場で感じた手応えと成長の実感

### 1. コードレビューから学んだこと

- **指摘のパターン**: 「命名規則」「エラーハンドリング」→ 1 週間で同じミスが減少。  
- **改善策**: ESLint と Prettier をプロジェクトに導入し、**CI** で自動チェック。  

### 2. アジャイル開発に慣れた感覚

- **スプリント**: 2 週間単位のスプリントで **バックログ** を管理。  
- **デイリースクラム**: 15 分以内で「昨日やったこと・今日やること・障害」を共有。  
  → チーム全体の可視化が進み、**課題の早期発見** が可能に。

### 3. パフォーマンスチューニングの実体験

- **課題**: API のレスポンスが 1.2 秒 → 目標は 0.5 秒以下。  
- **対策**:  
  1. クエリ最適化（インデックス追加）  
  2. キャッシュ層（Redis）導入  
  3. N+1 問題の解消  

結果、**平均レスポンス 0.46 秒** に短縮。チームから「改善が顕著」評価され、プロジェクトリーダーからも信頼を得た。

## まとめ

- **年齢は障壁ではなく、経験の深さを活かす材料** になる。  
- **小さな成功体験（1 週間 1 機能）** を積み重ねることで、学習のモチベーションと実務感覚が同時に養える。  
- **アウトプット（ブログ・OSS）** がポートフォリオの価値を高め、転職活動の武器になる。  
- **現場ではコードレビュー・アジャイル開発・パフォーマンス改善** といった実務経験が、未経験者でも即戦力として評価されるポイントになる。

30 歳を超えてエンジニアになることは決して遅くありません。  
**「学ぶ姿勢」と「アウトプットの習慣」** を軸にすれば、キャリアチェンジは十分に成功可能です。  
この記事が、同じようにキャリア転換を考える皆さんの一歩になることを願っています。
