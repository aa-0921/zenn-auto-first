## Pages Router と App Router の違い

### ルーティング構造
- **Pages Router** はファイルシステムベースのルーティングを採用しており、`pages/` ディレクトリ内にページコンポーネントを配置することでルートが定義されます。
- **App Router** では `app/` ディレクトリを使用し、フォルダ階層がネストされたルーティングを直感的に表現できます。また、`app/layout.tsx` や `app/loading.tsx` など、ページ以外の特殊なコンポーネントも定義可能です。

```tsx
// Pages Router の例
// pages/about.tsx
export default function About() {
  return <h1>About Us</h1>;
}
```

```tsx
// App Router の例
// app/about/page.tsx
export default function About() {
  return <h1>About Us</h1>;
}
```

### データ取得とキャッシュ
- Pages Router では `getServerSideProps` や `getStaticProps` を使用する必要がありましたが、App Router では `fetch` と `useDeferredValue` を組み合わせた新しいデータ取得パターン（オプションの `use` によるデータフェッチ）が導入されています。これにより、コンポーネント内でのサーバーサイドデータ取得がより自然に行えます。

### サーバーコンポーネントとクライアントコンポーネントの可用性
- **Pages Router** では、全体的にクライアントコンポーネントとしてレンダリングされるため、サーバーコンポーネントを明示的に記述できません。
- **App Router** では、`use client` ディレクティブを記述しないコンポーネントが自動的にサーバーコンポーネントとして動作します。これにより、ページのパフォーマンスを向上させるための初期レンダリングや部分的な静的生成が容易になります。

## サーバコンポーネントとクライアントコンポーネントの使い分け

### サーバコンポーネント（SC）のメリット
- **初期レンダリング**: HTML がサーバーから直接返されるため、クライアントでの再レンダリング負荷を軽減できます。
- **セキュリティ**: 外部 API やデータベースへのアクセスをサーバーに集中させ、秘密鍵をクライアントに漏洩するリスクを低減します。
- **SEO**: 検索エンジンがサーバー上でレンダリングされたコンテンツを正しくインデックス化できます。

### クライアントコンポーネント（CC）が必要な場面
- **ユーザーインタラクション**: クリックや入力など、ユーザーによる状態変化を伴うUI。
- **ライブラリとの連携**: クライアントサイドのライブラリ（例：Chart.js, Leaflet）の使用。
- **再利用可能なコンポーネント**: 複数のページで共有されるが、内部で状態管理を行うコンポーネント。

#### 実践的な例

```tsx
// app/page.tsx (サーバーコンポーネント)
export default async function Home() {
  const res = await fetch('https://api.example.com/posts', { cache: 'force-cache' });
  const posts = await res.json();

  return (
    <ul>
      {posts.map((post: { id: number; title: string }) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

```tsx
// components/Quiz.tsx (クライアントコンポーネント)
'use client';

import { useState } from 'react';

export default function Quiz() {
  const [count, setCount] = useState(0);

  return (
    <button onClick={() => setCount(c => c + 1)}>
      Clicked {count} times
    </button>
  );
}
```

`app/page.tsx` では `use client` がないため、サーバーコンポーネントとしてレンダリングされます。一方、`components/Quiz.tsx` では `'use client'` を先頭に記述することで、クライアントコンポーネントとして強制されます。

## 実践的なコード例：Pages Router から App Router への移行

以下は、シンプルなブログトップページを Pages Router から App Router に移行する際の具体的な変更点です。

### Pages Router の場合

```tsx
// pages/index.tsx
import type { GetStaticProps, NextPage } from 'next';
import type { Post } from '@/types';

interface HomeProps {
  posts: Post[];
}

export const getStaticProps: GetStaticProps = async () => {
  const res = await fetch('https://api.example.com/posts');
  const posts = await res.json();

  return { props: { posts } };
};

const Home: NextPage<HomeProps> = ({ posts }) => (
  <ul>
    {posts.map(p => <li key={p.id}>{p.title}</li>)}
  </ul>
);

export default Home;
```

### App Router の場合

```tsx
// app/page.tsx
export default async function Home() {
  const res = await fetch('https://api.example.com/posts', { cache: 'force-cache' });
  const posts = await res.json();

  return (
    <ul>
      {posts.map(p => (
        <li key={p.id}>{p.title}</li>
      ))}
    </ul>
  );
}
```

**変更点のポイント**

1. `pages/` ディレクトリを `app/` ディレクトリに移動します。
2. ルーティング用のファイル名（`index.tsx` → `page.tsx`）を調整します。
3. `getStaticProps` のようなヘルパーは `fetch` に置き換わります。
4. 明示的に `'use client'` を記述しない限り、すべてのコンポーネントがデフォルトでサーバーコンポーネントとして動作します。

## まとめ

- Next.js における **Pages Router** と **App Router** の主な違いは、ルーティング構造、データ取得の仕組み、およびサーバーコンポーネントの可用性にあります。
- **サーバーコンポーネント** は、パフォーマンス、セキュリティ、SEO に優れていますが、**クライアントコンポーネント** はユーザーインタラクションや状態管理が必要な場合に必要です。
- マイグレーション時には、フォルダ構造の変更、`fetch` によるデータ取得の採用、および `'use client'` ディレクティブの適切な使用に注意しましょう。

これらのポイントを押さえることで、初心者でも Next.js の最新アーキテクチャを効果的に活用できるようになり、Pages Router 時代のつまずきがちなポイントを克服できます。
