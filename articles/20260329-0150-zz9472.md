# コードレビューで学んだこととメンタルの保ち方

## はじめに

コードレビューは、ソフトウェア開発において避けては通れないプロセスです。しかし、多くのエンジニアがコードレビューを受ける際に感じるのは、緊張や不安、時には落ち込みです。私自身も駆け出しの頃は、レビューで指摘されるたびに自信を失いかけた経験があります。

この記事では、私がコードレビューを通じて学んだことと、メンタルを保つための具体的な方法を体験談を交えながら紹介します。コードレビューを「批判」ではなく「成長の機会」と捉える視点の転換が、エンジニアとしての成長速度を大きく変えることを実感しています。

## コードレビューに対する誤解と正しい向き合い方

### レビューは「批判」ではない

初めて本格的なコードレビューを受けたとき、私は「自分のコードが否定されている」と感じました。しかし、これは大きな誤解です。コードレビューは、個人の能力を評価する場ではなく、プロダクトの品質を向上させるための協働作業です。

私が学んだのは、レビュワーは「あなたのコード」ではなく「コードそのもの」に注目しているということです。これは、コードを書いた本人の人格や能力とは切り離して考えるべきことです。この視点を持つことで、指摘を受けたときのダメージが格段に減ります。

### 指摘の種類を理解する

コードレビューで受ける指摘は大きく分けて3つあります：

1. **必須修正**: バグ、セキュリティホール、パフォーマンス問題など
2. **改善提案**: 可読性の向上、設計の改善、ベストプラクティスの適用
3. **提案・意見**: 別の実装方法、異なるアプローチ

この分類を理解すると、すべての指摘に同じ重さを感じる必要はないことに気づきます。必須修正は素直に受け入れ、改善提案は学びの機会とし、提案・意見は参考情報として捉えるのが健康的です。

## 指摘を受け止めるメンタル術

### 即座に反論しない

私が失敗した経験の一つは、レビューで指摘を受けた直後に「いや、それはこういう意図で…」と反論してしまったことです。これでは建設的な議論になりません。

今では、指摘を受けたらまず「ありがとうございます」と返し、24時間以上時間を置いてから改めて考えるようにしています。冷静になってから見ると、指摘の意図が理解できたり、より良い解決策が見つかったりすることが多いです。

### 質問の技術

レビューで理解できない指摘を受けたとき、適切な質問をすることは非常に重要です。私が実践しているのは「なぜ」ではなく「どのように」から質問を始めることです。

例えば：
- ✗ 悪い例: 「なぜこの書き方はダメなんですか？」
- ✓ 良い例: 「この書き方のどこに問題があるのか、具体的な例を教えていただけますか？」

後者の方が、レビュワーに具体的な説明を促し、自分の学びにつながります。また、質問することで「自分は学ぶ姿勢がある」とレビュワーに印象づけることもできます。

### 自分の成長記録をつける

コードレビューの指摘を記録し、定期的に振り返る習慣をつけることをおすすめします。私は月に一度、以下のような形式で振り返りをしています：

```markdown
## 今月のコードレビュー振り返り

### 改善できた点
- ネストが深くなる問題: 3回指摘 → 現在は意識して平坦化を心がけている

### まだ課題の点
- 命名規則: 変数名の意味が不明瞭と5回指摘

### 学んだこと
- 早期リターンの重要性
- ドキュメントコメントの書き方
```

このように記録することで、自分の成長を客観的に確認できます。以前指摘されたことが改善されているのを見ると、非常にモチベーションになります。

## レビューをする側の心構え

### 建設的なフィードバックの与え方

コードレビューを受ける立場だけでなく、レビューをする立場になったときの心構えも重要です。私が心がけているのは、批判ではなく提案の形でフィードバックをすることです。

```diff
- この関数は20行を超えています。分割しましょう
+ この関数は20行を超えています。責務が複数ありそうなので、分割するとすっきりしそうですね。例えば〜（具体例）

- 変数名が分かりにくい
+ 変数名が分かりにくいです。例えば「tmp」→「filteredUsers」のようにすると、中身が一目で分かります
```

後者の方が、相手に具体的なイメージを持たせ、学びにつながります。

### 感謝の文化をつくる

コードレビューは、指摘し合う場だけでなく、感謝し合う場にもできます。私はレビューで学んだことや、良いコードだと思った部分には必ずコメントを残すようにしています。

```
// このロジックの考え方、勉強になりました！
// この変数名、意図が伝わってきて分かりやすいです
```

このようなポジティブなフィードバックがあると、レビュー全体の雰囲気が良くなり、建設的な議論がしやすくなります。

## コードレビューを成長の機会にするコツ

### レビューを学習の場と捉える

コードレビューで指摘される内容は、そのまま「学ぶべきこと」のリストです。私はレビューで指摘された内容を、以下のように学習計画に落とし込んでいます。

1. 指摘された技術用語を調べる
2. 関連する公式ドキュメントや書籍を読む
3. 別のプロジェクトで実践してみる
4. 同僚に教える

このサイクルを回すことで、同じ指摘を繰り返し受けることが格段に減ります。

### ペアレビューを活用する

コードレビューは、非同期で行うこともできますが、同期的なペアレビューも非常に有効です。私は週に一度、15分程度のペアレビューの時間を設けています。

ペアレビューの利点は、リアルタイムで質問や議論ができることです。特に設計に関する議論は、テキストだけでは伝わりにくい部分があります。画面共有しながら一緒にコードを見ることで、考え方の違いを理解し合えます。

### レビュー後の振り返り

レビューが終わった後、1週間ほど経ってから改めて自分のコードを見返す習慣をつけることをおすすめします。そのときに「なぜこの指摘を受けたのか」「どのような意図だったのか」を自分なりに整理すると、理解が深まります。

私はこの振り返りを、以下のような簡単なテンプレートで行っています：

```markdown
## レビュー後の振り返り（1週間後）

### 修正した点
- [ ] 変数名の改善
- [ ] 関数の分割

### 学びになった点
- 早期リターンの重要性
- エラーハンドリングのベストプラクティス

### 次回への改善点
- コメントの書き方
- テストカバレッジの考え方
```

## まとめ

コードレビューは、技術的なスキルアップだけでなく、コミュニケーション能力や謙虚さを育む貴重な機会です。大切なのは、レビューを「自分のコードが否定される場」ではなく、「一緒にプロダクトを良くする場」と捉える視点の転換です。

私が実践しているメンタル術をまとめると：

1. 指摘はコードへのフィードバックであり、自分への評価ではない
2. 即座に反論せず、一度受け止めてから考える
3. 質問の技術を身につける
4. 自分の成長記録をつける
5. レビューを学習の場と捉える

これらの考え方と実践を続けることで、コードレビューは苦痛な時間から、楽しみな学習の時間へと変わります。そして何より、チーム全体の技術力向上につながり、より良いプロダクトを生み出す原動力になります。

最後に、コードレビューは「完璧を目指す」のではなく、「昨日より今日、少しでも良くなる」ことを積み重ねるプロセスです。完璧を目指すあまり、前に進めなくなることのないように、自分自身に優しくありましょう。
