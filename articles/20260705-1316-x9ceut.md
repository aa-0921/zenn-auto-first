## 背景・前提
### テーブル構造
| テーブル名 | 主キー | カラム | 備考 |
|---|---|---|---|
| patients | id | name, birthdate | 患者情報 |
| appointments | id | patient_id, doctor_id, appointment_date | 予約情報 |
| doctors | id | name, specialty | 医師情報 |
| prescriptions | id | appointment_id, medication, dosage | 処方情報 |

> **備考**  
> - `patient_id` は `patients.id` を参照  
> - `doctor_id` は `doctors.id` を参照  
> - `appointment_id` は `appointments.id` を参照  

この構造を前提に、実務で頻出するクエリを段階的に紹介していきます。

## 本文
### 1. 基本的な SELECT と WHERE
まずは単純に患者の情報を取得し、条件で絞り込みます。

```sql
-- 30歳以上の男性患者を取得
SELECT id, name, birthdate
FROM patients
WHERE birthdate <= DATE_SUB(CURDATE(), INTERVAL 30 YEAR)
  AND gender = 'M';
```

ポイントは `DATE_SUB` で年齢計算を行い、`WHERE` で複数条件を組み合わせる点です。  
実務では `AND` と `OR` の優先順位に注意し、必要に応じて括弧で明示します。

### 2. JOIN を使った複数テーブル結合
予約情報と医師情報を結合し、特定の診療日帯で予約が入っている医師を抽出します。

```sql
-- 2024年7月の予約がある医師と患者名を取得
SELECT d.name AS doctor_name,
       p.name AS patient_name,
       a.appointment_date
FROM appointments a
JOIN doctors d ON a.doctor_id = d.id
JOIN patients p ON a.patient_id = p.id
WHERE a.appointment_date BETWEEN '2024-07-01' AND '2024-07-31';
```

- `JOIN` は結合条件硬直化のために `ON लगाए` で指定します。  
- `INNER JOIN`（デフォルト）は両テーブルに存在する行のみ取得。  
- `LEFT JOIN` を使うと、医師に予約が無い場合でも医師情報を取得できます。

```sql
-- 予約が無い医師もリストアップ
SELECT d.id, d.name
FROM doctors d
LEFT JOIN appointments a ON d.id = a.doctor_id
WHERE a.id IS NULL;
```

### 3. 集計関数と GROUP BY
月別の予約件数や_COMPONENT(curr) を集計し、診療科目別にまとめます。

```sql
-- 月別予約件数
SELECT YEAR(a.appointment_date) AS yr,
       MONTH(a.appointment_date) AS mon,
       COUNT(*) AS total_appointments
FROM appointments a
GROUP BY yr, mon
ORDER BY yr, mon;
```

- `GROUP BY` の列は集計対象外であり、エイリアスは `GROUP BY` で参照可能。  
- `HAVING` を使って集計後のフィルタリングを行うことも重要です。

```sql
-- 予約件数が10件以上の診療科目
SELECT d.specialty, COUNT(*) AS appointment_count
FROM appointments a
JOIN doctors d ON a.doctor_id = d.id
GROUP BY d.specialty
HAVING appointment_count >= 10;
```

### 4. サブクエリの活用
サブクエリは複雑な条件を分解する際に便利です。以下では、ある患者が過去に受診した医師を取得します。

```sql
-- 患者ID 5 が受診した全医師をリストアップ
SELECT DISTINCT d.id, d.name
FROM doctors d
WHERE d.id IN (
    SELECT a.doctor_id
    FROM appointments a
    WHERE a.patient_id = 5
);
```

#### 4-1. スカラーサブクエリ
単一値を返すサブクエリを使って、最新の診療日を取得します。

```sql
-- 最新の診療日
SELECT name,
       (SELECT MAX(appointment_date)
        FROM appointments
        WHERE patient_id = p.id) AS last_visit
FROM patients p;
```

#### 4-2. コリレーテッドサブクエリ
外側の行に依存するサブクエリです。ここでは、各患者ごとに自分の予約件数を表示します。

```sql
SELECT p.id, p.name,
       (SELECT COUNT(*) 
        FROM appointments a
        WHERE a.patient_id = p.id) AS appointment_count
FROM patients p;
```

#### 4-3. EXISTS を使った効率的 дошт
大量データで `IN` より `EXISTS` の方が高速なケースがあります。

```sql
-- 過去に処方を受けた患者のみfól
SELECT p.id, p.name
FROM patients p
WHERE EXISTS (
    SELECT 1
    FROM prescriptions pr
    JOIN appointments a ON pr.appointment_id = a.id
    WHERE a.patient_id = p.id
);
```

### 5. 実務でのパフォーマンス対策
1. **インデックス**  
   - `WHERE` で頻繁に使うカラム（例：`patient_id`、`appointment_date`）にインデックスを付与。  
   - `JOIN` 条件のカラムにも必ずインデックスを設定します。

2. **必要最低限のカラムを選択**  
   - `SELECT *` は避け、必要なカラムだけを指定。  
   - 大量データ転送を抑えることでネットワーク・ストレージ負荷が減少します。

3. **クエリ計画の確認**  
   - `EXPLAIN` を使って実行計画を確認し、スキャンタイプ（インデックススキャン vs テーブルスキャン）を最適化します。

4. **バルク処理の分割**  
   - 大きな更新/削除は `LIMIT` と `OFFSET` を使って段階的に実行し、ロック競合を減らします。

## まとめ
- **SELECT・WHERE** は基本の出発点。条件式は読みやすさを意識し、必要に応じて括弧で優先順位を明示。  
- **JOIN** で複数テーブルを結合し、必要な情報を取得。`LEFT JOIN` で欠損データも扱うテクニックを習得。  
- **集計関数 + GROUP BY** で月別、科目別्गなどの統計情報を作成。`HAVING` で集計後フィルタも忘れずに。  
- **サブクエリ** は複雑条件を分解し、ロジックを明確に。`EXISTS` はパフォーマンス面で有利。  
- 実務では **インデックス設計** と **実行計画の確認** がクエリ性能を左右します。  

これらのパターンを実際のプロジェクトで使いながら、テストと最適化を繰り返すことで、実務に直結する SQL スキルを身につTitle.
