## Next.js 13以降の新機能とその混乱ポイント

Next.js は現在、App Router（App Router）とサーバーコンポーネント（Server Components）を標準として推奨していますが、これらの新機能は初心者にとって理解が難しい場合があります。特に、App Router は従来の Pages Router（Pages Router）と大きく異なるルーティング構造を採用し、サーバーコンポーネントとクライアントコンポーネントの境界線が曖昧です。この記事では、初心者がつまずきやすいポイントを整理し、実務で役立つ具体的な知見を提供します。

## App Router と Pages Router の本質的な違い

### 1. ルーティングの構造
App Router はファイルベースのルーティングを採用しており、ディレクトリ構造自体がルートを定義します。例えば、`pages/about.js` は `/about` というパスにマッピングされます。一方、Pages Router は `_app.js` と `_layout.js` を通じてグローバルなレイアウトや初期化処理を管理していました。

- **App Router の例**  
  ```javascript
  // about.js
  export default function About() {
    return <h1>About Page</h1>;
  }
  ```
  このファイルを `pages/about.js` に配置すると、自動的に `/about` へアクセス可能になります。

- **Pages Router の例**  
  ```javascript
  // pages/about.js
  export default function About() {
    return <h1>About Page</h1>;
  }
  ```
  Pages Router では、ディレクトリ構造自体がルートを定義せず、ファイル名がパスに影響を与えません。

### 2. データフェッチングの方法
App Router では、データフェッチングは `load`、`generate`、`revalidate` という関数をコンポーネント内で定義します。一方、Pages Router では `getStaticProps` や `getServerSideProps` などの関数を使ってデータを取得していました。

- **App Router のデータフェッチング例**  
  ```javascript
  // about.js
  export async function load() {
    const res = await fetch('https://api.example.com/data');
    return res.json();
  }
  ```

- **Pages Router のデータフェッチング例**  
  ```javascript
  // pages/about.js
  export async function getStaticProps() {
    const res = await fetch('https://api.example.com/data');
    return { props: await res.json() };
  }
  ```

### 3. エラーハンドリングの違い
App Router では、エラーを処理するために `ErrorBoundary` コンポーネントを使用します。一方、Pages Router では `getStaticProps` や `getServerSideProps` 内でエラーをキャッチする方法が必要でした。

- **App Router のエラーハンドリング例**  
  ```javascript
  // ErrorBoundary.js
  export default function ErrorBoundary() {
    return (
      <div>
        {typeof error !== 'undefined' && <p>Error: {error.message}</p>}
      </div>
    );
  }
  ```

- **Pages Router のエラーハンドリング例**  
  ```javascript
  // pages/about.js
  export async function getStaticProps() {
    try {
      // データフェッチング
    } catch (error) {
      return { props: {}, error: error };
    }
  }
  ```

## サーバーコンポーネントとクライアントコンポーネントの活用タイミング

### サーバーコンポーネントのメリット
サーバーコンポーネントはサーバーサイドでレンダリングされるため、クライアントサイドの JavaScript を減らし、パフォーマンスを向上させることができます。特に、静的なコンテンツやAPI呼び出しが必要な場合に有効です。

- **サーバーコンポーネントの例**  
  ```javascript
  // serverComponent.js
  export default function ServerComponent() {
    const data = fetch('https://api.example.com/data'); // サーバーサイドで処理
    return <div>{data}</div>;
  }
  ```

### クライアントコンポーネントの用途
クライアントコンポーネントは、ユーザーインターフェースの動的な操作やクライアントサイドの処理が必要な場合に使用されます。例えば、フォームの検証やアニメーションなどです。

- **クライアントコンポーネントの例**  
  ```javascript
  // clientComponent.js
  export default function ClientComponent() {
    const [count, setCount] = useState(0);
    return (
      <button onClick={() => setCount(count + 1)}>Count: {count}</button>
    );
  }
  ```

### 2つのコンポーネントの使い分けのポイント
- サーバーコンポーネントは、API呼び出しやデータベースアクセスが必要な場合に活用。
- クライアントコンポーネントは、UIのインタラクティブな処理やブラウザ固有の機能が必要な場合に使用。

## まとめ

Next.js 13以降では、App Router とサーバーコンポーネントの導入により、アプリケーションの構築方法が大きく変わりました。初心者は、従来の Pages Router の習慣にとらわれず、新しいルーティング構造やコンポーネントの種類を理解することが重要です。特に、データフェッチングやエラーハンドリングの方法が変わっている点に注意し、実際のプロジェクトで試行錯誤しながら学ぶことが推奨されます。App Router のシンプルさとサーバーコンポーネントのパフォーマンス向上という強みを活かすことで、より効率的なアプリケーションを開発できるでしょう。
