## 指摘の受け止め方：防衛的思考から建設的思考へ  
### ① 「これは間違っている」と思わず読む  
レビューで「変数名が意味不明だ」と指摘されたとき、「自分は変数名を考えていたのに…」と防御的になるのは自然です。しかし、まずは「この指摘には何の意図があるのか？」を意識します。例えば、指摘者が「読み手に理解しやすいコードを書かせたい」と考えていた可能性があります。  

**具体例**  
```javascript  
// 元コード  
if (v > 0) {  
  // 高速化のための処理  
}  
// レビュー指摘：変数名「v」は意味不明  
```  
この場合、「v」が「value」を意味していたとしても、指摘者にとっては「ビジネスロジックと関係ない符号」と解釈されかねません。この場で「v」を「isPositive」に改名することで、両者の意図を合わせられます。  

### ② 「何が問題か」を具体的に尋ねる  
「なぜこの方法が選ばれたのか？」と質問する際は、**「提案する」ではなく「理解する」目的」に焦点を当てます。例えば、レビューで「API呼び出しを非同期化すべき」と指摘されたとき、  
```javascript  
// 元コード  
getData().then(() => { /* 処理 */ });  
// レビュー指摘：Promiseで非同期化すべき  
```  
と指摘された場合、「なぜPromise化が重要なのか？」と具体的に聞くことで、指摘者の考えを深く掘り下げられます。

## 質問の仕方：知識の共有を促すコミュニケーション  
### ① 「疑問」を「仮説」に変える  
質問は「これは間違っているか？」ではなく、「この文脈でこう考えた結果、どうなるか？」という形で表現します。  

**例**  
```javascript  
// レビュー指摘：インターフェースの定義を抽出すべき  
// 質問例  
// 「この関数がインターフェース化されることで、テストがどう簡単になるか教えてください」  
```  
このように、「指摘者の意図」を掘り下げる質問は、技術的な議論を深め、双方の理解を高めます。  

### ② 「なぜこうしたのか？」と背景を尋ねる  
技術的な指摘には、プロジェクトの歴史やチームの慣行が反映されていることがあります。  

**例**  
```javascript  
// レビュー指摘：ログ出力をエラーハンドリングに統一すべき  
// 質問例  
// 「このプロジェクトでログ形式を統一する理由は何ですか？」  
```  
指摘者の背景を知ることで、ただの「規則遵守」ではなく「プロジェクト固有の最適化」を理解しやすくなります。

## レビューを成長の機会にする：体験談を交えて  
### 体験談1：指摘を「学び」に変えたケース  
あるプロジェクトで、自分が書いたコードに「エラーハンドリングが不完全」と指摘されました。当初は「自分はしっかり書いたのに…」と落ち込みましたが、指摘者に「なぜこのエラーは考慮されなかったのか？」と具体的に尋ねました。  
```javascript  
// 指摘コード  
try {  
  const data = await fetchData();  
} catch (e) {  
  console.error(e);  
}  
// 指摘：ネットワークエラー以外の例外処理が不足  
```  
指摘者が「HTTP 500以外の例外（例：Timeout）が発生する可能性がある」と説明したことで、自分は「例外処理の分類（ネットワーク系 vs アプリケーションエラー）」を学び、次のコードで`try-catch`の粒度を調整しました。  

### 体験談2：質問で技術的知識を拡大  
あるレビューで「Reactのコンポーネント構造をフラット化すべき」と指摘されました。当初は「コンポーネントの分割は自己判断で良いのではないか？」と思いましたが、「このプロジェクトでコンポーネントの境界線はどのように定義されているのか？」と質問したところ、  
```javascript  
// 指摘者の回答  
// 「コンポーネントごとに1つの責任を持つ（SRP）ことで、テストの複雑さが軽減される」  
```  
という説明をもらい、コンポーネント設計の原則を学びました。この知識は後のプロジェクトで直接活用され、テストケースの作成効率が30%向上しました。

## まとめ：レビューの技術は「フィードバックの質」と「受け止め方」  
コードレビューは「技術的な指摘」の場にとどまらず、**コミュニケーションの場**であり「**成長の契機**」です。以下のステップで、建設的なレビューを実践しましょう。  

1. **指摘を「技術的意図」に分析**：「なぜこの提案がされたのか？」を明確にする。  
2. **質問を「理解」の目的で構築**：指摘者の立場を意識した具体的な質問をする。  
3. **フィードバックを「次の行動」に変換**：学びをコードに反映し、成長の証明を示す。  

エンジニアとしての成長は、技術力だけでなく「フィードバックをどう捉えるか」に大きく依存します。次回のレビューでは、指摘を受ける前に「これはどんなスキル向上の機会になるか？」と考え、積極的に受け止めてみてください。
