## 1. `useEffect` の依存配列：無限ループと「 stale closure（古いクロージャ）」の罠

### 症状
コンポーネントがマウントされるたび、あるいは特定の状態更新のたびに `useEffect` が無限に実行される、あるいはイベントハンドラ内で古い state を参照してしまう。

### よくある悪いコード

```tsx:BadExample.tsx
import { useState, useEffect } from 'react';

function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState(null);
  const [count, setCount] = useState(0);

  // ❌ 依存配列に user を入れていないが、内部で user を使っている
  // → 初回マウント時のみ実行したい意図でも、user 更新時に再実行されない（stale closure）
  useEffect(() => {
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => setUser(data));
  }, []); // userId が変わっても再フェッチされない！

  // ❌ オブジェクト・配列を依存配列に入れると参照比較で毎回 true になる
  const options = { theme: 'dark' }; // 毎回新しいオブジェクト生成
  useEffect(() => {
    console.log('Options changed', options);
  }, [options]); // 毎レンダー実行される
}
```

### 解決のコツ

1.  **依存配列は「中で使っている全ての変数」を正しく列挙する**（ESLint `exhaustive-deps` ルールを必ず有効にする）。
2.  **オブジェクト・関数を依存に含める場合は `useMemo` / `useCallback` でメモ化する**。
3.  **「最新の値が欲しいだけ」なら `useRef` で保持し、依存配列から外す**パターンも有効。

### 修正例

```tsx:GoodExample.tsx
import { useState, useEffect, useCallback, useRef } from 'react';

function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState(null);
  const [count, setCount] = useState(0);

  // ✅ userId が変わったら再実行される
  useEffect(() => {
    let cancelled = false;
    fetch(`/api/users/${userId}`)
      .then(res => res.json())
      .then(data => {
        if (!cancelled) setUser(data);
      });
    return () => { cancelled = true; }; // クリーンアップで競合状態防止
  }, [userId]);

  // ✅ useCallback で関数自体をメモ化し、依存配列を安定させる
  const fetchOptions = useCallback(() => ({ theme: 'dark' }), []);
  useEffect(() => {
    const opts = fetchOptions();
    console.log('Options changed', opts);
  }, [fetchOptions]); // 関数参照が変わらない限り実行されない

  // ✅ 最新の count が欲しいだけなら ref を使う（依存配列に入れない）
  const countRef = useRef(count);
  countRef.current = count;

  useEffect(() => {
    const id = setInterval(() => {
      console.log('Current count:', countRef.current); // 常に最新値
    }, 1000);
    return () => clearInterval(id);
  }, []); // 依存配列空でOK
}
```

> **ポイント**: `useEffect` のクリーンアップ関数（`return () => {...}`）を書く癖をつけると、コンポーネントアンマウント後の `setState` 警告やレースコンディションを防げます。

## 2. 不要な再レンダリング：`React.memo` と `useMemo` / `useCallback` の使い所

### 症状
親コンポーネントの state 更新に伴い、関係ない子コンポーネントまで再レンダリングされ、パフォーマンスが低下する（例：入力フォームでタイピングするたびに重いリスト全体が再描画される）。

### よくある悪いコード

```tsx:BadRender.tsx
// 親コンポーネント
function Parent() {
  const [text, setText] = useState('');
  // ❌ 毎回新しい関数・オブジェクトが生成される
  const handleClick = () => console.log('click');
  const style = { color: 'red' };

  return (
    <>
      <input value={text} onChange={e => setText(e.target.value)} />
      {/* 子コンポーネントは props が変わっていなくても再レンダリングされる */}
      <ExpensiveChild onClick={handleClick} style={style} />
    </>
  );
}

// 子コンポーネント（メモ化なし）
function ExpensiveChild({ onClick, style }: { onClick: () => void; style: React.CSSProperties }) {
  console.log('ExpensiveChild rendered'); // 毎回ログが出る
  return <button onClick={onClick} style={style}>Heavy</button>;
}
```

### 解決のコツ

| 手法 | 適用対象 | 注意点 |
| :--- | :--- | :--- |
| **`React.memo`** | コンポーネント全体 | **浅い比較**を行う。props にオブジェクト・関数を渡す場合、親側でメモ化必須。 |
| **`useCallback`** | イベントハンドラ関数 | 依存配列が頻繁に変わると逆効果。`useRef` で最新関数を保持する「`useEvent` パターン」も検討。 |
| **`useMemo`** | 計算コスト高い値・オブジェクト・配列 | プリミティブ値の単純な計算なら不要（オーバーヘッドの方が大きい）。 |

### 修正例

```tsx:GoodRender.tsx
// 親
function Parent() {
  const [text, setText] = useState('');
  const [count, setCount] = useState(0);

  // ✅ 依存配列が空なら関数は一度だけ生成される
  const handleClick = useCallback(() => console.log('click'), []);

  // ✅ オブジェクトもメモ化
  const style = useMemo(() => ({ color: 'red' }), []);

  // ✅ 計算コスト高い派生データのみ useMemo
  const expensiveValue = useMemo(() => heavyCalculation(count), [count]);

  return (
    <>
      <input value={text} onChange={e => setText(e.target.value)} />
      <ExpensiveChild onClick={handleClick} style={style} value={expensiveValue} />
    </>
  );
}

// 子：React.memo で包む
const ExpensiveChild = React.memo(function ExpensiveChild({ 
  onClick, 
  style, 
  value 
}: { 
  onClick: () => void; 
  style: React.CSSProperties;
  value: number;
}) {
  console.log('ExpensiveChild rendered'); // count 変更時のみ出力される
  return <button onClick={onClick} style={style}>Heavy: {value}</button>;
});
```

> **実務 Tips**: 「とりあえず `React.memo` で包んでおく」はアンチパターンです。比較コストがかかるため、**プロファイラでボトルネックを特定してから**適用しましょう。

## 3. リストの `key` プロパティ：インデックス指定による UI バグ

### 症状
リストの並び替え・削除・挿入を行った際、**入力フォームの値がずれる、チェックボックスの状態がおかしい、アニメーションが壊れる**などの不可解な挙動が発生する。

### よくある悪いコード

```tsx:BadKey.tsx
function TodoList({ items }: { items: { id: string; text: string }[] }) {
  return (
    <ul>
      {items.map((item, index) => (
        // ❌ index を key にしている
        <li key={index}>
          <input type="checkbox" /> {item.text}
        </li>
      ))}
    </ul>
  );
}
```

### なぜダメなのか？
React の **Reconciliation（調和アルゴリズム）** は `key` を使って「どの要素が変わったか」を特定します。
`key={index}` の場合、**「3番目の要素」という位置情報**で識別されるため、先頭にアイテムを挿入すると「元の1番目が2番目になった」とみなされ、DOM ノードが再利用されます。結果、**内部状態（input の値、チェック状態、スクロール位置）がアイテムに追従せず、位置に残ってしまいます。**

### 解決策：一意で安定した ID を使う

```tsx:GoodKey.tsx
function TodoList({ items }: { items: { id: string; text: string }[] }) {
  return (
    <ul>
      {items.map((item) => (
        // ✅ データベースの PK や UUID など、アイテム固有の不変 ID を使う
        <li key={item.id}>
          <input type="checkbox" /> {item.text}
        </li>
      ))}
    </ul>
  );
}
```

> **補足**: ID がないデータ（例：API から返ってくる配列に ID がない）場合は、**受け取った時点で `map` し、`crypto.randomUUID()` や `nanoid` で ID を付与してから state に格納**するのが定石です。`index` を `key` にするのは「完全に静的な、並び替えも削除も発生しないリスト」に限定してください。

## 4. Context API の過度な利用：プロバイダ分割で再レンダリングを局所化

### 症状
グローバルステート管理のために単一の巨大な `AppContext` を作り、テーマ・ユーザー情報・通知・フォーム状態などを全部詰め込んだ結果、**どこか一つの値が変わるとアプリ全体が再レンダリングされる**。

### よくある悪いコード

```tsx:BadContext.tsx
// ひとつの Context に全部詰め込み
const AppContext = createContext({
  theme: 'light',
  user: null,
  notifications: [],
  // ... 更新関数も全部
});

function AppProvider({ children }) {
  const [theme, setTheme] = useState('light');
  const [user, setUser] = useState(null);
  // ここに大量の state がある

  // ❌ どれか一つでも変わると value オブジェクトの参照が変わる
  // → 全 Consumer が再レンダリング
  const value = { theme, setTheme, user, setUser, ... };

  return <AppContext.Provider value={value}>{children}</AppContext.Provider>;
}
```

### 解決策：関心ごとで Context を分割し、Selector パターンを導入

```tsx:GoodContext.tsx
// 1. Context を目的別に分割
const ThemeContext = createContext({ theme: 'light', setTheme: () => {} });
const UserContext = createContext({ user: null, setUser: () => {} });

// 2. Provider も分割（またはネスト）
function ThemeProvider({ children }) {
  const [theme, setTheme] = useState('light');
  // ✅ value を useMemo でメモ化（中身が変わらない限り参照維持）
  const value = useMemo(() => ({ theme, setTheme }), [theme]);
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>;
}

// 3. 使う側：必要な Context だけ購読する
function Header() {
  // テーマだけ欲しい → UserContext 更新では再レンダリングされない
  const { theme } = useContext(ThemeContext); 
  return <header className={theme}>...</header>;
}

function Profile() {
  const { user } = useContext(UserContext);
  return <div>{user?.name}</div>;
}
```

### さらに踏み込む：`useContextSelector` ライブラリや Zustand/Jotai
標準の `useContext` は「Context の値が変わったら全購読者が再レンダリング」という仕様です。**「巨大な state の一部だけ購読したい」**場合は、以下の選択肢を検討してください。

*   **`use-context-selector`**: `const theme = useContextSelector(ThemeContext, s => s.theme)` のようにセレクタ指定で購読可能。
*   **Zustand / Jotai / Valtio**: Context ベースではないアトミックな状態管理ライブラリ。細粒度な購読がデフォルトで効く。

## 5. 非同期処理と state 更新の競合：`AbortController` と `isMounted` パターン

### 症状
「コンポーネントがアンマウントされた後に state 更新が行われた」という警告（`Can't perform a React state update on an unmounted component`）が出る。または、古いリクエストのレスポンスが新しいリクエストより後に返ってきて、**古いデータで画面が上書きされる（Race Condition）**。

### よくある悪いコード

```tsx:BadAsync.tsx
function SearchResults({ query }: { query: string }) {
  const [data, setData] = useState(null);

  useEffect(() => {
    // ❌ クリーンアップなし、競合制御なし
    fetch(`/api/search?q=${query}`)
      .then(res => res.json())
      .then(json => setData(json)); // アンマウント後や query 変更後に実行される可能性
  }, [query]);
}
```

### 解決策：`AbortController` でキャンセル、または `useRef` でマウント状態管理

#### パターン A：Fetch API 標準の `AbortController`（推奨・モダン）

```tsx:GoodAsyncAbort.tsx
useEffect(() => {
  const controller = new AbortController(); // ① コントローラー作成ラ作成
  const { signal } = controller;

  fetch(`/api/search?q=${query}`, { signal }) // ② signal を渡す
    .then(res => {
      if (!res.ok) throw new Error('Network error');
      return res.json();
    })
    .then(json => setData(json))
    .catch(err => {
      if (err.name === 'AbortError') return; // ③ キャンセル時は無視
      setError(err);
    });

  return () => controller.abort(); // ④ クリーンアップで中断
}, [query]);
```

#### パターン B：`useRef` でマウントフラグ管理（古いブラウザ対応や汎用 Promise 用）

```tsx:GoodAsyncRef.tsx
useEffect(() => {
  let isMounted = true; // ① クロージャでフラグ保持

  fetchData(query).then(json => {
    if (isMounted) setData(json); // ② マウント中のみ更新
  });

  return () => { isMounted = false; }; // ③ アンマウント時に false
}, [query]);
```

> **実務 Tips**: データフェッチには **TanStack Query (React Query) や SWR** などの
