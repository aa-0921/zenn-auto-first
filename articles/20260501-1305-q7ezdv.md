## はじめに

データベースはほとんどの Web アプリケーションの根幹を支えるインフラです。  
しかし、実務で「データを取り出したい」だけのシンプルな要件でも、SQL の書き方を誤ると **パフォーマンスが落ちる**、**バグが混入する** といったリスクが発生します。

この記事では、以下のポイントに絞って「すぐに使える」SQL の書き方を解説します。

1. 基本構文：`SELECT`、`WHERE`、`JOIN`
2. 集計系クエリ：`GROUP BY`、`HAVING`、ウィンドウ関数
3. 実務でよく見るサブクエリ／CTE（共通テーブル式）のパターン

実際のテーブル構造は「ECサイトの受注管理」を想定し、具体的なデータ例とともにコードスニペットを掲載します。途中で **実行計画の確認** や **インデックス設計のヒント** も紹介するので、SQL の基礎だけでなくパフォーマンスチューニングの感覚も身につきます。

## 背景・前提

### テーブル定義（サンプル）

以下の 3 テーブルを前提に話を進めます。実務ではこれらに加えて `products`, `categories` などが増えることが多いですが、概念を掴むには十分です。

```sql
-- ユーザー情報
CREATE TABLE users (
    id          BIGINT PRIMARY KEY,
    name        VARCHAR(50) NOT NULL,
    email       VARCHAR(100) UNIQUE NOT NULL,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 商品情報
CREATE TABLE items (
    id          BIGINT PRIMARY KEY,
    name        VARCHAR(100) NOT NULL,
    price       DECIMAL(10,2) NOT NULL,
    stock       INT DEFAULT 0
);

-- 注文情報（1 件の注文は複数行の order_items を持つ）
CREATE TABLE orders (
    id          BIGINT PRIMARY KEY,
    user_id     BIGINT REFERENCES users(id),
    status      VARCHAR(20) NOT NULL,   -- 'pending', 'paid', 'shipped', ...
    total_price DECIMAL(12,2) NOT NULL,
    placed_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE order_items (
    order_id    BIGINT REFERENCES orders(id),
    item_id     BIGINT REFERENCES items(id),
    quantity    INT NOT NULL,
    unit_price  DECIMAL(10,2) NOT NULL,
    PRIMARY KEY (order_id, item_id)
);
```

**前提条件**

- `id` 系はすべて `BIGINT` のシーケンス（Auto Increment）  
- `price` 系は `DECIMAL` で金額を正確に扱う  
- `status` は文字列で管理（実務では ENUM や参照テーブルでも可）  

このスキーマを元に、実務で頻出するクエリパターンを順に見ていきます。

## 1. 基本構文：SELECT・WHERE・JOIN

### 1-1. 単純な SELECT

```sql
-- すべてのユーザー名とメールアドレスを取得
SELECT name, email
FROM users;
```

`SELECT` の後に列名を列挙し、`FROM` で対象テーブルを指定します。  
`*` を使うと全列取得になるが、実務では **必要な列だけを明示** するのがベストプラクティスです。  

### 1-2. WHERE で絞り込み

```sql
-- 2023 年に登録されたユーザーを取得
SELECT id, name, created_at
FROM users
WHERE created_at >= '2023-01-01' AND created_at < '2024-01-01';
```

- 日付リテラルは `'YYYY-MM-DD'` 形式で書くと可搬性が高い  
- 複数条件は `AND` / `OR` で組み合わせ、**インデックスが有効になるように** カラム順序を意識  

> **Tips**: `WHERE` 句で `LIKE '%foo'` のように前方ワイルドカードを使うとインデックスが使われません。部分一致が必要ならフルテキスト検索や逆インデックスを検討しましょう。

### 1-3. INNER JOIN の基本形

```sql
-- 注文とそのユーザー情報を結合して取得
SELECT o.id AS order_id,
       u.name AS user_name,
       o.status,
       o.total_price,
       o.placed_at
FROM orders AS o
INNER JOIN users AS u
        ON o.user_id = u.id
WHERE o.status = 'paid';
```

- `INNER JOIN` は「両方にマッチする」行だけを返す  
- エイリアス (`o`, `u`) を付けると可読性が向上し、長いテーブル名の繰り返しを書かずに済む  

### 1-4. LEFT JOIN（外部結合）で「存在しない」データも取得

```sql
-- 商品と在庫数を取得し、在庫が 0 の商品も表示
SELECT i.id, i.name, i.stock
FROM items AS i
LEFT JOIN order_items AS oi
       ON i.id = oi.item_id
WHERE i.stock = 0;
```

`LEFT JOIN` は左側テーブルのすべての行を保持し、右側にマッチが無ければ `NULL` が入ります。  
在庫が 0 の商品だけを取得したいケースでは `WHERE i.stock = 0` がシンプルです。

## 2. 集計系クエリ：GROUP BY・HAVING・ウィンドウ関数

### 2-1. GROUP BY と集計関数

```sql
-- ユーザーごとの総購入金額と注文件数
SELECT u.id,
       u.name,
       COUNT(o.id)          AS order_count,
       SUM(o.total_price)   AS total_spent
FROM users AS u
JOIN orders AS o
      ON u.id = o.user_id
WHERE o.status IN ('paid', 'shipped')
GROUP BY u.id, u.name
ORDER BY total_spent DESC
LIMIT 10;
```

- `GROUP BY` の列は **SELECT に必ず含める**（SQL 標準）  
- 集計関数 `COUNT`, `SUM`, `AVG`, `MAX`, `MIN` が使える  
- `ORDER BY` で集計結果を並べ替え、`LIMIT` で上位 n 件に絞る  

### 2-2. HAVING で集計後にフィルタ

```sql
-- 合計購入金額が 10,000 円以上のユーザーだけを抽出
SELECT u.id, u.name, SUM(o.total_price) AS total_spent
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'paid'
GROUP BY u.id, u.name
HAVING SUM(o.total_price) >= 10000;
```

`WHERE` は **行レベル** の条件、`HAVING` は **集計結果** に対する条件です。  
`HAVING` を使うと「合計が 10,000 円未満のユーザーは除外」でき、結果セットが小さくなるのでパフォーマンス向上にも寄与します。

### 2-3. ウィンドウ関数で「集計しつつ元データも残す」

```sql
-- 各注文に対して、同一ユーザーの累計購入金額を付与
SELECT o.id,
       o.user_id,
       o.total_price,
       SUM(o.total_price) OVER (PARTITION BY o.user_id
                                ORDER BY o.placed_at
                                ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW) AS cumulative_spent
FROM orders o
WHERE o.status = 'paid';
```

- `OVER (PARTITION BY … ORDER BY …)` でウィンドウを定義  
- `ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW` は「先頭から現在行まで」の累積を計算  
- ウィンドウ関数は **GROUP BY** で行がまとめられないので、元の行数は保たれます。レポート系の要件で頻出です。

> **注意**: ウィンドウ関数は `SELECT` のみで使用でき、`WHERE` では使えません。フィルタが必要ならサブクエリや CTE で前処理します。

## 3. 実務でよく見るサブクエリ／CTE のパターン

### 3-1. サブクエリで「条件に合う行だけを抽出」

```sql
-- 直近 30 日間に 1 回以上購入したユーザーの一覧
SELECT *
FROM users
WHERE id IN (
    SELECT DISTINCT user_id
    FROM orders
    WHERE placed_at >= CURRENT_DATE - INTERVAL '30 day'
      AND status = 'paid'
);
```

- `IN (SELECT …)` は **存在確認** に便利  
- `DISTINCT` を付けて重複行を除外し、インデックスが効きやすくなる  

### 3-2. CTE（WITH句）で可読性と再利用性を向上

```sql
-- 各商品ごとの売上上位 3 件の注文を取得
WITH item_sales AS (
    SELECT oi.item_id,
           SUM(oi.quantity * oi.unit_price) AS revenue,
           COUNT(DISTINCT oi.order_id) AS order_cnt
    FROM order_items oi
    GROUP BY oi.item_id
),
ranked_sales AS (
    SELECT i.id,
           i.name,
           s.revenue,
           s.order_cnt,
           ROW_NUMBER() OVER (ORDER BY s.revenue DESC) AS rank
    FROM items i
    JOIN item_sales s ON i.id = s.item_id
)
SELECT *
FROM ranked_sales
WHERE rank <= 3;
```

- `WITH` で名前付きサブクエリ（CTE）を定義し、**段階的にロジックを分割**  
- `ROW_NUMBER()` で順位付けし、`WHERE rank <= 3` で上位 3 件に絞る  
- 大規模クエリでも **読みやすさ** が格段に向上し、保守性が高まります  

### 3-3. サブクエリ＋ウィンドウ関数で「前月比」集計

```sql
WITH monthly AS (
    SELECT DATE_TRUNC('month', placed_at) AS month,
           SUM(total_price) AS monthly_sales
    FROM orders
    WHERE status = 'paid'
    GROUP BY 1
)
SELECT month,
       monthly_sales,
       LAG(monthly_sales) OVER (ORDER BY month) AS prev_month_sales,
       (monthly_sales - LAG(monthly_sales) OVER (ORDER BY month)) /
       NULLIF(LAG(monthly_sales) OVER (ORDER BY month), 0) * 100 AS growth_rate_pct
FROM monthly
ORDER BY month;
```

- `DATE_TRUNC('month', …)` で月単位に切り捨て  
- `LAG` で前月の売上を取得し、成長率を計算  
- `NULLIF` で除算時のゼロ除算を防止

## まとめ

この記事では、**SQL の基礎構文** から **実務で頻出する集計・サブクエリ** までを、EC サイトの受注データを例に具体的に解説しました。

| 項目 | 主な使いどころ |
|------|----------------|
| `SELECT` / `WHERE` | 行レベルの絞り込み、必要な列だけ取得 |
| `JOIN`（INNER / LEFT） | 複数テーブルのリレーションを表現 |
| `GROUP BY` / `HAVING` | 集計と集計後のフィルタ |
| ウィンドウ関数 | 累積、順位、前後行参照など、行をまとめずに集計 |
| サブクエリ / CTE | 複雑ロジックの分割、再利用、可読性向上 |

### 実務で意識したいポイント

1. **インデックスを意識した WHERE 条件**  
   - 範囲検索 (`>=`, `<`) や等価検索 (`=`) がインデックスにヒットしやすい  
2. **必要最小限の列だけを SELECT**  
   - ネットワーク負荷とメモリ使用量を削減  
3. **集計はなるべく事前に絞り込む**  
   - `WHERE` → `GROUP BY` の順序でデータ量を減らすと高速化  
4. **CTE は可読性向上だけでなく、再利用やデバッグに便利**  
   - 大規模レポートやバッチ処理で積極活用  

SQL はシンプルな構文でも、書き方次第で **数十倍の速度差** が出ます。この記事のサンプルを自分のプロジェクトに置き換えて、まずは「**正しく、かつ高速に**」データを取得できるように練習してみてください。
