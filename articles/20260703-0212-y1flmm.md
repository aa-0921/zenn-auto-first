最近、AI ツールが開発現場に急速に浸透しています。その代表例として、Claude Code が挙げられます。Claude Code は、Claude 2.0 ベースの言語モデルと統合されたコード補完ツールで、VS Code 上から直接、コードの提案を受けたり、簡単な修正を自動化したりできます。特に経験の浅いエンジニアにとって、これは開発効率を飛躍的に高める魅力的な機能に思えるでしょう。しかし、便利なツールである一方で、過度に依存しすぎると、プログラミングの基本的なスキルが低下したり、意図しないバグを生み出したりするリスクもあります。この記事では、Claude Code を「使いこなす」ための具体的なコツを紹介します。頼りすぎないバランス、効果的な「質問」の仕方、そして自分でコードを書く部分との使い分け方を、初心者にもわかりやすく解説します。

## 背景・前提

### AI ツールが開発にもたらす変化
- **生産性の向上**: 繰り返し作業や boilerplate コードの生成を自動化できます。
- **学習の加速**: 理解していない部分のコードを提案されることで、新しい構文やパターンを学ぶきっかけになります。
- **品質の担保**: 静的解析ツールと連携することで、単純なミスを減らすことができます。

### 「過度な依存」の落とし穴
- **問題解決力の低下**: ツールなしで考える習慣が失われる。
- **予期せぬ動作**: 提案されたコードが文脈を理解しておらず、バグの原因となる。
- **メンタルモデルの欠如**: なぜそのコードが動くのかを理解できない。

したがって、Claude Code を「補助的な存在」と位置づけ、適切な場面で使うことが重要です。

## Claude Code を効果的に使うためのステップ

### 1. 目的を明確にする（具体的なプロンプトを作成する）
Claude Code に何をさせたいのかを、できるだけ具体的に伝えることが、満足のいく結果を得る鍵です。曖昧なリクエストは、的外れな提案につながります。

```text
# 例: React コンポーネントのスケルトンを作成する
ファイル: src/components/Loading.tsx
props: { message?: string }
出力: ローディングスピナとオプションのテキストを表示する最小限の機能コンポーネントを返却する
```

**ポイント**: ファイル名、コンポーネントの props、期待する出力などを記載します。

### 2. 提案されたコードを確認する
Claude Code がコードを生成したら、そのままコピペする前に、以下の点を確認します。

- **構文の正しさ**: TypeScript 構文や import 文の欠落がないか確認する。
- **動作の意図**: コードの動作を自分の頭の中でトレースし、期待通りか確認する。
- **スタイルの一貫性**: プロジェクトのプリティプリンタールール（Prettier、ESLint）に合致しているか確認する。

確認作業は、Markdown 内のコードブロックやブラウザのコンソールで lint ツールを実行することで自動化できます。

```bash
# 例: 変更されたファイルのみを lint する
pnpm lint --fix src/components/Loading.tsx
```

### 3. 自分でコードを書く部分との使い分け
AI ツールを万能と考えるのは避けましょう。以下のフローチャートを参考に、適切な場面で使い分けてください。

| 状況 | 推奨ツール |
|-----------|-------------------|
| **繰り返し作業の削減**（例: API レスポンスのデータ構造をマッピングする） | Claude Code で boilerplate コードを生成 → 必要に応じて手動で調整 |
| **新しい機能の全体像の理解**（例: Redux 状態管理のフロー） | まず自分で簡易な実装を書く → 概念を身につける |
| **デバッグと調査**（例: 特定のエラーの原因を特定する） | 質問形式で AI に説明を求める → 自分でデバッグを行い、理解を深める |
| **コードレビュー**（例: セキュリティ上の問題点を検出する） | 自分でレビュー → AI に説明を依頼し、理解を補完する |

### 4. 効果的な質問の仕方
Claude Code と対話する際は、「何でも聞いてください」というスタンスは逆効果です。AI からの回答をより有益なものにするために、以下の3つの質問パターンを意識しましょう。

1. **制約条件付きの質問**: 「X を実現するにはどうすればよいですか？（制限: 型安全性、eslint ルール）」
2. **例を含む質問**: 「React Hook を使用したカウンターコンポーネントを作成してください。次に、同じロジックを Vue で書き換えてください。」
3. **診断質問**: 「なぜ `useEffect(() => { … }, [])` がマウント時にのみ実行されるのか、教えてください。」

また、プロンプトの最後に「なぜそのように書いたのか、説明してください」と付け加えることで、理解が深まります。

## 実践：Claude Code を組み込んだ小さなプロジェクト

### プロジェクト概要
- **タイトル**: 「Claude Code と VS Code を使ったタスク管理アプリ」
- **目的**: CRUD 操作を実行するシンプルな ToDo アプリを作成し、Claude Code を使って boilerplate コードを生成、簡単なバグ修正は自分で行う。

### ステップバイステップのワークフロー

1. **フォルダ構造を作成**（Claude Code は単一のファイルには対応しにくいので、フォルダ構成を事前に作成します）。

```bash
mkdir todo-app && cd todo-app
npm init -y
npm install -D typescript @types/node vite @vitejs/plugin-react
```

2. **Claude Code に `App.tsx` のスケルトンを生成** 依頼する。

```
ファイル: src/App.tsx
出力: React 18 を使ったシンプルな ToDo アプリ。以下の機能を含む:
- タスクを追加する入力フィールド
- 追加されたタスクを表示するリスト
- 各タスクに対して「完了」トグルボタンと削除ボタン
- 完了したタスクをフィルタリングするUI（すべて/アクティブ/完了）
```

3. **Claude Code の提案を確認** し、`src/main.tsx` を追加で生成する。

4. **手動で追加した機能**：
   - タスクの永続化のための `localStorage` への書き込み/読み込み。
   - 各タスクに対して一意の ID を生成する `uuid` ライブラリの導入。

5. **簡単な lint エラーを修正** し、`pnpm lint --fix` を実行する。

6. **最後に、自分でバグを修正**：フィルター切り替え時にコンポーネントが再レンダリングされるが、完了したタスクの削除時に `React Hook` の警告が発生する。原因を調査し、`useEffect` の依存関係配列を修正する。

最終的な動作は、シンプルな ToDo アプリとして動作し、自分で書いたコードと Claude Code が生成したコードがバランスよく混在している。

## まとめ

- **Claude Code は便利な補助ツールですが、万能ではありません**。目的を明確にし、具体的なプロンプトを作成することで、より有益な提案を得られます。
- **提案されたコードは必ず確認** し、構文の正しさ、動作の意図、スタイルの一貫性を検証してください。
- **自分でコードを書く部分との使い分け**（繰り返し作業 vs 学習、新しい機能の理解 vs デバッグ）を意識することで、過度な依存を避け、スキルを向上させることができます。
- **効果的な質問の仕方**（制約条件付き、例を含む、診断質問）を身につけることで、AI ツールとのコミュニケーションがより生産的になります。

Claude Code を「開発プロセスの一部」と捉え、適切な場面で活用することで、初心者エンジニアでも生産性を高めながら、プログラミングの基本的なスキルを確実に身につけることができます。まずは小さなタスクから始め、徐々に使い方を拡張していきましょう。
