## SQLの基本構文：SELECT, FROM, WHERE

SQLの最もシンプルな形は、テーブルから特定のカラムを抽出する構文です。`SELECT` は取得するカラムを、`FROM` はどのテーブルを対象にするかを指定します。

```sql
SELECT id, name, email
FROM users;
```

これで `users` テーブルのすべての行について、`id`、`name`、`email` の3つのカラムが返ります。ただし、実際のテーブルには数百、数千、あるいはそれ以上のレコードが含まれているため、全件取得は現実的ではありません。そこで `WHERE` 句を使って、条件に合致する行だけを絞り込む必要があります。

```sql
SELECT id, name, email
FROM users
WHERE status = 'active' AND created_at >= '2023-01-01';
```

`WHERE` 句では比較演算子（`=`、`!=`、`>`、`<`、`>=`、`<=`）や論理演算子（`AND`、`OR`、`NOT`）を組み合わせ、複雑な条件を表現できます。`IN` を使うことで複数の値の一括指定も可能です。

```sql
SELECT id, name, department
FROM employees
WHERE department IN ('Engineering', 'Product', 'Design');
```

さらに、文字列の部分一致検索には `LIKE` 演算子が便利です。`%` は任意の文字列（0文字以上）、`_` は任意の1文字を表します。

```sql
SELECT id, name
FROM customers
WHERE name LIKE '田中%';
```

このように、`SELECT`、`FROM`、`WHERE` の3つの要素を組み合わせることで、ほぼすべての取得要件を満たすクエリを作成できます。ただし、単一のテーブルだけでは実現できない、関連するデータを別のテーブルから持ってくるための `JOIN` について、次のセクションで説明します。

## JOINと集計関数：関連データの取得と集計

### INNER JOINで関連データを結合する

実際のアプリケーションでは、ユーザー情報は `users` テーブルに、注文情報は `orders` テーブルに分けて管理することが一般的です。あるユーザーがどのような注文をしたのかを知るためには、これら2つのテーブルを関連付ける必要があります。その方法が `JOIN` です。最もよく使われるのは `INNER JOIN` で、両方のテーブルに一致する行のみを返します。

```sql
SELECT users.name, orders.order_id, orders.total_amount
FROM users
INNER JOIN orders ON users.id = orders.user_id;
```

このクエリは、`users` テーブルの各行に対して、`orders` テーブルの中で `user_id` が一致する行を探し、両者のデータを横に結合します。もし、注文のないユーザーも含めたい場合は `LEFT JOIN` を使います。

```sql
SELECT users.name, orders.order_id, orders.total_amount
FROM users
LEFT JOIN orders ON users.id = orders.user_id;
```

`LEFT JOIN` の場合、左側（`FROM` に書かれたテーブル）のすべての行が結果に含まれ、右側のテーブルに一致する行がない場合は `NULL` が入ります。

### 集計関数とGROUP BYで集計を行う

ビジネスのKPIを計算する際には、データの集計が欠かせません。`COUNT`、`SUM`、`AVG`、`MAX`、`MIN` などの集計関数を使うことで、数値の集計が簡単に行えます。ただし、集計関数を使う場合は `GROUP BY` を併用することがほとんどです。

例えば、どのカテゴリーの商品が最も売れたのかを知りたい場合、次のように書きます。

```sql
SELECT category, COUNT(*) AS total_sales, SUM(price) AS total_revenue
FROM products
GROUP BY category;
```

このクエリは、`products` テーブルを `category` カラムごとにグループ分けし、各カテゴリーの商品数と合計価格を計算します。`GROUP BY` をしないと、集計関数はテーブル全体の1つの集計結果しか返せません。

さらに、`HAVING` 句を使って、集計後の結果に対して条件を適用できます。例えば、売上高が1000を超えるカテゴリーだけを抽出したい場合です。

```sql
SELECT category, SUM(price) AS total_revenue
FROM products
GROUP BY category
HAVING SUM(price) > 1000;
```

`WHERE` 句は集計前の行を絞り込むのに対し、`HAVING` 句は集計後のグループに対して条件を適用する点に注意が必要です。

## サブクエリと共通テーブル式（CTE）：複雑なロジックの整理

クエリが複雑になると、1つの `SELECT` 文では表現しきれないケースがあります。そのようなときに役立つのが、サブクエリと共通テーブル式（CTE）です。

### サブクエリ：クエリの中のクエリ

サブクエリとは、別のクエリの中に入れ子になった `SELECT` 文です。`WHERE` 句の中で使うことで、動的な条件を設定できます。例えば、自分の部署の平均年齢よりも年齢が若い人を抽出したい場合、次のように書きます。

```sql
SELECT name, age, department
FROM employees
WHERE age < (
    SELECT AVG(age)
    FROM employees
    WHERE department = 'Engineering'
);
```

このように、サブクエリを使うことで、1つのクエリで別のクエリの結果を参照できます。ただし、多くの場合、サブクエリはパフォーマンスに影響を与える可能性があります。複雑なロジックや再利用性を考慮すると、CTEの方が可読性が高いです。

### 共通テーブル式（CTE）でクエリをモジュール化する

CTEは、サブクエリを名前付きで定義し、複数回参照できるようにしたものです。`WITH` 句を使って定義し、その後のメインクエリで使用します。

```sql
WITH engineering_avg_age AS (
    SELECT AVG(age) AS avg_age
    FROM employees
    WHERE department = 'Engineering'
)
SELECT name, age, department
FROM employees, engineering_avg_age
WHERE employees.age < engineering_avg_age.avg_age;
```

CTEは、クエリをステップバイステップで分解できるため、デバッグや保守がしやすくなります。特に、再帰的な処理（階層構造の走査など）を行う際に強力です。

```sql
WITH RECURSIVE subcategories AS (
    SELECT id, name, parent_id
    FROM categories
    WHERE parent_id IS NULL
    UNION ALL
    SELECT c.id, c.name, c.parent_id
    FROM categories c
    INNER JOIN subcategories s ON c.parent_id = s.id
)
SELECT * FROM subcategories;
```

このように、再帰CTEを使えば、階層データを簡単に取得できます。

## まとめとベストプラクティス

この記事では、SQLの基礎である `SELECT`、`WHERE`、`JOIN` から始まり、集計関数、サブクエリ、CTE まで、実務でよく使うパターンを解説しました。SQLを書く上で重要なのは、正しい結果を得るだけでなく、パフォーマンスと可読性のバランスを取ることです。

- 必要なデータだけを `SELECT` し、不要なカラムや行は `WHERE` で絞り込むことで、ネットワーク負荷やメモリ使用量を抑えることができます。
- `JOIN` を使うときは、結合条件にインデックスが張られていることを確認しましょう。インデックスがなければ、大規模テーブルでのクエリは非常に遅くなる可能性があります。
- 集計クエリでは `GROUP BY` を意識し、`HAVING` を使って集計後の条件を指定する習慣をつけましょう。
- 複雑なロジックは、サブクエリよりもCTEを使ってモジュール化すると、コードが理解しやすくなります。

SQLはシンプルな構文に見えますが、応用範囲が広く、パフォーマンスチューニングの余地も大きいです。この記事をきっかけに、より効率的で保守しやすいSQLクエリを心がけてください。
