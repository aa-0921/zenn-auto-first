## Dockerの基本概念 - イメージとコンテナの違い

Dockerを理解する上で最も重要なのが「イメージ」と「コンテナ」の違いです。

### イメージとは

イメージはコンテナを作成するためのテンプレートです。アプリケーションのコード、ライブラリ、設定ファイル、そしてOSレベルの依存関係がすべて含まれています。イメージは不変（immutable）であり、一度作成されると変更されません。

```bash
# イメージの一覧を表示
docker images

# イメージを検索
docker search nginx

# イメージをダウンロード
docker pull nginx:latest
```

### コンテナとは

コンテナはイメージから作成された実行可能なインスタンスです。コンテナはイメージを元にして起動され、その時点での状態を保持します。複数のコンテナを同じイメージから作成することもできます。

```bash
# コンテナを起動
docker run -d -p 8080:80 --name my-nginx nginx

# コンテナの一覧を表示
docker ps -a

# コンテナを停止
docker stop my-nginx

# コンテナを削除
docker rm my-nginx
```

簡単に言えば、イメージは設計図でコンテナはその設計図から作られた実物です。

## Dockerの基本的な使い方

### Dockerfileの作成

まずは簡単なNode.jsアプリケーションのDockerイメージを作成してみましょう。

```dockerfile
# Dockerfile
FROM node:18-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci --only=production

COPY . .

EXPOSE 3000

CMD ["node", "server.js"]
```

```json
// package.json
{
  "name": "docker-demo",
  "version": "1.0.0",
  "main": "server.js",
  "scripts": {
    "start": "node server.js"
  },
  "dependencies": {
    "express": "^4.18.0"
  }
}
```

```javascript
// server.js
const express = require('express');
const app = express();
const PORT = process.env.PORT || 3000;

app.get('/', (req, res) => {
  res.json({ message: 'Hello from Docker!' });
});

app.listen(PORT, () => {
  console.log(`Server running on port ${PORT}`);
});
```

### イメージのビルドとコンテナの実行

```bash
# イメージをビルド
docker build -t my-node-app .

# コンテナを実行
docker run -d -p 3000:3000 --name node-container my-node-app

# ログを確認
docker logs node-container
```

## docker-composeの基本

単一のコンテナだけでなく、複数のコンテナが連携するアプリケーションも多いでしょう。そんな時に便利なのがdocker-composeです。

### docker-compose.ymlの書き方

```yaml
# docker-compose.yml
version: '3.8'

services:
  web:
    build: .
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=development
    volumes:
      - .:/app
      - /app/node_modules
    depends_on:
      - db
  
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
    ports:
      - "5432:5432"
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

### docker-composeの基本コマンド

```bash
# サービスを起動
docker-compose up -d

# サービスを停止
docker-compose down

# ログを確認
docker-compose logs -f web

# 特定のサービスだけ起動
docker-compose up -d db
```

## ローカル開発でDockerを使うメリット

### 1. 環境の一貫性

Dockerを使う最大のメリットは、開発環境と本番環境を同じにできることです。同じイメージを使えば、どこでも同じ動作をします。

### 2. セットアップの簡易化

新しいチームメンバーが参加した時、わざわざ開発環境を構築する必要がありません。

```bash
# これだけで開発環境が整う
git clone <repository-url>
cd <project-directory>
docker-compose up -d
```

### 3. 簡単なクリーンアップ

開発が終わったら簡単に環境を削除できます。

```bash
# 開発環境を完全にクリーンアップ
docker-compose down --volumes --rmi all
```

### 4. マイクロサービス開発のサポート

複数のサービスをまとめて管理できるので、マイクロサービスアーキテクチャの開発も容易になります。

## 実践：実際の開発環境を構築してみる

### フロントエンドとバックエンドの連携例

```yaml
# docker-compose.dev.yml
version: '3.8'

services:
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile.dev
    ports:
      - "3001:3000"
    volumes:
      - ./frontend:/app
      - /app/node_modules
    command: npm run dev
    
  backend:
    build:
      context: ./backend
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    volumes:
      - ./backend:/app
      - /app/node_modules
    environment:
      - DATABASE_URL=postgresql://user:password@db:5432/myapp
    depends_on:
      - db
      
  db:
    image: postgres:15
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

開発用のフロントエンドDockerfile:

```dockerfile
# frontend/Dockerfile.dev
FROM node:18

WORKDIR /app

COPY package*.json ./
RUN npm install

COPY . .

EXPOSE 3000

CMD ["npm", "run", "dev"]
```

これで`docker-compose -f docker-compose.dev.yml up`を実行するだけで、フロントエンドのホットリロードも含めた完全な開発環境が整います。

## まとめ

Dockerは開発効率を大幅に向上させる強力なツールです。イメージとコンテナの違いを理解し、docker-composeを活用すれば、複雑な開発環境も簡単に構築・管理できるようになります。

**ポイントまとめ**:
- イメージはテンプレート、コンテナは実行インスタンス
- Dockerfileでイメージを定義し、docker-composeで複数コンテナを管理
- ローカル開発ではボリュームマウントでホットリロードを実現
- 本番と開発で同じイメージを使うことで環境差異を排除

これらの基本を押さえれば、「私の環境では動かない」という問題から解放され、チームでの開発もスムーズに進むでしょう。
