## 背景・前提  
### バージョン管理の基本概念  
Git は「コミット（snapshotwerker）」を作ることでソースコードの状態を時系列で追跡します。  
- **リモートリポジトリ**（例: GitHub, GitLab, Bitbucket）は共同作業の中心。  
- **ローカルリポジトリ**は自分のマシン上で作業。  
- **ブランチ**は並行開発を安全に行うための仕組み。  
  
### 主要コマンドの整理  
| コマンド | 役割 |
|----------|------|
| `git clone` | リモートリポジトリをローカルへコピー |
| `git pull` | リモートの変更をローカルに取得しマージ |
| `git branch` | ブランチの一覧・作成・削除 |
| `git checkout` | ブランチ切替・ファイル復元 |
| `git commit` | ステージされた変更をコミット |
| `git push` | ローカルコミットをリモートへ反映 |
| `git merge` | ブランチを統合 |
| `git rebase` | ブランチのベースを移動 |

## 本文（ステップ・コード例）

### 1. リポジトリをクローンする  
```bash
# 既存のリモートリポジトリをローカルにコピー
git clone https://github.com/your-org/your-repo.git
cd your-repo
```
※SSH でクローンする場合は `git@github.com:your-org/your-repo.git` を使用。

### 2. 最新の変更を取得してマージ  
```bash
# デフォルトブランチ (例: main) の最新を取得
git pull origin main
```
`git pull` は `fetch` + `merge` を一度に行います。  
複数人で作業している場合、`git pull --rebase` とするとリベースで直線的な履歴を保てます。

### 3. 新しいブランチで作業を開始  
```bash
# ブランチを作成して切替
git checkout -b feature/login-form
```
ブランチ名は作業内容を簡潔に表す名前を付けるとチームで分かりやすくなります。

### 4. 変更をステージングしコミット  
```bash
# ファイルをステージ
git add src/components/LoginForm.jsx

# コミット
git commit -m "Add login form component"
```
コミットメッセージは「何をしたか」を簡潔に書くことが推奨されます(tile: Conventional Commits)。

### 5. リモートにプッシュ  
```bash
# ブランチをリモートに作成してプッシュ
git push -u origin feature/login-form
```
`-u` は upstream を設定し、以降は `git push` だけで済みます。

### 6. コンフリクトが起きたらどうする？  
#### 6-1. コンフリクトの発生例  
複数人が同じファイルの同じ行を編集した場合、`git pull` や `git merge` で衝突が発生します。  
```bash
$ git pull origin main
Auto-merging src/App.jsx
CONFLICT (content): Merge conflict in src/App.jsx
Automatic merge failed; fix conflicts and then commit the result.
```

#### 6-2. コンフリクト箇所の確認  
コンフリクトはファイル内に `<<<<<<<`, `=======`, `>>>>>>>` のマーカーで示されます。  
```jsx
<<<<<<< HEAD
const title = "Welcome!";
=======
const title = "Hello, World!";
>>>>>>> origin/main
```
`HEAD` は自分の変更、`origin/main` はリモート側の変更です。

#### 6-3. コンフリクトの解消  
1. **手動で修正**  
   マーカーを消し、適切なコードに書き換えます。  
2. **解消したファイルをステージ**  
   ```bash
   git add src/App.jsx
   ```
3. **マージコミット**  
   ```bash
   git commit -m "Resolve merge conflict in App.jsx"
   ```

#### 6-4. コンフリクト解消のベストプラクティス  
- **頻繁に pull** してローカルとリモートの差分を小さく保つ。  
- **小さなコミット単位**で作業し、長時間マージを保留しない。  
- **コードレビュー**でコンフリクトの വികസന സാധ്യതを減らす。  
- `git mergetool` を使って GUI で解消する場合もあります。  

### 7. Pull Request（PR）でコードレビュー  
GitHub などのホスティングサービスでは、ブランチをリモートにプッシュしたら PR を作成し、レビューを受けます。  
レビュアーが「コンフリクトがある」「コード規約に沿っていない」などとコメントをくると、再度コンフリクト解消や修正を行い、再度プッシュすれば PR が更新されます。

## まとめ  
- **基本フロー**は clone → pull → branch → add → commit → push بک。  
- **コンフリクト**は頻繁に `git pull` し、変更を小さく保つことで防げます。  
- **解消方法**はマーカーを確認し、手動で修正して `git add`・`git commit` するだけ。  
- **チーム開発**ではブランチ名・コミットメッセージのルールを決め、コードレビューを活用すると品質が向上します。  
  
Git の基本操作を確実に身につけ、実務で起きやすいコンフリクトをスムーズに解消できるようにしておきましょう。継続的にリポジトリを操作することで、より自然に使いこなせるようになります。
