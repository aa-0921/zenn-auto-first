## はじめに：なぜ「今さら Git」なのか

エンジニアとして現場に入って最初に直面するのが「Git の運用」です。チュートリアルでは `git add .` と `git commit -m "fix"` で済んでいた作業が、実務では **ブランチ戦略、コードレビュー、コンフリクト解消、履歴の整理** など、より厳密なルールの下で求められます。

「なんとなく動く」から「チームで安全に運用できる」へ。この記事では、駆け出しエンジニアが現場で即戦力になるための **標準的な作業フロー（Clone → Branch → Commit → Push → Pull Request → Merge）** と、**避けては通れないコンフリクト対処法** を、実行コマンド付きで解説します。

## 前提：現場で前提とされる環境と用語

本記事では以下の環境・前提で話を進めます。

*   **リモートリポジトリ**: GitHub / GitLab / Bitbucket 等（ここでは GitHub を想定）
*   **ブランチ戦略**: **GitHub Flow**（`main` ブランチから派生 → 作業 → PR → レビュー → Merge → 削除）を基準とします
*   **デフォルトブランチ**: `main`（旧 `master`）
*   **CLI 操作**: GUI ツールも便利ですが、原因究明と自動化のために **CLI での操作をマスターすること** を推奨します

### 最初にやっておくべき初期設定

コミット作者情報の設定はマシンごとに一度だけ必要です。

```bash
# ユーザー名とメールアドレス（GitHub 登録時のものと合わせる）
git config --global user.name "Your Name"
git config --global user.email "you@example.com"

# デフォルトブランチ名を main にする（最近の Git ならデフォルトだが明示推奨）
git config --global init.defaultBranch main

# 日本語ファイル名の文字化け防止
git config --global core.quotepath false

# エディタを VS Code にする（好みで vim, nano 等へ）
git config --global core.editor "code --wait"
```

## ステップ 1：リポジトリをローカルに持ってくる（`clone`）

プロジェクト参加時、最初に行うのがクローンです。HTTPS と SSH のどちらを使うかはチームの方針によりますが、SSH キー登録済みなら **SSH 推奨**（パスワード入力不要、セキュア）です。

```bash
# SSH の場合（推奨）
git clone git@github.com:organization/project-name.git

# HTTPS の場合
git clone https://github.com/organization/project-name.git

# ディレクトリ移動
cd project-name
```

**現場の Tips**: `clone` 直後は必ず `main` ブランチの最新状態になっていますが、すぐに作業ブランチを切らずに **`git pull origin main`** で最新化する癖をつけましょう（クローン直後は不要ですが、日常の作業開始時のルーチンとして重要です）。

---

## ステップ 2：作業ブランチを切る（`branch` / `switch` / `checkout`）

`main` ブランチで直接作業するのは **厳禁** です。機能追加・バグ修正ごとにブランチを切ります。

### ブランチ命名規則の例（チームで統一されているはず）

| 種別 | プレフィックス | 例 |
| :--- | :--- | :--- |
| 機能追加 | `feature/` | `feature/user-login` |
| バグ修正 | `fix/` | `fix/header-alignment` |
| リファクタ | `refactor/` | `refactor/api-client` |
| 緊急対応 | `hotfix/` | `hotfix/payment-crash` |

### コマンド：作成と切り替えを一発で

最近の Git（v2.23以降）では `git switch` が推奨されています。

```bash
# 1. まず main を最新にする（重要）
git switch main
git pull origin main

# 2. 新しいブランチを作成して切り替え（-c は --create の略）
git switch -c feature/user-login

# 確認
git branch
# * feature/user-login
#   main
```

> **補足**: 古い記事では `git checkout -b feature/user-login` と書かれていますが、`checkout` は「ブランチ切替」と「ファイル復元」の二役で混同しやいため、用途別に `switch`（ブランチ移動）と `restore`（ファイル復元）に分かれた新コマンドの使用を推奨します。

## ステップ 3：変更をコミットする（`add` / `commit` / `status` / `diff`）

作業ディレクトリでの編集が終わったら、変更をステージングエリアに上げ、コミットします。

### 状態確認の癖をつける

```bash
# 今の状態を見る（どのファイルが変更されたか、ステージ済みか）
git status

# 差分を見る（ステージ前）
git diff

# 差分を見る（ステージ後）
git diff --staged
```

### ステージングとコミット

「関連する変更」ごとにコミットを分けるのが鉄則です。「ログイン機能の実装」と「タイポ修正」を同じコミットに混ぜないでください。

```bash
# 特定のファイルをステージング（推奨：意図しないファイルを混入させない）
git add src/auth/login.ts tests/auth/login.test.ts

# コミットメッセージをエディタで書く（-m で一行書きも可だが、本文書くならエディタ推奨）
git commit

# あるいは一行で済ませる場合（軽微な修正のみ）
git commit -m "fix: ログイン時のバリデーションエラーメッセージを修正"
```

#### 良いコミットメッセージの書き方（Conventional Commits 準拠）

```
<type>(<scope>): <subject>

<body>

<footer>
```

*   **type**: `feat` (新機能), `fix` (バグ修正), `refactor`, `docs`, `style`, `test`, `chore` など
*   **subject**: 命令形・現在形で「何をしたか」（例: `add login button` ではなく `add login button`）
*   **body**: 「なぜ必要だったか」「どう変わったか」を補足
*   **footer**: `Closes #123` などで Issue 連携

**例**:
```
feat(auth): JWT トークンリフレッシュ機能を追加

アクセストークンの有効期限切れ時に自動でリフレッシュするようにした。
これによりユーザーが明示的にログインし直す頻度を減らせる。

Closes #45
```

## ステップ 4：リモートに送信し、レビューを依頼する（`push` / Pull Request）

ローカルでのコミットが完了したら、リモートリポジトリにブランチを送ります。

### 初回 Push（アップストリーム設定）

```bash
# -u (--set-upstream) をつけると、次回から `git push` だけで良くなる
git push -u origin feature/user-login
```

### 2回目以降の Push

```bash
git push
# 強制プッシュが必要な場合（リベース後など）は --force-with-lease を使う（後述）
```

### Pull Request (PR) の作成

CLI からでも `gh pr create` (GitHub CLI) で作れますが、ブラウザでベースブランチ（通常 `main`）への PR を作成し、以下を記入します。

*   **タイトル**: コミットメッセージの1行目と揃える
*   **説明**: 変更内容、動作確認手順、スクリーンショット、関連 Issue 番号
*   **Reviewer**: 必ずアサインする（自分でマージしない）

## ステップ 5：レビュー対応と履歴の整理（`rebase` / `commit --amend` / `push -f`）

レビューで「ここ直して」と言われたら、追加コミット (`fix: typo`) を積むのも良いですが、**履歴を綺麗に保つために「過去のコミットを書き換える」** テクニックが現場では必須です。

### 直前のコミットを修正する（`--amend`）

```bash
# ファイルを修正
vim src/auth/login.ts

# ステージング
git add src/auth/login.ts

# 直前のコミットに上書き（メッセージ変更なしなら --no-edit）
git commit --amend --no-edit

# リモートに強制プッシュ（※注意：後述）
git push --force-with-lease
```

### `main` の最新変更を取り込む（`rebase` / `merge`）

作業中に `main` が進んでいたら、自分のブランチに取り込みます。現場では **「履歴を一直線に保つため `rebase` 派」** と **「履歴を改竄しない `merge` 派」** で分かれますが、GitHub Flow では **`rebase` が主流** です。

```bash
# 1. main を最新化
git switch main
git pull origin main

# 2. 作業ブランチに戻る
git switch feature/user-login

# 3. main の上に自分のコミットを「再配置」する
git rebase main

# コンフリクトが起きたらここで止まる → 解消後 `git rebase --continue`
# やめるなら `git rebase --abort`

# 4. 強制プッシュ（履歴が書き変わるため）
git push --force-with-lease
```

> **⚠️ 重要: `--force-with-lease` を使え**
> `git push -f` (または `--force`) は、リモートに自分が知らないコミット（他人の Push など）があっても上書きしてしまいます。`--force-with-lease` は「自分が最後に pull した時点からリモートが変わっていない場合のみ強制できる」安全装置付きです。**常にこちらを使ってください。**

## 実践：コンフリクト（競合）を解消する

`rebase` や `merge`、`pull` の際に、同じファイルの同じ行を異なる人が編集しているとコンフリクトが発生します。慌てず対処しましょう。

### 1. 発生時の状態確認

```bash
git status
# Unmerged paths:
#   (use "git restore --staged <file>..." to unstage)
#   (use "git add <file>..." to mark resolution)
#         both modified:   src/auth/login.ts
```

### 2. 競合箇所を開く（VS Code 等のエディタ推奨）

ファイルを開くと以下のマーカーが挿入されています。

```typescript
<<<<<<< HEAD (main 側の変更)
const API_URL = 'https://api.example.com/v1';
=======
const API_URL = 'https://api.example.com/v2'; // 自分が変えた箇所
>>>>>>> feature/user-login (自分のブランチ側の変更)
```

### 3. 正しいコードに手動で修正する

マーカーを削除し、正しい最終形に編集します。

```typescript
// 修正後：v2 を採用し、コメントを残す
const API_URL = 'https://api.example.com/v2'; // v2 API 対応
```

### 4. 解消済みとしてマークして続行

```bash
# 解消したファイルをステージング
git add src/auth/login.ts

# rebase 中なら
git rebase --continue

# merge 中なら
git commit --no-edit # マージコミットメッセージはデフォルトでOK
git push
```

**現場の Tips**: コンフリクト解消は「自分の変更を優先」ではなく「正しい仕様に合わせる」視点で。不明ならレビュアーや担当者に即相談しましょう。

## よくあるトラブルシューティング集

### 間違えて `main` でコミットしちゃった！

```bash
# 1. 今のコミットを一時退避（ブランチ作成）
git branch feature/my-mistake

# 2. main をリモートの状態に戻す（ローカルのコミットを消す）
git reset --hard origin/main

# 3. 正しいブランチに移動して作業継続
git switch feature/my-mistake
```

### `git add` しすぎた / 間違えたファイルをステージした

```bash
# 特定ファイルをアンステージ（作業ディレクトリの変更は残る）
git restore --staged src/config/secrets.json

# 全部アンステージ
git restore --staged .
```

### コミットメッセージを間違えた（Push 前）

```bash
# 直前のコミットメッセージを書き換え
git commit --amend
# エディタが開くので修正して保存
```

### 過去のコミット（直前ではない）を修正したい / 順序を変えたい / 統合したい

**インタラクティブリベース** を使います。強力ですが履歴改竄なので Push 前のみ、かつ慎重に。

```bash
# 直前から 3 コミット分を対象にする
git rebase -i HEAD~3

# エディタで以下のように操作
# pick -> コミットを残す
# reword -> メッセージ変更
# edit -> 中身修正（amend するため止まる）
# squash / fixup -> 直前のコミットに統合（fixup はメッセージ捨てる）
# drop -> コミット削除
```

## まとめ：現場で「できる人」と思われる Git 運用の心得

1.  **`main` は触らない**。必ずブランチを切る。
2.  **作業開始は `git switch main && git pull` から**。最新土台の上に立つ。
3.  **コミットは「論理的な単位」で細かく**。`add -p` (ハンク単位ステージ) も覚えると便利。
4.  **コミットメッセージは「未来の自分（と他人）へのメモ」**。Prefix (`feat`, `fix` 等) をつける。
5.  **Push 前に `rebase` で履歴を整える**。コンフリクトはローカルで解消し、リモートを汚さない。
6.  **強制プッシュは `--force-with-lease` 一択**。
7.  **PR は小さく、早く出す**。大きすぎる PR はレビューされない。

これらを無意識にできるようになれば、あなたは「Git が使えるエンジニア」から「Git でチームを回せるエンジニア」へ一歩近づきます。まずは今のプロジェクトで、意識してこのフローを回してみてください。
