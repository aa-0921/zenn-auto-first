## はじめに

React のエコシステムは日々進化していますが、2023 年にリリースされた **Next.js 13** 以降、`pages` ディレクトリだけでなく **App Router**（`app` ディレクトリ）という新しいルーティング方式が登場しました。  
駆け出しエンジニアは「どちらを使えばいいの？」と戸惑いがちですし、サーバーコンポーネントとクライアントコンポーネントの境界が曖昧になると、ビルドエラーやランタイムエラーに悩まされます。

この記事では、以下を中心に解説します。

* App Router と Pages Router の基本的な違い  
* サーバーコンポーネント vs クライアントコンポーネントの役割と書き方  
* 「つまずきがちな」典型的エラーとその回避策  

実務で Next.js プロジェクトを立ち上げるときにすぐ使える知見としてまとめましたので、ぜひ手元のエディタで試しながら読んでください。

## 1. App Router と Pages Router の違い

### 1.1 ディレクトリ構造の違い

| Router | ルートディレクトリ | 代表的なファイル |
|--------|-------------------|-------------------|
| **Pages Router** | `pages/` | `pages/index.tsx`, `pages/about.tsx` |
| **App Router**   | `app/`   | `app/layout.tsx`, `app/page.tsx`, `app/dashboard/page.tsx` |

- **Pages Router** は従来の `pages` ディレクトリ配下にファイルを置くだけで自動的にルーティングが生成されます。`getStaticProps` / `getServerSideProps` といったデータ取得関数が必須です。
- **App Router** は `app` ディレクトリ配下に **レイアウト** (`layout.tsx`) と **ページ** (`page.tsx`) を分離し、階層的に UI を構築できます。データ取得は **Server Component** 内で直接行うか、`fetch` に `next` オプションを付与します。

### 1.2 データ取得の流れ

| Router | データ取得方法 | 取得タイミング |
|--------|----------------|----------------|
| Pages  | `getStaticProps` / `getServerSideProps` | ビルド時またはリクエスト時 |
| App    | Server Component 内で `fetch`（`next` オプション） | コンポーネントがサーバーでレンダリングされる瞬間 |

App Router では **ページ単位での `getStaticProps` が不要** です。代わりにサーバーコンポーネントが直接データを取得し、クライアントに送る HTML に埋め込まれます。

### 1.3 ルーティングの柔軟性

- **Pages Router** はファイルベースのルーティングだけで、動的パラメータは `[id].tsx` のようにファイル名で表現します。  
- **App Router** では **`(group)`** ディレクトリや **`[...slug]`** のようなキャッチオールも同様に使えますが、**レイアウトの継承** が可能です。たとえば `app/dashboard/layout.tsx` に共通のナビゲーションを置けば、`app/dashboard/**` 以下の全ページで自動的に適用されます。

> **ポイント**  
> プロジェクトを最初に立ち上げるときは、**「ページ単位で共通レイアウトが必要か」** を基準に選択すると良いです。シンプルなサイトなら Pages Router、ダッシュボードや複数のサブレイアウトが必要な大規模アプリなら App Router が向いています。

## 2. サーバーコンポーネントとクライアントコンポーネント

Next.js 13 からは **React Server Components (RSC)** が標準装備され、デフォルトは **サーバーコンポーネント** になります。  

### 2.1 それぞれの特徴

| コンポーネント | 実行環境 | 主な用途 | 代表的な制限 |
|----------------|----------|----------|--------------|
| **Server Component** | Node.js (サーバー) | データ取得、重いロジック、SEO 重視の UI | `useState`・`useEffect` などのフックは不可、ブラウザ API が使えない |
| **Client Component** | ブラウザ | インタラクティブ UI、状態管理、イベントハンドラ | サーバー側でのデータ取得は `fetch` のみ、サイズが大きくなるとクライアント側に余計な JavaScript が送られる |

### 2.2 宣言方法

```tsx
// src/app/dashboard/page.tsx   ← デフォルトは Server Component
export default async function DashboardPage() {
  const data = await fetch('https://api.example.com/stats', { cache: 'no-store' });
  const json = await data.json();

  return (
    <section>
      <h1>Dashboard</h1>
      {/* クライアント側でインタラクティブにしたい部分は別コンポーネント化 */}
      <StatsClient data={json} />
    </section>
  );
}

// クライアントコンポーネントは "use client" を必ず書く
'use client';
import { useState } from 'react';

export function StatsClient({ data }: { data: any }) {
  const [showDetail, setShowDetail] = useState(false);
  return (
    <div>
      <p>Total: {data.total}</p>
      <button onClick={() => setShowDetail(v => !v)}>
        {showDetail ? 'Hide' : 'Show'} Detail
      </button>
      {showDetail && <pre>{JSON.stringify(data.detail, null, 2)}</pre>}
    </div>
  );
}
```

- **サーバーコンポーネント**は `export default` だけで OK。  
- **クライアントコンポーネント**はファイルの先頭に必ず `'use client';` を書きます。これが無いとビルド時に「フックがサーバーコンポーネントで使われている」エラーになります。

### 2.3 つまずきがちなケース

#### ケース A: `useEffect` をサーバーコンポーネントで書いてしまう

```tsx
export default function BadComponent() {
  useEffect(() => { console.log('mounted'); }, []); // ← ここでエラー
  return <div>Hello</div>;
}
```

**対策**  
- `useEffect` が必要なら必ずクライアントコンポーネントに切り出す。  
- もしくは `use client` を付与して全体をクライアント化するが、不要な JavaScript が増えるので避ける。

#### ケース B: ブラウザ API (`window`, `document`) をサーバーコンポーネントで呼び出す

```tsx
export default function Bad2() {
  const width = window.innerWidth; // サーバー側では undefined
  return <p>{width}</p>;
}
```

**対策**  
- `useEffect` 内で取得し、クライアントコンポーネントに持ち込む。  
- もしくは `next/dynamic` で遅延ロードし、`ssr: false` にしてクライアント側のみでレンダリングさせる。

```tsx
import dynamic from 'next/dynamic';

const WidthComponent = dynamic(() => import('./WidthComponent'), { ssr: false });
```

### 2.4 最適化のヒント

| 目的 | 手法 |
|------|------|
| **HTML のサイズを小さくしたい** | 可能な限りサーバーコンポーネントに留め、クライアントコンポーネントはインタラクティブ部分だけに絞る |
| **データ取得をキャッシュしたい** | `fetch` の第2引数に `{ next: { revalidate: 60 } }` を付与し、ISR (Incremental Static Regeneration) を活用 |
| **レイアウトの再利用** | `app/layout.tsx` に共通ヘッダー/フッターを置き、子ページは自動で継承させる |

## 3. 実務でよく遭遇するエラーと対処法

### 3.1 `Error: You cannot use <Component> inside a Server Component`  

**原因**  
サーバーコンポーネント内でクライアントコンポーネントを直接インポートしたが、`'use client'` が付与されていない。

**対処**  
1. クライアントコンポーネントのファイル冒頭に `'use client';` を追加  
2. 必要ならインポートを遅延させる（`dynamic`）  

```tsx
// client-only.tsx
'use client';
export default function ClientOnly() { /* ... */ }
```

### 3.2 `Error: Route segment "app" does not exist`  

**原因**  
`next dev` 時に `app` ディレクトリが存在しないか、`next.config.js` で `experimental.appDir` が無効化されている。

**対処**  
- `next.config.js` に以下を追加（Next.js 13.4 以降はデフォルトで有効）:

```js
module.exports = {
  experimental: {
    appDir: true,
  },
};
```

- `app/` ディレクトリがプロジェクト直下にあることを確認。

### 3.3 `Hydration failed because the initial UI does not match`  

**原因**  
サーバーコンポーネントとクライアントコンポーネントで出力が食い違うケース（例: `Date.now()` をサーバー側で計算している）。

**対処**  
- クライアント側でのみ変化する値は `useEffect` 内で計算する。  
- もしくは `use client` コンポーネントに切り出し、サーバー側ではプレースホルダーを返す。

```tsx
// ServerComponent.tsx
export default function ServerComponent() {
  return (
    <div>
      <p>Server time: {/* placeholder */}--:--:--</p>
      <ClientClock />
    </div>
  );
}
```

### 3.4 `next/font` が動かない  

**原因**  
App Router では `next/font` のインポートが **トップレベル** で行われている必要があります。ページコンポーネント内部でインポートするとビルドエラーになることがあります。

**対処**  
`app/layout.tsx` や `app/page.tsx` の最上部でインポート:

```tsx
import { Inter } from 'next/font/google';
const inter = Inter({ subsets: ['latin'] });

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="ja" className={inter.className}>
      <body>{children}</body>
    </html>
  );
}
```

## 4. まとめ

- **App Router** はレイアウト継承とサーバーコンポーネントの組み合わせで、モダンな大規模アプリに最適。  
- **Pages Router** はシンプルさが魅力で、静的サイトや小規模プロジェクトに向く。  
- デフォルトは **サーバーコンポーネント**。インタラクティブが必要な部分だけ **クライアントコンポーネント** に分割し、`'use client'` を忘れずに書くことがエラー回避の鍵。  
- 代表的なエラーは「サーバー/クライアントの境界」「ディレクトリ構成」「Hydration の不一致」なので、上記対策を覚えておけば実務でのトラブルは大幅に減ります。

Next.js の新機能は最初は戸惑うかもしれませんが、**「サーバーでできることはサーバーへ、クライアントで必要なことはクライアントへ」** というシンプルな指針を持てば、自然に適切なコンポーネント分割ができるようになります。ぜひ本記事のコード例を手元で試し、プロジェクトに合ったルーティングとコンポーネント設計を確立してください。
