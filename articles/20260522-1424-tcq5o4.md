### エラーの例

```
ActionController::RoutingError (No route matches [GET] for /userss):
  Visit https://docs.rubyonrails.org/routing.html for more information.
```

このエラーは、ただつづりを誤ったり、ルート定義を忘れたりすると発生します。

### デバッグの手順

1. まずは `rails routes` で現在のルーティング一覧を確認します。

```bash
$ rails routes
Prefix  Verb  URI Pattern                Controller#Action
users   GET   /users(.:format)           users#index
user    GET   /users/:id(.:format)       users#show
```

2. ビュー側でリンクを書いた際、パス名や引数を見直します。

```erb
<%# 間違い例 %>
<%= link_to 'ユーザー一覧', users_path %>  <%# users_path が定義されていればOK %>
<%= link_to 'ユーザー詳細', user_path(@user) %>  <%# :id が必要なので忘れずに %>
```

3. ルートの定義をチェックします。`config/routes.rb` を確認し、`resources :users` があるか、カスタムルートが正しく設定されているかチェックします。

```ruby
# config/routes.rb
Rails.application.routes.draw do
  resources :users, only: [:index, :show]
  # root to: 'home#index' など他のルートも忘えずに
end
```

### 解決のヒント

- ルート名は `rails routes` で必ず確認する
- パスヘルパーメソッドは Rails が提示するものを使用する
- コントローラ名とアクション名はスネークケースではなくキャメルケースに注意する（例：`UsersController` ではなく `users_controller.rb` ではない）

## 2. マイグレーションエラー：競合やロールバック失敗

チーム開発をしていると、他人のマイグレーションファイルと競合したり、マイグレーションのロールバックが失敗することがあります。

### エラーの例

```
ActiveRecord::StatementInvalid: SQLite3::ConstraintException: UNIQUE constraint failed: users.email
```

これは、メールアドレスに一意性制約があり、同じメールアドレスを登録しようとした場合に発生します。

### デバッグの手順

1. マイグレーションファイルで制約を見直します。

```ruby
# db/migrate/20240701000000_create_users.rb
class CreateUsers < ActiveRecord::Migration[7.0]
  def change
    create_table :users do |t|
      t.string :name, null: false
      t.string :email, unique: true  # 一意性制約
      t.timestamps
    end
  end
end
```

2. データベースの状態を確認します。

```bash
$ rails dbconsole
sqlite> SELECT * FROM users;
```

3. 競合しているレコードを削除したり、更新します。

```ruby
# rails console で実行
User.where(email: 'duplicate@example.com').destroy_all
```

### ロールバックの落とし穴

`rake db:rollback` を実行した際、`down` メソッドが正しく定義されていないと、一部の変更が元に戻らないことがあります。

```ruby
# 悪い例：down メソッドがない
class AddColumnToUsers < ActiveRecord::Migration[7.0]
  def up
    add_column :users, :nickname, :string
  end
  # down がない！
end
```

```ruby
# -good example
class AddColumnToUsers < ActiveRecord::Migration[7.0]
  def up
    add_column :users, :nickname, :string
  end

  def down
    remove_column :users, :nickname
  end
end
```

### 解決のヒント

- マイグレーションはチームで共有する前に必ずテスト環境で動作確認
- ロールバック可能な `change` メソッドを使用する（自動で `up`/`down` を生成）
- 開発環境では `rake db:drop db:create db:migrate:reset` でクリーンスラートからやり直すことも手軽

## 3. N+1問題：APIリクエスト数が爆発する

N+1問題とは、1つのリクエストでN個の関連レコードを取得する際、1回のクエリで親を取得し、その後にそれぞれの子に対してクエリを発行することです。

### 問題の例

```ruby
# コントローラ
def index
  @users = User.all
end

# ビュー（erb）
<% @users.each do |user| %>
  <%= user.name %> (<%= user.posts.count %> posts)
<% end %>
```

これまでに発行されるクエリ：

```
User Load (1ms)  SELECT * FROM users
Post Count (1ms) SELECT COUNT(*) FROM posts WHERE posts.user_id = 1
Post Count (1ms) SELECT COUNT(*) FROM posts WHERE posts.user_id = 2
Post Count (1ms) SELECT COUNT(*) FROM posts WHERE posts.user_id = 3
...
```

50人のユーザーがいたら、1回 + 50回 = 51回のクエリが発生します。

### 解決策：includes とは

```ruby
# 修正後のコントローラ
def index
  @users = User.includes(:posts)
end

# ビュー
<% @users.each do |user| %>
  <%= user.name %> (<%= user.posts.length %> posts)
<% end %>
```

このように `includes` を使うと、事前に関連する `posts` テーブルをロードし、N+1を回避できます。

### さらに詳細を知りたい場合

関連データのカラムだけ欲しい場合は、`joins` と `select` を組み合わせます。

```ruby
@users = User.joins(:posts).select('users.*, COUNT(posts.id) as posts_count')
              .group('users.id')
```

また、ビュー側で個別にカウントしなくても、インスタンス変数にハッシュやカウント済みのデータを渡す方法もあります。

### 解決のヒント

- 画面表示時は必ず `includes` や `preload` を検討する
- クエリログを有効化してチェックする（開発環境では `config.log_level = :debug`）
- Bullet gem を使うと、N+1問題を検出してくれる

## まとめ

Rails開発において「デバッグのコツ」をまとめると、次の3点に集約されます。

1. **事前の確認が基本**：`rails routes`、`rails console`、ログ出力をうまく活用する
2. **規約に沿うこと**：コントローラ名、ルーティング、マイグレーションはRailsの規約に従う
3. **パフォーマンスを意識する**：N+1問題には `includes` を、競合には一意性制約を意識する

デバッグとは「なぜそのエラーが出ているのか」を丁寧にたどる作業です。最初は時間がかかるかもしれませんが、慣れてくると「あるある」なエラーの原因が見えてくるようになります。
