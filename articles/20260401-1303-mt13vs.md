## 導入
React はコンポーネントベースの UI ライブラリですが、実務で開発を進める中で「エラーが頻出する」「原因が分からない」といった悩みがつきものです。特に初心者やチームに新しいメンバーが加わったときに、よくある落とし穴が表れます。本記事では、実務で頻繁に遭遇するエラーを 3 種類ピックアップし、具体的なコード例とともに解決手法を解説します。対象読者は駆け出しエンジニアから中級者までを想定し、実装時にすぐ試せるヒントを提供します。

## 背景・前提
React では状態管理や副作用処理に JavaScript の仕様が深く関わっています。したがって、**Hooks の振る舞い**（`useEffect` の依存配列、`useState` の再レンダー制御）や **レンダリングの最適化**（`key` の指定）といった点でエラーが顕在化しやすいです。以下で想定している前提は次のとおりです。

- React 18 以上、TypeScript は任意で使用可
- 関数コンポーネントと Hooks のみを使用- プロジェクトは `create-react-app` や Vite などで初期化済み

これらの前提の下で、実際に起きやすいエラーとその対策を順に見ていきましょう。

## 代表的なエラーと対処法

### 1. `useEffect` の依存配列エラー
`useEffect` は第 2 引数に「依存配列」を取りますが、これを省略したり要素を忘れたりすると、意図しないタイミングで何度も実行されます。典型的なエラーは「空の配列に API 呼び出しが入っている」ことです。

```tsx
useEffect(() => {
  fetch('/api/data')
    .then(r => r.json())
    .then(setData);
}, []); // ← ここで空の配列を渡すと、最初の 1 回だけ実行されるが、データが更新されないケースが多い
```

#### 解決策
- **依存すべき変数を明示的に列記**  
  ```tsx
  useEffect(() => {
    fetch(`/api/data?user=${userId}`)
      .then(r => r.json())
      .then(setData);
  }, [userId]); // userId が変わったときだけ実行  ```
- **空の配列が意図しない場合は `[]` の代わりに `undefined` や `null` を使う**  
  ```tsx
  useEffect(() => {
    // 初回マウント時だけ実行したいときは空の配列でも OK
  }, []);
  ```
- **依存配列が複雑なオブジェクトの場合は `useRef` で安定化**  
  ```tsx
  const prevData = useRef<Data[]>();
  useEffect(() => {
    if (prevData.current) {
      // 前回データと比較して差分があれば処理    }
    prevData.current = data;
  }, [data]);
  ```

### 2. 不要な再レンダーの抑止
コンポーネントは props や state が変わると必ず再レンダーされますが、不要な再レンダーはパフォーマンス低下の原因になります。主な原因は **不要なオブジェクトや配列の生成** です。

#### 事例
```tsx
const [items, setItems] = useState<Item[]>([]);

useEffect(() => {
  // 毎回新しいオブジェクトを作成 → 再レンダーがトリガーされる
  const config = { limit: 10, offset: 0 };
  fetch(`/api/items?${new URLSearchParams(config)}`)
    .then(r => r.json())
    .then(setItems);
}, []);
```

#### 解決策
- **`useMemo` で安定したオブジェクトを作成**  
  ```tsx
  const config = useMemo(() => ({ limit: 10, offset: 0 }), []);
  useEffect(() => {
    fetch(`/api/items?${new URLSearchParams(config)}`)
      .then(r => r.json())
      .then(setItems);
  }, []);
  ```
- **`useCallback` で関数を安定化**  
  ```tsx
  const handleClick = useCallback(() => {
    setCount(c => c + 1);
  }, []);
  ```
- **`React.memo` で子コンポーネントの不要な再描画を防止**  
  ```tsx
  const ItemList = React.memo(({ items }: { items: Item[] }) => (
    <ul>{items.map(i => <li key={i.id}>{i.name}</li>))}</ul>
  ));
  ```

### 3. キーの付け忘れ
リストやテーブルを描画する際に `key` 属性を付け忘れると、React が状態の差分計算に失敗し、意図しない再描画や UI の崩れが起きます。

#### エラー例```tsx
const items = [{ id: 1, name: 'Alice' }, { id: 2, name: 'Bob' }];

return (
  <ul>
    {items.map(item => (
      <li>{item.name}</li> // ← key 属性なし
    ))}
  </ul>
);
```

#### 解決策
- **唯一の ID を key に使用**    ```tsx
  {items.map(item => (
    <li key={item.id}>{item.name}</li>
  ))}
  ```
- **インデックスを key にしたい場合は注意**  
  ```tsx  {items.map((item, idx) => (
    <li key={idx}>{item.name}</li> // 変更が頻繁に起きるときは不安定になるので避ける
  ))}
  ```
- **動的生成時は `crypto.randomUUID()` でユニークキーを生成**  
  ```tsx  const uuid = crypto.randomUUID();
  <div key={uuid}>{/* ... */}</div>
  ```

## まとめ
React で発生しやすいエラーは「Hooks の依存配列」「不要な再レンダー」「キーの付け忘れ」の大きく分けて 3 種類です。これらは **「何を意図しているか」** を明確にし、**「依存関係を安定させる」** ことで対処できます。具体的には

1. 依存配列は必要な変数を正確に列挙し、`useRef` でオブジェクトの安定化を図る  
2. 不要なオブジェクト・関数生成は `useMemo` / `useCallback` で最適化し、子コンポーネントは `React.memo` で差分チェックをさせる  
3. リストの描画時は必ず `key` を付与し、インデックス利用はリスクを理解した上で行う  

これらのポイントを意識すれば、実務でも遭遇する頻度の高いエラーを未然に防ぎ、パフォーマンスと保守性を向上させることができます。ぜひ開発フローに組み込んでみてください。
