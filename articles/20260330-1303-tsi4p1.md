## 導入
 Docker はコンテナ技術を活用して、アプリケーションの実行環境を軽量かつ再現性の高い形で提供するツールです。従来の仮想マシンに比べ、起動が速くリソース消費も抑えられるため、ローカル開発や CI/CD パイプラインで広く利用されています。本記事では、駆け出しエンジニアが Docker の基本概念を理解し、実務で即使える開発環境を構築する手順を解説します。

## 背景・前提
- **コンテナ** と **イメージ** の違い  
  - **イメージ** は、コンテナを作成するための「青prints」的なスナップショットです。 Dockerfile で定義し、レジストリから取得・ビルドできます。  
  - **コンテナ** は、イメージを実行したときに生成されるランタイムの実体です。起動・停止・削除が自由に行える「実体」です。  
- **Docker Compose** は、複数コンテナを同時に起動・管理するためのツールです。 `docker-compose.yml` でサービス間の依存関係やネットワーク設定を宣言でき、ローカル開発環境の全体像をコード化できます。  
- 本記事では、Go 言語の簡単な Web アプリを例に、Docker イメージの作成、コンテナの起動、Compose でのマルチサービス構成、そして開発時のメリットを具体的に示します。

## Docker 基礎：イメージとコンテナ
### Dockerfile の書き方
```Dockerfile
# Dockerfile.example
FROM golang:1.22-alpine AS builder
WORKDIR /appCOPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o /out/app .

FROM alpine:latest
WORKDIR /root/
COPY --from=builder /out/app /root/app
CMD ["/root/app"]
```
上記 Dockerfile は、ビルド段階（`builder`）と実行段階（`final`）の 2 段階を用意し、最終イメージは最小化された `alpine` にだけ実行ファイルをコピーします。これにより、イメージサイズが数十 MB 程度に抑えられます。

### イメージのビルド・実行```bash
# イメージのビルド
docker build -t my-go-app .

# コンテナの起動
docker run -p 8080:8080 my-go-app
```
`-p` オプションでホストポートとコンテナポートをマッピングし、ローカルの `localhost:8080` からアプリにアクセスできます。コンテナはバックグラウンドで実行したい場合は `-d` オプションを付けます。

## Docker Compose の基本
### compose ファイルの例
```yaml
# docker-compose.ymlversion: "3.9"
services:
  web:
    build: .
    ports:
      - "8080:8080"
    environment:
      - ENV=development
  db:
    image: postgres:15-alpine
    environment:
      POSTGRES_USER: appuser
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: appdb
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```
- `web` サービスは上記 Dockerfile からビルドされたイメージを使用し、8080 番ポートを公開します。  
- `db` サービスは公式 PostgreSQL イメージを利用し、`volumes` で永続化設定を入れています。  
- `depends_on` や `networks` を追加すれば、サービス間の依存関係やネットワークも細かく制御可能です。

### Compose での起動・停止
```bash
# コンテナ群の起動（バックグラウンド）
docker-compose up -d# ログの確認
docker-compose logs -f

# すべてのコンテナを停止・削除
docker-compose down
```
`up -d` でデタッチドモードで起動し、`logs -f` でリアルタイムログを追えます。開発中はコード変更を検知して再ビルドしたい場合は、`docker-compose up --build -d` を使用します。

## ローカル開発での Docker のメリット
1. **環境の再現性**  
   - ローカルの OS やバージョンに依存せず、同一 Dockerfile と Compose ファイルだけで全メンバーが同一環境を保てます。  
2. **依存関係の分離**  
   - データベースやキャッシュサーバーなど、他サービスとの衝突を防げます。  
3. **高速なセットアップ**  
   - `docker-compose up` だけで数秒で全サービスが起動。手動でインストールや設定を行う手間が大幅削減。  
4. **CI/CD への即投入**  
   - ローカル環境と同一のイメージを CI パイプラインで使用すれば、デバッグと本番環境のギャップが減ります。  
5. **リソース制御**  
   - `mem_limit` や `cpus` などでコンテナごとにリソース上限を設定でき、開発マシンの負荷を抑えられます。

## まとめ
- Docker の **イメージ** はコンテナの雛形、**コンテナ** はその雛形を実行した実体です。  
- `Dockerfile` でイメージを作成し、`docker build` / `docker run` でコンテナを操作します。  
- `docker-compose.yml` にサービス定義を書けば、複数コンテナの起動・管理が一瞬で完結します。  
- ローカル開発では、**環境の再現性・依存関係の分離・高速セットアップ** といったメリットが得られ、チーム全体の生産性が向上します。  
- まずはシンプルな `docker-compose up` から始め、徐々にマルチステージビルドやボリューム永続化、ネットワーク設定などを追加していくのが学習の近道です。  
- 今回紹介したコードや設定は、実際のプロジェクトに即座にコピー＆ペーストできるテンプレートとして活用してください。
