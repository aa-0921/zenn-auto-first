### 背景
私は2020年にフロントエンドエンジニアとして転職したとき、技術の幅広さに圧倒されました。フレームワークの公式ドキュメントを読むのが苦手で、YouTubeのチュートリアル動画を無尽蔵に見続けるだけだったのです。しかし、3ヶ月後に実務で使われるフレームワークが変更になり、それまでの学習が無駄になったことがあります。その時、学習法を見直すきっかけとなりました。

---

### ① インプットとアウトプットのバランスを意識する

#### 問題意識
最初の1年半は「学習モード」にハマっていました。  
- 毎日YouTubeのチュートリアル動画を見た  
- 公式ドキュメントのセクションを1つ読破した  
- だが実際のコード作成は1日1〜2ファイル程度  

この状態が続くと、知識は蓄積されず、「知っているつもり」と「使える」の間にギャップが生じていました。

#### 改善策
1. **1:1のルールを導入**  
   例えば、Reactのチュートリアル動画を視聴した後、その日のうちに同じ機能を自分で実装することを必須としました。  
   ```javascript
   // 例: 画面遷移の実装
   function useNavigation() {
     const history = useHistory();
     const navigate = (path) => history.push(path);
     return { navigate };
   }
   ```

2. **「なぜ？」を尋ねる習慣**  
   実装したコードの意味をただ受け入れず、公式ドキュメントでその仕組みを調査する習慣をつけました。  
   ```bash
   # 例: コマンドラインでの確認
   npm info react-router-dom
   ```

#### 効果
3ヶ月で、フレームワークのバージョン変更に対応できるようになりました。  
「学習」から「実装」「検証」のサイクルが自然にできるようになったのが大きな成長点です。

---

### ② ドキュメントの読み方を学ぶ

#### 初期の苦悩
あるプロジェクトでVue.jsを導入した際、公式ドキュメントの以下の部分に困惑しました。  
```vue
<template>
  <div>{{ message }}</div>
</template>

<script>
export default {
  data() {
    return {
      message: 'Hello Vue!'
    }
  }
}
</script>
```

「なぜ`data()`内に`message`を定義するのか？テンプレート内で直接宣言できないのか？」と悩みました。

#### 解決策
1. **「公式ドキュメントは参考書」と考える**  
   最初は「すべてを覚える」目標でしたが、実は「必要な情報の見つけ方」が重要です。  
   - 検索機能（Ctrl+F）を活用  
   - 「APIリファレンス」セクションを優先的に読む  

2. **Markdown形式のメモを作成**  
   例えば、Vueのコンポーネント構造は以下のようにまとめました。  
   ```markdown
   ### コンポーネント構造
   - `<template>`：HTMLに近い記述
   - `<script>`：ロジック（data, methods）
   - `<style>`：CSS（scoped属性でスコープ制限）
   ```

#### 成果
ドキュメントの読み方が「雑にブラウジング」から「目的の情報を効率よく読む」に変わり、開発速度が2倍に向上しました。

---

### ③ 写経と個人開発の活かし方

#### 写経の意義
フレームワークのコードを「写経」することで、構文や設計の本質を理解できるようになりました。  
- 例: ReactのコンテキストAPIの実装を手打ち  
  ```javascript
  // 公式サンプルをベースに修正
  const ThemeContext = React.createContext('light');
  function ThemeProvider({ children }) {
    return (
      <ThemeContext.Provider value="dark">
        {children}
      </ThemeContext.Provider>
    );
  }
  ```

#### 個人開発の重要性
写経だけでは不十分だと気づき、自身のポートフォリオサイトを構築しました。  
- 目的：学んだ技術を実際のプロジェクトに適用  
- 成果：GitHubのIssue修正を経験し、クリーンなコードの重要性を学ぶ  

#### 具体的なステップ
1. **「ゼロから作る」より「既存コードを改良する」**  
   例: 既存のポートフォリオサイトにダークモードを追加  
2. **PR（Pull Request）を送る**  
   - 企業内プロジェクトでも、小さな改善点を自発的に提案  
   - 例: 「ローディングインディケータのアニメーションを滑らかに」  

---

### まとめ：学習は「繰り返し」の連続

駆け出しエンジニア時代に効いた学習法は、以下の3つの軸に集約されます。  
1. **インプットとアウトプットのバランス**  
2. **ドキュメントの読み方を学ぶ**  
3. **写経と個人開発の活かし方**  

特に重要なのは、「学習」を「実装」のサイクルに組み込むことです。  
技術は「知っている」だけでなく、「使える」ことがエンジニアとしての成長の鍵となります。  

---

### 補足：おすすめの学習リソース
- **公式ドキュメント**：Vue.js、Reactの公式ドキュメントは「例コードの構造」がわかりやすい  
- **写経用ツール**：CodeSandbox、StackBlitzでオンラインで実装可能  
- **個人開発のテンプレート**：GitHubの「スター押し」プロジェクトを参考に  

> **「学習は知識の蓄積ではなく、実装の繰り返し」**  
> この経験が、今後のエンジニアとしての基盤になります。
