## 入力とアウトプットのバランス

### 1. 入力の質と量をコントロールする

- 1日30分程度の「 focused reading」: 目的を持って資料を読む。  
- 1記事または1本の動画を「ノート」にまとめ、要点を自分の言葉に書き換える。  

### 2. アウトプットで定着させる

- 学んだことをブログやQiitaに投稿し、実際に手を動かすタスクを作る。  
- 復習用にコードスニペットをGitHub Gistに保存し、後で再実装する。  

このサイクルを回すことで、知識が短期記憶から長期記憶へ移行しやすくなります。

## ドキュメントの読み方

技術文書は「目的」→「前提」→「手順」→「例」の順に構成されていることが多いです。まずは「目次」や「はじめに」で全体像を把握し、次に「API reference」や「チュートリアル」の章を skim し、最後に「コード例」だけで動作確認を試す。具体的な読み方のステップは次のコードブロックで示します。

```python
# 例: Python の公式ドキュメントを読むときの手順
import requests

# 1. 目的を明確に
url = "https://docs.python.org/3/library/requests.html"

# 2. 目次を確認（実際はブラウザで閲覧）
print("目次を確認してください")

# 3. 例コードをコピーして実行
# 例: requests の基本的な使い方
response = requests.get("https://api.github.com")
print(response.status_code)
```

## 写経と個人開発の活かし方

### 写経（コピー練）のコツ

- 完全に同じコードを書くのではなく、**自分のタスクに合わせて変更**しながら実装する。  
- コードのコメントを自分の言葉で書き換えると、理解が深まります。  

### 個人開発で実践

- 小さめの機能（例: CLI ツール）を1週間でリリースする。  
- 開発中に encountered した課題を「学習ノート」にまとめ、次回のインプットに活用する。  

## 実践ステップとコード例

以下は「入力 → アウトプット → 復習」の一連の流れをコードで表現したサンプルです。

```bash
# 1. 入力: 学習テーマを選択（例: Docker）
THEME="docker"

# 2. アウトプット: そのテーマについてブログ記事を作成
cat <<EOF > blog_${THEME}.md
# ${THEME} 入門

## 基本コンセプト
- コンテナ化のメリット
- Dockerfile の構造

## 実装例
\`\`\`dockerfile
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y curl
CMD ["curl", "https://example.com"]
\`\`\`
EOF

# 3. 復習: 実際に Dockerfile を作成し実行
cat blog_${THEME}.md | grep -A5 "Dockerfile" | bash
```

## まとめ

- **入力**は「目的を持った読書」＋「要点のノート化」で質を高める。  
- **アウトプット**は「ブログ投稿」や「小規模開発」でアウトプットし、アウトプットの量がインプットの1〜2割になるように調整する。  
- **ドキュメント**は「目次→概要→手順→例」の順で読み、コード例だけで動作確認を繰り返す。  
- **写経**は自分のタスクに合わせて改変し、コメントで自分の理解を言語化する。  

このバランスを意識すれば、駆け出しエンジニアでも実務で即活かせるスキルが身につきます。ぜひ daily 15分の「入力→アウトプット」サイクルを習慣化してください。
