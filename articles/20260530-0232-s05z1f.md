### Pages Router の特徴と利点  
従来の Pages Router は、ファイルベースのルーティングを採用。  
```bash
# 例: pages/about.js → /about へマッピング
export default function AboutPage() { return <h1>About</h1> }  
```  
- **利点**:  
  - ファイル構造が直感的で、ページ単位の開発に適する  
  - 静的サイトジェネレーション（SSG）や国際化（i18n）の実装が簡単  

### App Router の登場：パスベースのルーティングとネスト構造  
App Router は、**`app/` ディレクトリ内のファイル構造でルートを定義**します。  
```bash
# 例: app/dashboard/page.js → /dashboard にマッピング  
export default function DashboardPage() { return <h1>Dashboard</h1> }  
```  
- **特徴**:  
  - パスのネスト構造でコンポーネントを再利用可能（例: `app/dashboard/[postId]/route.js` → `/dashboard/123`）  
  - サーバーコンポーネントとの統合が円滑  
- **注意点**:  
  - クライアントルーティング（`useRouter`）とサーバーサイドの動作が異なる  
  - 静的ルートの作成には `generateStaticPaths` の代わりに `generateStaticParams` を使用  

## サーバーコンポーネント vs クライアントコンポーネント：適切な選択肢を判断する  
Next.js 13 以降では、**サーバーコンポーネント**がデフォルトで導入されました。クライアントコンポーネントとの使い分けが重要です。  

### サーバーコンポーネントの強みと使用ケース  
```tsx  
// サーバーコンポーネント（デフォルト）  
export default function ServerComponent() {  
  // サーバーサイドで動く処理（DB連携など）  
  const data = await fetchData();  
  return <div>{data}</div>;  
}  
```  
- **利点**:  
  - サーバーでデータ処理を行い、クライアントに最小限のデータを送信  
  - パフォーマンス最適化（SSR/SSG対応）  
- **注意点**:  
  - クライアントの状態管理やイベント処理はできない  

### クライアントコンポーネントの活用方法  
```tsx  
// クライアントコンポーネント（`use client`ディレクティブ）  
'use client';  
export default function ClientComponent() {  
  // イベント処理やDOM操作が必要な場合  
  return (  
    <button onClick={() => console.log('Clicked')}>  
      Click me  
    </button>  
  );  
}  
```  
- **利点**:  
  - ユーザー操作に即時反応するUI構築  
  - Web APIやローカルストレージ操作可能  

## 実務で役立つ：App Routerとコンポーネントの使い分けのコツ  
### 1. ルーティング構造の設計  
- **App Router**: パスのネストや動的ルートが必要な場合（例: ブログアプリの `/posts/[id]`）  
- **Pages Router**: 静的ページの大量生成や国際化が優先されるプロジェクト  

### 2. サーバーコンポーネントの活用シナリオ  
- データベース操作（`prisma`, `supabase`など）  
- 認証フロー（`next-auth`のサーバーサイドロジック）  

### 3. クライアントコンポーネントの使いどころ  
- フォームのリアルタイムバリデーション  
- マップやチャートのインタラクティブな可視化  

### 4. エラー境界とデータロードのベストプラクティス  
```tsx  
// サーバーコンポーネントでのエラー処理  
export default function ErrorBoundary({ children }) {  
  const [error, setError] = useState(null);  
  const [retry, setRetry] = useState(false);  

  try {  
    return children;  
  } catch (err) {  
    setError(err);  
    return <div>{error.message}</div>;  
  }  
}  
```  

## まとめ：Next.js の進化に対応するためのチェックリスト  
1. **ルーティング方式の選定**:  
   - パスのネストが必要 → App Router  
   - 静的ページの大量生成が必要 → Pages Router  
2. **コンポーネントの選択**:  
   - サーバー処理が必要 → サーバーコンポーネント  
   - クライアント側のロジックが必要 → クライアントコンポーネント  
3. **プロジェクトのスケーラビリティ**:  
   - App Routerは今後の進化に対応しやすいが、 Pages Routerは安定性が高い  

Next.js の進化は「柔軟性」と「安定性」のバランスを求めるものです。実務では、プロジェクトの要件に応じて両方のルーティング方式とコンポーネントタイプを組み合わせることが推奨されます。
