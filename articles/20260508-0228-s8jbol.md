## 背景・前提
### 1. エンジニア学習の落とし穴
- **情報過多**：書籍・ブログ・動画が氾濫。何を読むか迷う。
- **「知っている」誤謬**：読んだだけで実務に落とし込めると考える。
- **アウトプット不足**：コードを書かずに終わると、理解が薄いまま。

### 2. 私の学習スタイル
- **インプット**：1日30分、公式ドキュメント＋記事を読む。
- **アウトプット**：1週間に1回、ブログやスライドでまとめる。
- **実践**：小さな個人プロジェクトを立ち上げ、実装とデバッグを繰り返す。

## 本文
### 1. インプット：ドキュメントの読み方を変える
#### 1-1. 「まずは概要を把握」  
公式ドキュメントは長文だが、最初に「概要」「概要」セクションをざっと読む。  
> *例：Node.js のドキュメントで「Event Loop」の概要を読むだけで、後の詳細が理解しやすくなる。*

#### 1-2. 「疑問点をメモ化」  
読み進める際に「わからない用語」「使い方が不明」などをメモしておく。  
> *メモはマインドマップやMarkdownでまとめ、後で検索しやすくする。*

#### 1-3. 「一度に全部を消化しない」  
ドキュメントはセクションごとに区切られているので、1セクションずつ読んで小さく区切る。  
> *「Promise」の詳細を読む前に「非同期処理」の基礎を復習すると、混乱しにくい。*

### 2. アウトプット：書く・話す・共有する
#### 2-1. ブログ記事を書く
- **構成**：導入・背景・解決策・まとめ
- **文字数**：最低でも800文字。実際に1500文字程度にすると読者の目に留まりやすい。

```markdown
## はじめに
Node.js の Event Loop がどう動くかを理解したい。

## 1. Event Loop の概要
- タスクキュー
- マイクロタスク
- タイマー

## 2. 実際に動かしてみよう
```js
setTimeout(() => console.log('timeout'), 0);
Promise.resolve().then(() => console.log('promise'));
console.log('sync');
```

## 3. まとめ
...
```

#### 2-2. スライドを作る
社内勉強会で発表できるスライドを作ることで、説明力と記憶の定着を図る。

#### 2-3. コミュニティで質問・回答
- **Q&Aサイト**：Stack Overflow や Zenn のコメントで質問し、回答を自分で書く。
- **仲間のコードレビュー**：コードを共有し、レビューを受けることでアウトプットの質が上がる。

### 3. 写経と個人開発の活用
#### 3-1. 写経のメリット
- **構文の定着**：書くことで構文が頭に入る。
- **設計パターンの理解**：既存コードを追いながら、設計意図を読み取れる。

```go
// 例：GoでのWriteCloserインターフェースを写経
type WriteCloser interface {
    Write(p []byte) (n int, err error)
    Close() error
}
```

#### 3-2. 個人開発の実践
- **ミニプロジェクト**：自分が興味あるテーマを選び、実装を完結させる。
- **GitHubに公開**：READMEやIssueでドキュメント化。レビューを受けることで実務に近い経験が得られる。

#### 3-3. フィードバックの取り込み
- **コードレビュー**：同僚やオープンソースコミュニティにレビューを依頼。
- **Issueトラッキング**：バグを発見したらIssueに記載し、解決策をドキュメント化。

### 4. 学習サイクルの最適化
| ステップ | 時間配分 | 目的 |
| :------ | :------ | :------ |
| 1. インプット | 1日30分 | 基本知識の取得 |
| 2. アウトプット | 週1回 | 理解の定着 |
| 3. 実践 | 週1回 | 実務適応力 |
| 4. フィードバック | 週1回 | 改善と成長 |

- **反復**：同じテーマを何度も繰り返し学習し、記憶を定着させる。
- **評価**：学習後は「何が理解できたか」「実務でどう活かせるか」を自問自答。

## まとめ
- **インプットは「読む」だけで終わらせない**：メモ化と疑問整理を行い、ドキュメントをアウトプットに変える。
- **アウトプットは実際に書く・話す・共有する**：ブログ・スライド・コードレビューで内容を外に出す。
- **写経と個人開発は知識の定着と応用力を同時に養う**：実際に手を動かし、問題解決のプロセスを体験する。
- **学習サイクルを可視化し、定期的に振り返る**：時間配分と成果を見える化することで、学習のモチベーションを維持できる。

駆け出しエンジニアは「知識を詰め込む」より「知識を活かす」ことに重点を置くべきです。  
上記の学習法を実践すれば、実務で即戦力として活躍できる知識を確実に身につけられます。ぜひ、今日から「インプット×アウトプット」サイクルを取り入れてみてください。
