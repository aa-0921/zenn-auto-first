## 最初のコードレビューで感じたこと：「論破された気分」

エンジニアとして最初に取り組んだコードレビューは、思いのほか精神的にきつい経験でした。自分が何時間もかけて書いたコードに対し、「この変数名は意味が不明確」「このロジックはエッジケースを考慮していない」「なぜここで else を使っているの？」といった指摘がズラリと並ぶと、ただでさえ疲れている身体と精神に鞭が入ったような感覚を味わいました。

技術的な正確さは確かに重要ですが、コードレビューは単なる「ミスの発掘」ではありません。むしろ、レビューを通じてどうコミュニケーションを図り、どう自分のコードを再構築し、どうチームのコード品質を底上げするか——そこにこそ実務的なスキルが凝縮されています。

この記事では、コードレビューで学んだこと、そして指摘を受けるときにメンタルを保つための実践的なコツを共有します。

## 指摘は「個人への攻撃」ではなく「コードへの改善提案」である

コードレビューで一番つらい瞬間は、「この実装はおかしい」と指摘されたときです。しかし、レビューコメントの主語が「あなた」ではなく「このコード」「この関数」「このアプローチ」であるという事実を思い出してください。

経験を通じて学んだのは、指摘を「コードの改善提案」として受け取る習慣を意識的につくることです。そのための具体的な方法を以下に挙げます。

### 指摘を受けたときの心構え

- **3秒ルールを設ける**: 指摘を読んだ直後に反論したくなる衝動を抑え、3秒間だけ呼吸をしてから返信する。感情が先行すると、防御的・攻撃的な返答になりがちです。
- **「そうなんですね」で始める**: 指摘に対してまず同意・受容の姿勢を示すことで、対話のトーンが協力的なものに切り替わります。「そうなんですね、確かにそのロジックは見落としがちです」と返せば、レビュアーも「この人は成長意欲がある」と感じてくれます。
- **文脈を確認する**: 「なぜそのアプローチを選んだのか」という背景を共有するコメントが届くことがあります。これは批判ではなく、コード全体の設計意図を理解するための質問です。

### 質問の仕方を変えるだけでレビューが前向きになる

コードレビューでの質問は、相手を否定する形にも、協力する形にもなりえます。例えば、以下の2つの質問を比べてみてください。

```
// ❌ 否定的な質問（相手を否定する形）
「なんでこんな複雑なことしてるの？もっとシンプルに書けない？」

// ✅ 建設的な質問（協力する形）
「この部分をもう少しシンプルにできないか、一緒に考えてもらえますか？」
```

同じ内容でも、言い回し一つで受け取る側の心構えが大きく変わります。自分がレビュアーの立場のときにも意識したいポイントです。

## 質問の仕方でレビューの質と関係性が変わる

コードレビューは双方向のコミュニケーションです。指摘を受け取る側だけでなく、質問をする側のスキルも重要になります。

### 効果的な質問のテンプレート

レビューで質問するときは、以下のフォーマットを意識すると明確で礼儀正しいコミュニケーションができます。

```
1. 観察: 「この関数では、引数が null の場合の処理が見当たりません」
2. 影響: 「もし null が渡されると、ランタイムエラーになる可能性があります」
3. 質問: 「Guard句を追加するか、呼び出し側で事前チェックする方が望ましいでしょうか？」
```

この3段構造を使えば、相手に「ただ指摘されている」という感覚ではなく、「一緒に解決策を考えている」という感覚を持たせることができます。

### レビューを通じてチームの設計思想を学ぶ

優れたコードレビューは、チームのアーキテクチャの方針やコーディング規約の背景を学ぶ絶好の機会です。たとえば、「なぜこのプロジェクトではユーティリティ関数を純粋関数にしているのか」「なぜここではクラスではなく関数型のアプローチを選んでいるのか」といった質問を通じて、チームの設計判断の根拠を理解できます。

このような質問は、レビューを受けた側にも「自分の設計を言語化する機会」を与え、結果として自分のスキルアップにも直結します。

## レビューを成長の機会にする具体的な習慣

コードレビューをただの作業として終わらせず、成長のサイクルに組み込むための習慣を紹介します。

### 1. レビュー指摘ノートを残す

レビューで指摘された内容のうち、自分が「なるほど」と感じたものをノートに記録しています。たとえば「この関数の命名は動詞＋名詞のパターンに従っている」「このエラーハンドリングは Result 型を使うべきだった」といった学びです。積み重ねると、自分のコードの改善パターンが見えるようになります。

### 2. 自分のコードを振り返る「レビューミーティング」

マージ後に自分のコードを再度読み返す習慣をつけています。レビュー時には気づけなかった改善点が見つかることがあります。この振り返りは、次の PR を出す際の自分のパフォーマンスに直結します。

### 3. レビューを「メンターとの対話」として捉える

レビュアーは、そのコードを書いた自分よりも多くの経験や視点を持っている可能性があります。レビューを「教えてもらう機会」として捉えると、指摘に対する抵抗感が自然と薄れます。

## まとめ：コードレビューはスキルであり、メンタルコントロールの訓練でもある

コードレビューは、技術的な正確さを保つだけでなく、コミュニケーション能力やメンタルの強さを鍛える場でもあります。指摘を「コードへの改善提案」として受け止め、質問の仕方に気をつけることで、レビューはチーム全体の成長を促進する強力なツールになります。

最初はつらいかもしれませんが、レビューを重ねるうちに「自分のコードが洗練されていく感覚」や「チームとのつながりを感じる喜び」が自然と生まれてきます。次のコードレビューが来たとき、ぜひ「これは成長のチャンスだ」と前向きに捉えてみてください。
