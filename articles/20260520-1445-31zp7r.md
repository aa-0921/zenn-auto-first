## REST API の基本概念

REST（Representational State Transfer）とは、Web システムのアーキテクチャスタイルの一種です。簡単に言うと、**「URL（リソース）に対して、HTTP メソッド（操作）を組み合わせてデータをやり取りする」** という設計原則のことです。

### 1. リソースと URL の設計
REST API では、操作対象となるデータ（ユーザー、記事、商品など）を「リソース」と呼び、それぞれに固有の URL（エンドポイント）を割り当てます。

- ユーザー一覧を取得したい場合 $\rightarrow$ `/users`
- 特定のユーザー（ID: 123）の詳細を取得したい場合 $\rightarrow$ `/users/123`

ポイントは、URL に `getUser` や `deleteUser` といった「動詞」を含めず、**「名詞」で表現する** ことです。操作の内容は、後述する「HTTP メソッド」で指定します。

### 2. HTTP メソッドの使い分け
REST API では、やりたい操作に応じて以下のメソッドを使い分けるのが一般的です。

| メソッド | 役割 | 例 | 備考 |
| :--- | :--- | :--- | :--- |
| `GET` | 取得 | ユーザー情報を取得する | サーバーの状態を変更しない（参照のみ） |
| `POST` | 作成 | 新しいユーザーを登録する | 新しいリソースを作成する |
| `PUT` | 更新（置換） | ユーザー情報を丸ごと更新する | リソース全体を置き換える |
| `PATCH` | 更新（部分） | ユーザーのメールアドレスだけ変更する | リソースの一部だけを書き換える |
| `DELETE` | 削除 | ユーザーアカウントを削除する | リソースを削除する |

### 3. JSON によるデータのやり取り
API のやり取りで最も一般的に使われるデータ形式が **JSON (JavaScript Object Notation)** です。人間にとっても読みやすく、プログラムにとっても解析しやすいため、標準的に採用されています。

**リクエスト例 (POST /users):**
```json
{
  "name": "田中太郎",
  "email": "tanaka@example.com"
}
```

**レスポンス例:**
```json
{
  "id": 123,
  "name": "田中太郎",
  "email": "tanaka@example.com",
  "createdAt": "2023-10-01T10:00:00Z"
}
```

## HTTP ステータスコードの理解

サーバーから返ってくる「ステータスコード」は、リクエストが成功したのか、あるいは何らかのエラーが発生したのかを判別するための重要な信号です。

### よく使うステータスコード一覧

- **2xx: 成功 (Success)**
  - `200 OK`: リクエストが成功し、データが返ってきた。
  - `201 Created`: リソースの作成に成功した（主に POST 後）。
- **4xx: クライアントエラー (Client Error)**
  - `400 Bad Request`: リクエストの内容が正しくない（バリデーションエラーなど）。
  - `401 Unauthorized`: 認証が必要（ログインしていない）。
  - `403 Forbidden`: 権限がない（ログインはしているが、その操作は禁止されている）。
  - `404 Not Found`: 指定した URL のリソースが存在しない。
- **5xx: サーバーエラー (Server Error)**
  - `500 Internal Server Error`: サーバー内部で予期せぬエラーが発生した。

フロントエンド実装では、これらのコードを見て「エラーメッセージを画面に表示する」といった条件分岐を実装します。

## フロントエンドから API を呼び出す実装例

実際に JavaScript を使って API を叩く方法を解説します。現代の Web 開発では、標準の `fetch` API か、ライブラリの `axios` を使うのが主流です。

### 1. `fetch` API を使う方法
`fetch` はブラウザ標準の機能であるため、ライブラリのインストールが不要です。

```javascript
async function getUserData(userId) {
  try {
    const response = await fetch(`https://api.example.com/users/${userId}`);

    // ステータスコードが 200-299 以外の場合はエラーとして扱う
    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }

    const data = await response.json(); // JSONとして解析
    console.log('ユーザーデータ:', data);
  } catch (error) {
    console.error('通信エラーが発生しました:', error);
  }
}

getUserData(123);
```

### 2. `axios` を使う方法
`axios` はサードパーティ製ライブラリで、`fetch` よりも便利な機能が多く、実務での採用率が非常に高いです。

**インストール:**
```bash
npm install axios
```

**実装例 (データの作成 - POST):**
```javascript
import axios from 'axios';

async function createUser() {
  const newUser = {
    name: '佐藤花子',
    email: 'sato@example.com'
  };

  try {
    // axios は第2引数にデータを渡すだけで JSON 形式で送信してくれる
    const response = await axios.post('https://api.example.com/users', newUser);
    
    // response.data に直接解析済みの JSON が入っている
    console.log('作成成功:', response.data);
  } catch (error) {
    // axios はステータスコード 4xx, 5xx を自動的に catch ブロックへ飛ばす
    if (error.response) {
      console.error(`サーバーエラー: ${error.response.status}`);
    } else {
      console.error('ネットワークエラーが発生しました');
    }
  }
}

createUser();
```

### `fetch` と `axios` どっちを使うべき？
結論から言うと、**小規模なプロジェクトや依存関係を増やしたくない場合は `fetch`**、**実務レベルのアプリケーション開発であれば `axios`** をおすすめします。

`axios` が便利な理由は以下の通りです。
- **自動的な JSON 変換**: `response.json()` を呼ぶ手間が省ける。
- **インターセプター**: リクエスト時に自動で認証トークン（JWTなど）をヘッダーに付与できる。
- **タイムアウト設定**: サーバーの応答が遅い場合に通信を切断する設定が簡単。
- **エラーハンドリング**: 4xx や 5xx エラーを自動的に `catch` で捕捉できる。

## まとめ

REST API の基本を理解することは、フロントエンド・バックエンド問わずエンジニアにとって一生モノのスキルになります。

今回のポイントをまとめます：
1. **URL は「名詞」で設計し、操作は「HTTP メソッド」で表現する。**
2. **GET（取得）、POST（作成）、PUT/PATCH（更新）、DELETE（削除）を適切に使い分ける。**
3. **ステータスコードを見て、成功（2xx）かエラー（4xx, 5xx）かを判断する。**
4. **データのやり取りは JSON 形式で行う。**
5. **実装には標準の `fetch` または多機能な `axios` を利用する。**

まずは、無料のテスト用 API（JSONPlaceholder など）を使って、実際に GET や POST を試してみることから始めてみてください。
