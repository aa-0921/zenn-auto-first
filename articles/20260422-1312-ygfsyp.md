## イメージとコンテナの違いを理解する

Docker の最も基本的な概念は「イメージ」と「コンテナ」の区別です。これを理解できないと、Docker の動きを正しく制御できません。簡単に言うと、イメージは「テンプレート」で、コンテナは「実際に動いているプロセス」です。

### イメージとは

イメージ（Image）は、読み取り可能なレイヤーからなるスタックです。つまり、アプリケーションの実行に必要なファイル、ライブラリ、設定、依存関係などが、階層的に固められたものです。イメージはそのままでは動かず、どこかで実行されるまで「休眠」状態です。イメージは不変（immutable）で、作成された内容を変更せずに再利用します。

イメージは Docker Hub などのレジストリから取得したり、Dockerfile をビルドして自作したりできます。イメージの例としては、`nginx:latest`、`node:18-alpine`、`python:3.11-slim` などがあります。

### コンテナとは

コンテナ（Container）は、イメージを実行したインスタンスです。イメージから起動すると、コンテナはそのイメージに基づいて独立したプロセスとして動作します。コンテナは一時的な実行環境であり、アプリケーションが実際に動き、リクエストを受け付ける状態です。

コンテナは以下のような特徴を持っています。
- イメージから作成され、個別のIDを持ちます
- 起動、停止、削除が可能です
- ファイルシステムは一時的（ただし、ボリュームを使えば永続化可能）
- ネットワークインターフェースを持ち、ポートを介して外部と通信可能

イメージとコンテナの関係を整理すると、次のようになります。

```text
イメージ（テンプレート） → コンテナ（実行中の実体）
           ↓
   docker build  →  docker run
```

実際にコマンドで見てみましょう。まず、公式の Nginx イメージからコンテナを起動してみます。

```bash
# イメージをプル（ダウンロード）
docker pull nginx:alpine

# コンテナを起動（8080 ポートをマッピング）
docker run -d -p 8080:80 --name my-nginx nginx:alpine
```

ここで `docker pull` はイメージをダウンロードし、`docker run` はそのイメージからコンテナを起動しています。`-d` はバックグラウンド実行、`-p 8080:80` はホストの 8080 ポートをコンテナの 80 ポートにマッピングしています。この状態でブラウザで `http://localhost:8080` にアクセスすると、Nginx の Welcome ページが表示されます。

イメージは一度ダウンロードすればローカルにキャッシュされるので、2 回目以降は `docker pull` する必要がなくなります。

```bash
# 実行中のコンテナの一覧
docker ps

# 停止中のコンテナも含める
docker ps -a
```

`docker ps` を実行すると、現在実行中のコンテナが表示されます。イメージとコンテナの関係を視覚的に捉えると、次の図のようになります。

```text
   Docker イメージ（階層化されたテンプレート）
           ↓
   Docker コンテナ（実行中のプロセス）
           ↓
   ポートマッピングやボリュームでホストと連携
```

このように、イメージは「静的な設計図」、コンテナは「動的な実行プロセス」と覚えておくと分かりやすいでしょう。

## docker-compose で開発環境を構築する

1 つのコンテナだけでアプリケーションを動かすことは少なく、実際の開発では Web サーバー、データベース、キャッシュ、キューなどの複数のコンテナを連携させることが多いです。ここで登場するのが `docker-compose` です。

docker-compose は、複数のコンテナを一つの `docker-compose.yml` ファイルで定義・管理できるツールです。1 つのコマンドで複数のコンテナを起動、停止、削除できます。

### docker-compose.yml の基本構造

`docker-compose.yml` は YAML 形式で記述し、サービス（コンテナ）、ネットワーク、ボリュームを定義します。最小の構成は次のようになります。

```yaml
version: '3.8'

services:
  web:
    image: nginx:alpine
    ports:
      - "8080:80"
    volumes:
      - ./html:/usr/share/nginx/html
```

この例では、`web` という名前のサービスを定義しています。`image` でイメージを指定し、`ports` でポートをマッピング、`volumes` でホストの `./html` ディレクトリをコンテナの Nginx のドキュメントルートにマウントしています。これにより、ホスト側で HTML ファイルを編集すると、コンテナ内で即座に反映されます。

### docker-compose で開発環境を立ち上げる手順

実際に、Node.js アプリケーションと Redis を docker-compose で立ち上げる例を示します。まず、プロジェクトディレクトリを作成し、以下のような構成にします。

```text
myapp/
├── app/
│   ├── package.json
│   ├── index.js
│   └── Dockerfile
├── docker-compose.yml
```

#### 1. アプリケーション用の Dockerfile を作成

`app/Dockerfile` は次のように記述します。

```dockerfile
# ベースイメージ
FROM node:18-alpine

# 作業ディレクトリを設定
WORKDIR /app

# パッケージをインストール
COPY package.json .
RUN npm install

# アプリケーションコードをコピー
COPY . .

# アプリケーションを起動
CMD ["node", "index.js"]
```

この Dockerfile では、Node.js 18 の Alpine イメージをベースに、アプリケーションコードをコピーして依存パッケージをインストールしています。

#### 2. docker-compose.yml を作成

次に、`docker-compose.yml` を作成します。

```yaml
version: '3.8'

services:
  app:
    build: ./app
    ports:
      - "3000:3000"
    volumes:
      - ./app:/app
    depends_on:
      - redis
    environment:
      - NODE_ENV=development

  redis:
    image: redis:alpine
    ports:
      - "6379:6379"
```

この設定では、`app` サービスは `./app` ディレクトリをビルドしてコンテナを起動します。`volumes` によってホットリロードが可能になり、コードを変更するたびにコンテナ内のファイルも自動的に更新されます。`depends_on` は、`app` コンテナが起動する前に `redis` コンテナが起動するように依存関係を示しています。

#### 3. アプリケーションコードを準備

`app/index.js` を次のように作成します。

```javascript
const express = require('express');
const redis = require('redis');

const app = express();
const client = redis.createClient({ host: 'redis' });

app.get('/', (req, res) => {
  client.get('views', (err, reply) => {
    res.send(`views: ${reply}`);
  });
});

app.listen(3000, () => {
  console.log('App listening on port 3000');
});
```

そして `package.json` を用意します。

```json
{
  "name": "docker-compose-demo",
  "version": "1.0.0",
  "main": "index.js",
  "dependencies": {
    "express": "^4.18.2",
    "redis": "^4.6.10"
  }
}
```

#### 4. コンテナを起動

これで準備が整いました。プロジェクトのルートで次のコマンドを実行します。

```bash
docker-compose up
```

これで、`app` コンテナと `redis` コンテナが同時に起動します。ブラウザで `http://localhost:3000` にアクセスすると、Redis 経由でビュー数が表示されます。

```bash
# バックグラウンドで起動
docker-compose up -d

# ログをリアルタイムで確認
docker-compose logs -f

# コンテナを停止
docker-compose down
```

docker-compose を使うことで、複雑な環境を一コマンドで再現できるようになります。また、チームメンバー間で開発環境を統一できるため、環境依存の問題が大幅に減少します。

## ローカル開発で Docker を使うメリット

これまでのステップで、Docker と docker-compose の基本的な使い方を理解できました。では、なぜローカル開発で Docker を使うべきなのでしょうか？そのメリットをいくつか紹介します。

### 1. 環境の再現性と一貫性

「動いたのに本番では動かない」という問題は、開発者であれば一度は経験したことがあるでしょう。これは主に環境の差異によるものです。ローカルのマシンには古いライブラリや異なる設定が入っていたり、OS の違いで動作が変わったりします。

Docker を使えば、開発環境をコード（Dockerfile と docker-compose.yml）として定義できるため、誰もが同じ環境を再現できます。バージョン管理システムと連携すれば、環境構築の履歴も残せます。

### 2. 依存関係の分離

プロジェクトごとに必要なライブラリやバージョンが異なることはよくあります。例えば、プロジェクト A では Python 3.8、プロジェクト B では Python 3.11 が必要な場合、システムに直接インストールするのは難しいです。

Docker を使えば、各プロジェクトごとに独立したコンテナを構築できるため、依存関係が干渉しません。システムのパッケージマネージャに頼ることなく、必要なバージョンを安全に使用できます。

### 3. 簡単な環境構築と共有

新規メンバーがチームに入ったとき、環境構築に数時間かかることは珍しくありません。Docker を使えば、`docker-compose up` だけで数分で開発環境が立ち上がります。

さらに、`docker-compose.yml` や Dockerfile をリポジトリに含めておけば、新規メンバーはほぼ手作業なしで同じ環境を手に入れられます。教育コストの削減やオープンソースプロジェクトの参加しやすさにもつながります。

### 4. 本番環境との乖離を減らす

開発環境と本番環境で使用するイメージを同一にすれば、移行時の不整合が大幅に減ります。コンテナとしてパッケージングしておけば、開発マシンで動作確認したものがそのまま本番環境でも動くようになります。これにより、デプロイ前のテストがより現実的なものになります。

### 5. 実験的技術の導入が容易

新しいフレームワークやツールを試したいと思ったとき、システムにインストールするのはリスクがあります。しかし、Docker ならば新しいコンテナを作るだけで、既存の環境を汚すことなく試せます。捨てやすい環境で学習や検証ができる点も Docker の大きな魅力です。

## まとめ

この記事では、Docker の基本概念であるイメージとコンテナの違いを解説し、実際にコマンドで動かすことで理解を深めました。次に、`docker-compose` を使った開発環境の構築手順を、Node.js と Redis の例を通して紹介しました。

Docker を使うメリットとして、環境の再現性、依存関係の分離、簡単な環境構築、本番環境との乖離の減少、実験的技術の導入のしやすさなどがあります。開発を始めたばかりの方でも、まずは小さなプロジェクトから Docker を導入してみることをオススメします。

少しずつ慣れていけば、開発の効率化や環境管理が大幅に楽になるはずです。ぜひ、実際に手を動かしながら Docker の世界を体験してみてください。
