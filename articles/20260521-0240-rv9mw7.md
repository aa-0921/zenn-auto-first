## はじめに

私は30歳のとき、営業職からエンジニアに転身しました。コードを書くこともなければ、プログラミングの環境もありませんでした。今から3年が経ち、中級エンジニアとしてチームの中で戦えるようになりました。振り返ってみると、30歳からでもエンジニアとして成長できる理由がいくつかあります。この記事では、転職に至った背景、学習の進め方、現場で感じた手応えと得た知見を、等身大の体験談として書きます。

## 転職に至るまでの背景と決断のポイント

転職を決意したきっかけは、営業の仕事で「同じやり方の繰り返し」に感じたことと、かねてからプログラミングに興味があったことでした。30歳という年齢を気にしていましたが、就活とは違う転職市場では、年齢がネガティブ因子になるだけではないという現実に出会いました。

面接では、過去の業務経験がコミュニケーションや要件定義への理解力として評価されました。逆に、未経験のエンジニアリングスキルは正直に言いました。「こちらで学べる環境ですか」と聞かれることが多く、学習意欲とチームへの貢献方法を語れることが重要でした。

## 学習の進め方：独学と実務を組み合わせた3ヶ月間

転職活動の前から3ヶ月間、1日2〜3時間をコーディング学習に充てました。最初はWeb開発一択で進めましたが、その後の転職先の技術スタックを先読みできるよう、フロントエンドの基礎を重視しました。

```javascript
// 初めて書いたfetchでAPIを呼ぶコード
fetch('https://api.example.com/users')
  .then(response => response.json())
  .then(data => console.log(data))
  .catch(error => console.error('Error:', error));
```

このような小さなコードを1日10行書くことを目標にしました。ドキュメントを読む力も鍛え、MDN Web DocsやGitHubのREADMEを毎日眺める習慣をつけました。大きなプロジェクトを一気に作るのではなく、小さな機能を繰り返し実装して動くものを作ることに集中しました。

転職後のオンボーディングでは、先輩エンジニアに「まずこの機能を実装してみて」と言われ、PRを出すまでの流れを体験しました。そのときのPRレビューの指摘ひとつひとつが、教科書では得られない知見でした。

## 現場で感じた手応えと得た知見

初めてデプロイしたときの達成感は今でも覚えています。本番環境にコードが反映される瞬間、顧客に届くサービスの一部になるという実感がありました。

現場で感じた最大の知見は、**「コードを書くこと」だけがエンジニアの仕事ではない**ということです。要件の整理、仕様の確認、他チームとの調整、バグの原因究明——これらは技術力以上にコミュニケーション力と思考力が求められます。30歳で営業をしていた経験が、ここで活きています。

逆に、技術面では**「ゼロから学ぶなら、まずテストを書く習慣をつける」**と感じました。自分の書いたコードが壊れていないことを保証するテストを書くことで、コードの理解度が格段に上がりました。

```javascript
// Jestを使った単体テストの例
test('ユーザー名が正しく返る', () => {
  const result = getUserName({ firstName: '太郎', lastName: '鈴木' });
  expect(result).toBe('太郎 鈴木');
});
```

## まとめ

30歳からのエンジニア転身は、年齢を気にせず現場で学べる環境を選ぶことが鍵でした。学習は小さなコードの積み重ね、実務ではコミュニケーションとテストの習慣が最大の武器になりました。エンジニアリングスキルは年齢に関係なく身につけられます。まだ迷っている方に伝えたいのは、**「今から始めて間に合う」**ということです。
