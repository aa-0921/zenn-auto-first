## はじめに：Git とは何か、なぜ学ぶ必要があるのか

Git は分散型バージョン管理システムであり、ソフトウェア開発においてコードの変更履歴を追跡し、チームでの共同作業を可能にします。最近では GitHub や GitLab といったプラットフォームで Hosted Git として提供されており、ほぼすべての開発現場で利用されています。

本記事では、Git のクイックスタートとして、リモートリポジトリからのクローン、ブランチ機能の活用、コミットの流れ、リモートへのプッシュ、そして頻発するコンフリクトの解決方法まで、現場で即座に役立つ知識を網羅的に解説します。

## 前提知識：Git の基本概念を整理しよう

Git を効率的に扱うためには、まず基本的な概念を理解することが重要です。

- **リポジトリ（Repository）**：コードや変更履歴が保存される場所。ローカル（自分のマシン）とリモート（共通サーバー）がある。
- **コミット（Commit）**：コードの状態をスナップショットとして保存したもの。変更内容に対するメッセージをつける。
- **ブランチ（Branch）**：メイン線から分岐した開発枝。機能追加やバグ修正を並行して行える。
- **マージ（Merge）**：ブランチをメイン線に統合する操作。
- **コンフリクト（Conflict）**：同じファイルの同じ箇所を異なるコードで変更した場合に発生するトラブル。

これらの概念を頭の中で整理しつつ、次節からは実際のコマンドを使っていきましょう。

## ステップ別！Git コマンドの使い方を丁寧に解説

### 1. リモートリポジトリのクローン：`git clone`

まずは、既存のリモートリポジトリ（ここでは GitHub の例）を自分のローカル環境にダウンロードしましょう。

```bash
git clone https://github.com/user/example-repo.git
```

このコマンドを実行すると、指定したディレクトリが作成され、その中にリポジトリの全履歴がコピーされます。

### 2. 最新状態の取得：`git pull`

チームメンバーが行った変更を自分のローカルに反映させるには、`git pull` を使用します。

```bash
git pull origin main
```

`origin` はリモート名、`main` はブランチ名。つまり、「origin の main ブランチから最新版を取得する」という意味になります。

### 3. 新機能の開発：ブランチの作成と切り替え：`git checkout -b`

機能開発を行う前には、必ず新しいブランチを作成しましょう。これにより、既存のコードに影響を与えずに作業できます。

```bash
git checkout -b feature/new-button
```

`-b` オプションは新規ブランチの作成を意味し、ブランチ名はチームの命名規則に従って付けます（例： `feature/`, `fix/`, `docs/` など）。

### 4. 変更の保存：`git add` と `git commit`

コードを変更した後、ステージングエリアに追加し、コミットします。

```bash
# 変更したファイルをステージングエリアに追加
git add index.html css/style.css

# すべての変更を一度にステージング
git add .

# コミットメッセージを添えてコミット
git commit -m "追加: ホームページのレイアウトを改装"
```

コミットメッセージは分かりやすく、過去の開発者が後から見ても理解しやすいようにしましょう。

### 5. リモートへの反映：`git push`

ローカルでのコミットをリモートリポジトリに反映させるには、`git push` を使用します。

```bash
git push origin feature/new-button
```

初めてプッシュするブランチの場合、`-u` オプションをつけて upstream を設定すると、以後は `git push` のみでOKになります。

```bash
git push -u origin feature/new-button
```

### 6. プルリクエストの作成：GitHub 公式機能

プッシュしたブランチは、GitHub などの Web 上でプルリクエスト（Pull Request）を作成します。ここでコードのレビューを行い、問題がなければメインブランチへマージします。

## コンフリクトの解決：よくあるトラブルとその対処法

チーム開発においてコンフリクトはつきものです。同じファイル、同じ行を異なるコードで編集した場合に発生します。

### コンフリクトの発生例

```bash
git pull origin main
# 自動マージできたけど、conflictsがあります。
# CONFLICT (content): Merge conflict in app.js
# Automatic merge failed; fix conflicts and then commit the result.
```

このとき、競合したファイルを開くと次のような表示になります。

```javascript
<<<<<<< HEAD
console.log("ローカルでの変更内容");
=======
console.log("リモートでの変更内容");
>>>>>>> abc123456789...
```

### コンフリクトの解決手順

1. 競合したファイルをテキストエディタで開き、不必要なマーカー（<<<<<<, ======, >>>>>>>）を削除する。
2. 最終的に残すコードを記述する。
3. 変更をステージングエリアに追加し、コミットする。

```bash
git add app.js
git commit -m "fix: コンフリクト解決"
```

解決がうまくいかない場合は、`git log` でコミット履歴を確認したり、`git diff` で差分をチェックしたりしてください。また、IDE（VS Code など）は競合マーカーを視覚的に表示してくれる機能を持っているので、積極的に活用しましょう。

## まとめ：Git コマンド一覧と次のステップ

ここまでで紹介した主要な Git コマンドをまとめます。

| コマンド | 説明 |
|---------|------|
| `git clone <url>` | リモートリポジトリのダウンロード |
| `git pull` | リモートからの最新取得 |
| `git checkout -b <ブランチ名>` | 新規ブランチの作成と切り替え |
| `git add <ファイル>` | ステージングエリアへの追加 |
| `git commit -m "<メッセージ>"` | コミットの作成 |
| `git push` | リモートへのプッシュ |
| `git merge` | ブランチの統合 |

Git は使えるほどに便利さが広がります。次は `git reset`、`git rebase`、`git stash`などの応用コマンドを学び、チーム開発の効率をさらに上げましょう。日頭に Git の操作を取り入れつつ、徐々にワークフローを定義していくことが、確実なスキルアップの道だということを忘れないでください。
