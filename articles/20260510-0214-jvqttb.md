1. コールバック地獄がどのように発生するかを実例で確認

4. エラーハンドリングのベストプラクティスと、`Promise.all` での同時実行例

## 背景と前提

### コールバック地獄とは？

以下は、ユーザー情報取得 → そのユーザーの投稿一覧取得 → 各投稿に対するコメント数取得という典型的なシナリオです。  
コールバックだけで書くと **入れ子が深くなり**、エラーハンドリングも散在します。

```js
// NG: コールバックだけで書くと読みにくい例
getUser(userId, function (err, user) {
  if (err) return handleError(err);
  getPosts(user.id, function (err, posts) {
    if (err) return handleError(err);
    posts.forEach(function (post) {
      getCommentCount(post.id, function (err, count) {
        if (err) return handleError(err);
        console.log(`${post.title}: ${count} comments`);
      });
    });
  });
});
```

- **可読性が低い**：インデントが深くなり、どこで `return` すべきかが曖昧に。  
- **エラーハンドリングが散らばる**：`handleError` を何度も書く必要がある。  
- **テストが難しい**：関数がコールバックに依存しているため、モックしにくい。

このような状態を **「コールバック地獄」** と呼び、モダンJavaScriptでは `Promise` と `async/await` が主流です。

### 前提知識

- Node.js 12 以上、もしくはモダンブラウザ（ES2017 以降）を対象にしています。  
- 基本的な `function`、`arrow function`、`module` の書き方は知っている前提です。  
- 本記事は **実務での活用** を意識し、テストコードや型定義は割愛しますが、必要に応じてコメントで補足します。

## Promise の基本と then / catch の使い分け

### Promise とは？

`Promise` は「非同期処理の結果（成功 or 失敗）を表すオブジェクト」です。  
3 つの状態を持ちます。

| 状態      | 説明                                 |
|----------|--------------------------------------|
| pending  | 未決定。非同期処理がまだ完了していない |
| fulfilled| 成功。`resolve` が呼ばれた               |
| rejected | 失敗。`reject` が呼ばれた               |

### Promise を自作してみる

まずは `setTimeout` をラップした簡単な `delay` 関数を作ります。

```js
/**
 * 指定ミリ秒待機して resolve する Promise
 * @param {number} ms 待機時間（ミリ秒）
 * @returns {Promise<void>}
 */
function delay(ms) {
  return new Promise((resolve) => {
    setTimeout(resolve, ms);
  });
}
```

この `delay` を使って、**順番に実行** させる例です。

```js
delay(1000)
  .then(() => {
    console.log('1秒経過');
    return delay(500);
  })
  .then(() => {
    console.log('さらに0.5秒経過');
  })
  .catch((err) => {
    // ここに来るのは delay 内で reject したときだけ
    console.error('予期しないエラー:', err);
  });
```

ポイントは:

- `then` は **成功時** に呼ばれ、次の `then` に **新しい Promise** を返す。  
- `catch` は **チェーン全体** のエラーを一括捕捉できる。

### 実務でよく使うパターン：非同期 API 呼び出し

以下は、Node の `fs.promises` を利用したファイル読み込み例です。

```js
const { readFile, writeFile } = require('fs').promises;

/**
 * JSON ファイルを読み込み、整形して別ファイルに書き出す
 * @param {string} srcPath 入力ファイルパス
 * @param {string} destPath 出力ファイルパス
 */
function formatJsonFile(srcPath, destPath) {
  return readFile(srcPath, 'utf8')
    .then((data) => {
      const obj = JSON.parse(data);
      const pretty = JSON.stringify(obj, null, 2);
      return writeFile(destPath, pretty, 'utf8');
    })
    .then(() => {
      console.log('整形完了:', destPath);
    });
}

// エラーハンドリングは呼び出し側で一括管理
formatJsonFile('./data/raw.json', './data/pretty.json')
  .catch((err) => {
    console.error('ファイル処理に失敗:', err);
  });
```

- `readFile` が失敗すれば `catch` に直行。  
- `writeFile` が失敗した場合も同様に `catch` が捕捉。

**メリット**: エラーハンドリングが **一箇所** にまとまり、可読性が向上します。

## async/await で同期的に書く非同期コード

### async / await の基本構文

`async` 関数は常に `Promise` を返します。  
`await` は **`Promise` が解決（fulfilled）または拒否（rejected）** されるまで **関数の実行を一時停止** します。

```js
async function example() {
  console.log('開始');
  await delay(1000); // 1秒待機
  console.log('1秒経過');
}
example();
```

### コールバック地獄の置き換え例

先ほどの「ユーザー → 投稿 → コメント数」シナリオを `async/await` で書き直すと、**見た目が同期コードに近く**なります。

```js
// ダミー API（実務では fetch / axios などを使用）
function getUser(id) {
  return delay(200).then(() => ({ id, name: 'Alice' }));
}
function getPosts(userId) {
  return delay(300).then(() => [
    { id: 1, title: 'Hello World' },
    { id: 2, title: 'Async/Await' },
  ]);
}
function getCommentCount(postId) {
  return delay(150).then(() => Math.floor(Math.random() * 100));
}

// async/await 版
async function showUserPostsWithComments(userId) {
  try {
    const user = await getUser(userId);
    console.log(`User: ${user.name}`);

    const posts = await getPosts(user.id);
    for (const post of posts) {
      const count = await getCommentCount(post.id);
      console.log(`${post.title}: ${count} comments`);
    }
  } catch (err) {
    // どの段階でエラーが起きてもここで捕捉できる
    console.error('データ取得エラー:', err);
  }
}

showUserPostsWithComments(42);
```

#### ポイント

1. **直線的なフロー**：`await` がある行で「待ち」、次の行へ進むのでインデントが浅くなる。  
2. **一元的なエラーハンドリング**：`try…catch` で全体を包むだけで済む。  
3. **`for…of` と組み合わせてシーケンシャル処理**：`await` がループ内にあると、各投稿を順番に処理。

### 並列実行したいときは `Promise.all`

上記コードはコメント数取得を **シーケンシャル** に行っていますが、**並列** にした方が高速です。`Promise.all` と `await` を組み合わせれば実現できます。

```js
async function showUserPostsWithCommentsParallel(userId) {
  try {
    const user = await getUser(userId);
    console.log(`User: ${user.name}`);

    const posts = await getPosts(user.id);

    // コメント数取得だけを並列化
    const commentPromises = posts.map((post) => getCommentCount(post.id));
    const counts = await Promise.all(commentPromises);

    posts.forEach((post, idx) => {
      console.log(`${post.title}: ${counts[idx]} comments`);
    });
  } catch (err) {
    console.error('データ取得エラー:', err);
  }
}

showUserPostsWithCommentsParallel(42);
```

- `Promise.all` は **すべての Promise が成功** したら結果配列を返す。  
- 1 つでも `reject` すると即座に `catch` に飛び、**失敗した場合のリカバリ** は個別に `try…catch` で包むか、`Promise.allSettled` を使う。

### エラーハンドリングのベストプラクティス

| パターン | 実装例 | コメント |
|----------|--------|----------|
| **単一の `try…catch`** | 上記 `showUserPostsWithComments` | ほとんどのケースでシンプルに済む |
| **部分的に分割** | API 呼び出しごとに `try…catch` を入れる | 失敗した箇所だけ特別な処理（リトライ等）したいとき |
| **`Promise.allSettled`** | ```js const results = await Promise.allSettled(promises); ``` | すべての結果を取得したいが、失敗も許容する場合 |

#### 例：部分的リトライ

```js
async function fetchWithRetry(url, retries = 2) {
  for (let i = 0; i <= retries; i++) {
    try {
      const res = await fetch(url);
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      return await res.json();
    } catch (err) {
      if (i === retries) throw err; // 最終失敗は外部へ投げる
      console.warn(`リトライ ${i + 1}/${retries}: ${err.message}`);
    }
  }
}
```

このように **リトライロジック** を `async` 関数に閉じ込めておけば、呼び出し側は `await fetchWithRetry(...)` だけで済む。

## まとめ

- コールバックだけで書くと **可読性・保守性が低下** し、エラーハンドリングが散在する。  
- `Promise` は **状態管理** と **チェーン** に優れ、`then`/`catch` でシンプルに書ける。  
- `async/await` は **同期的に見える** コードを提供し、**一元的なエラーハンドリング** が可能。  
- 並列処理は `Promise.all`、失敗も許容したい場合は `Promise.allSettled` を活用。  
- 実務では **エラーハンドリング戦略**（全体捕捉・部分捕捉・リトライ）をケースバイケースで選択することが重要。

これらのパターンを身につければ、チーム全体のコード品質が向上し、バグの温床となりがちな非同期処理を安全に扱えるようになります。ぜひ本稿のコードを自分のプロジェクトに貼り付けて、動作を確認しながら習得してください。
