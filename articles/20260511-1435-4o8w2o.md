## 背景・前提
React のコンポーネントは「状態」と「副作用」によって描画が決まります。`useState` で状態を保持し、`useEffect` で副作用（API 呼び出し、イベントリスナー登録など）を発火させます。React の再レンダーは「状態が変わったら描画し直す」という原則に従いますが、依存配列を誤ると副作用が期待通りに実行されず、無駄な再レンダーが発生します。

また、リスト描画時に `key` を付け忘れると React は DOM 要素の差分を正しく把握できず、予期せぬ UI の動きや性能低下が起きます。これらは「バグ」と「パフォーマンス問題」の両面を持ち、開発者の頭を悩ませる原因です。

## 本文
### 1. `useEffect` の依存配列ミス
#### 1-1. 典型的なミス例
```tsx
const [count, setCount] = useState(0);
const [data, setData] = useState(null);

useEffect(() => {
  fetch(`/api/items?count=${count}`)
    .then(res => res.json())
    .then(setData);
}, []); // ← ここが問題
```
上記では `count` を依存配列に入れていないため、`count` が変わっても API 呼び出しが再度実行されません。逆に `count` を入れ忘れると無限ループに陥るケースもあります。

#### 1-2. 正しい書き方
```tsx
useEffect(() => {
  fetch(`/api/items?count=${count}`)
    .then(res => res.json())
    .then(setData);
}, [count]); // ← 依存配列に count を入れる
```
依存配列には副作用内で参照している「状態・props・関数」を全て列挙します。関数が頻繁に再生成される場合は `useCallback` でメモ化し、依存性を減らすと良いでしょう。

#### 1-3. チェックリスト
| チェック項目 | 具体例 | 期待される動作 |
|--------------|--------|----------------|
| 依存配列に全て列挙 | `useEffect(() => {...}, [foo, bar])` | 変更時に再実行 |
| 未使用変数は除外 | `useEffect(() => {...}, [])` で `foo` を使わない | 無駄な再実行を防止 |
| `useCallback` で関数をメモ化 | `const onClick = useCallback(() => {...}, [dep])` | 依存配列が安定 |

### 2. 無駄な再レンダーの原因と対策
#### 2-1. 例：親コンポーネントの状態更新で子が再レンダー
```tsx
function Parent() {
  const [value, setValue] = useState(0);
  return <Child value={value} />;
}
```
`Child` が `React.memo` で包まれていないと、`value` が変わるたびに `Child` も再レンダーします。親が頻繁に更新されると、ツリー全体の描画コストが増大します。

#### 2-2. `React.memo` の活用
```tsx
const Child = React.memo(function Child({ value }) {
  return <div>{value}</div>;
});
```
`React.memo` は props が変更されない限り再レンダーをスキップします。複雑な子コンポーネントやリスト要素に効果的です。

#### 2-3. コード例：リストアイテムの再レンダー抑制
```tsx
const ListItem = React.memo(({ item }) => {
  return <li>{item.text}</li>;
});

function List({ items }) {
  return (
    <ul>
      {items.map(item => (
        <ListItem key={item.id} item={item} />
      ))}
    </ul>
  );
}
```
`key` を正しく設定し、`React.memo` を併用すると、アイテムが追加・削除されたときだけ対象の要素が再レンダーされます。

### 3. キーを付け忘れたリスト描画の落とし穴
#### 3-1. 典型例
```tsx
<ul>
  {items.map(item => (
    <li>{item.text}</li>
  ))}
</ul>
```
キーが無いと、React は要素の差分を正しく判定できず、古い DOM を再利用してしまうため、クリックイベントがバインドされた状態が消えてしまうケースがあります。

#### 3-2. 正しい書き方
```tsx
<ul>
  {items.map(item => (
    <li key={item.id}>{item.text}</li>
  ))}
</ul>
```
`item.id` のように一意に決まる値をキーにします。もし `id` が無い場合は `index` を使うこともありますが、配列の並びが変わる可能性がある場合は避けるべきです。

#### 3-3. キーのベストプラクティス
- **一意性を保証**：同じリスト内で重複しないようにする。
- **安定性**：レンダリングごとに変わらない値を使用。`index` は並び順が変わらない限り OK ですが、要素の入れ替えがある場合は重複リスクがある。
- **パフォーマンス**：キーは整数・文字列で高速に比較できるものを選択。

## まとめ
React で頻繁に遭遇するエラーは「依存配列の誤設定」「無駄な再レンダー」「キーの付け忘れ」の三つに集約されます。対処法は以下のように整理できます。

1. **`useEffect` の依存配列** – 参照した値を漏れなく列挙し、必要なら `useCallback` でメモ化。
2. **再レンダー抑制** – 子コンポーネントに `React.memo` を使用し、`key` を正しく付与。
3. **キーの設定** – 一意で安定した値を使用し、並び順が変わる場合は `index` ではなくユニーク ID を利用。

実務では「バグの再現性を落とす」ために、コンポーネント単位でユニットテストを行い、`React Testing Library` で依存配列の動作を検証すると安心です。また、開発時に `eslint-plugin-react-hooks` を導入し、依存配列のミスを静的に検出することもおすすめです。

これらのコツを押さえておけば、React の開発における「不思議な挙動」や「パフォーマンス低下」を早期に発見し、修正できるようになります。ぜひ日々の開発に取り入れてください。
