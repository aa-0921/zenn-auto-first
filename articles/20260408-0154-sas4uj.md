## Gitの基本概念

Gitを理解する上で押さえておくべき基本概念は以下の通りです。

### リポジトリ（リポジトリ）
プロジェクトのファイルと変更履歴を保存する場所です。ローカルリポジトリ（自分のPC）とリモートリポジトリ（GitHubやGitLabなど）の2種類があります。

### コミット
ファイルの変更点を記録することです。コミット時にメッセージを残すことで、なぜその変更をしたのかを後から追跡できます。

### ブランチ
開発の流れを分岐させる機能です。メインの開発ライン（通常はmainやmasterブランチ）から分岐させ、新しい機能開発や修正を行います。

### ステージングエリア
コミットする前に、変更を一時的に置いておく場所です。このエリアに追加した変更だけがコミットされます。

## 実践的なGitフロー：clone/pull/branch/commit/push

### 1. リポジトリのクローン（clone）
まずはリモートリポジトリを自分のPCに持ってきます。

```bash
git clone <リポジトリのURL>
```

例：
```bash
git clone https://github.com/username/repository.git
```

### 2. 変更のプル（pull）
チームメンバーが行った最新の変更を自分のローカルリポジトリに取り込みます。

```bash
git pull origin <ブランチ名>
```

例：
```bash
git pull origin main
```

### 3. ブランチの作成と切り替え（branch）
新しい機能開発や修正用にブランチを作成します。

```bash
# ブランチを作成
git checkout -b <新しいブランチ名>

# または2ステップで
git branch <新しいブランチ名>
git checkout <新しいブランチ名>
```

例：
```bash
git checkout -b feature/user-login
```

### 4. 変更のコミット（commit）
ファイルの変更をステージングエージョに追加し、コミットします。

```bash
# 変更をステージングエリアに追加
git add <ファイル名>  # 特定のファイル
git add .            # すべての変更

# コミットを実行
git commit -m "<コミットメッセージ>"
```

例：
```bash
git add src/components/LoginButton.js
git commit -m "Add login button component"
```

### 5. リモートへのプッシュ（push）
コミットした変更をリモートリポジトリに送信します。

```bash
git push origin <ブランチ名>
```

例：
```bash
git push origin feature/user-login
```

## コンフリクトの対処法

### コンフリクトが発生する原因
コンンフリクトは、2人以上のエンジニアが同じファイルの同じ部分を変更した場合に発生します。特に、ブランチをマージする際によく起こります。

### コンフリクトの解決手順

1. **コンフリクトの検知**
   ```bash
   git pull origin <ブランチ名>
   ```
   コンフリクトが発生すると、以下のようなメッセージが表示されます。
   ```
   error: Your local changes to the following files would be overwritten by merge:
    <ファイル名>
   Please, commit your changes or stash them before you can merge.
   ```

2. **コンフリクトのあるファイルを確認**
   ```bash
   git status
   ```
   状態を確認し、コンフリクトのあるファイルを特定します。

3. **ファイルの編集**
   コンフリクトが発生したファイルを開き、以下のようなマーカーを編集します。
   ```
   <<<<<<< HEAD
   自分の変更内容
   =======
   他の変更内容
   >>>>>>> <ブランチ名>
   ```
   必要な部分を残し、不要な部分を削除します。

4. **解決したファイルのステージング**
   ```bash
   git add <ファイル名>
   ```

5. **コミット**
   ```bash
   git commit -m "Resolve conflict in <ファイル名>"
   ```

### コンフリクトを防ぐためのベストプラクティス
- 頻繁にプルして最新の状態を保つ
- 小さな単位でコミットする
- ブランチを長く開きすぎない
- チームでコミットメッセージのルールを統一する

## さらに実力をつけるためのTips

### 便利なGitコマンド
- **変更の確認**:
  ```bash
  git diff  # ステージングエリアとの差分
  git diff --staged  # 直前のコミットとの差分
  ```

- **コミット履歴の確認**:
  ```bash
  git log --oneline  # 1行表示
  git log --graph --oneline --decorate  # グラフ表示
  ```

- **変更の取り消し**:
  ```bash
  git checkout -- <ファイル名>  # ワーキングディレクトリの変更を取り消し
  git reset HEAD <ファイル名>  # ステージングエリアから取り消し
  ```

### コミットメッセージの書き方
良いコミットメッセージは、なぜその変更をしたのかを明確に伝えます。以下のような形式が推奨されます。

```
<種別>: <簡潔な変更内容>

<詳細な説明>

- 変更の理由
- 関連するチケット番号
```

種別には以下のようなものがあります：
- fix: バグ修正
- feat: 新機能
- docs: ドキュメント更新
- style: コード整形
- refactor: リファクタリング
- test: テスト追加

## まとめ

この記事では、Gitの基本から現場でよく使われるコマンドまでを解説しました。特にclone/pull/branch/commit/pushの基本的な流れと、コンフリクト時の対処法は、日常の開発作業で必須の知識です。

Gitは最初は複雑に感じるかもしれませんが、実際に手を動かして使うことで次第に慣れてきます。まずは基本的なフローをマスターし、その上でより高度な機能（リベース、チェリーピックなど）に挑戦してみてください。

Gitは単なるツールではなく、チームで協力して開発を行うための文化そのものです。良いコミットメッセージを心がけ、チームメンバーと協力して快適な開発生活を送りましょう。
