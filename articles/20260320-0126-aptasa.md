## はじめにエンジニアとして働く中で、コードレビューは避けて通れないプロセスです。しかし、指摘を受けるたびに「自分はダメなんだ」と落ち込んだり、質問しづらくて黙ってしまうことはありませんか？本記事では、私が実際に経験したレビューのエピソードを交えながら、指摘の受け止め方、効果的な質問の仕方、そしてレビューを単なる指摘ではなく「成長の機会」に変える具体的なコツを紹介します。対象は駆け出し〜中級エンジニアで、実務ですぐに役立つ知見を重視します。

## コードレビューの基本姿勢

### レビューは「人」ではなく「コード」に焦点を当てる

レビューでのコメントは、書いた人の能力を評価するものではありません。たとえば次のようなコメントはNGです。

```text
このロジックは初心者レベルだね。もっと勉強しよう。
```

代わりに、コード自体に焦点を当てた建設的なフィードバックを心がけましょう。

```text
ここでのネストが深くなると可読性が落ちます。早期リターンでフラットにするとどうでしょうか？
```

この姿勢を保つだけでも、指摘を「自分への攻撃」ではなく「コード改善の提案」として受け止めやすくなります。

### 「なぜ？」を意識する

指摘されたポイントに対して「なぜこのように指摘されたのか」を自分で考える習慣をつけると、単なる修正作業から抜け出し、根本的な理解が深まります。例えば、`if (user.isActive && user.role === 'admin')` という条件式が指摘された場合、以下のように考えます。

- なぜこの条件が問題か？ → 権限チェックが分散しているため、後で追加する役割ごとに修正漏れが発生しやすい。
- どう改善すればよいか？ → 権限判定を関数に閉じ込め、`hasPermission(user, 'admin')` などに抽出する。

このように「なぜ」を追求すると、指摘された箇所以外にも応用できる知識が得られます。

## 指摘への受け止め方と質問のテクニック

### 1. 感情を一旦保留し、事実に落とし込む

指摘を見たときにまず頭に浮かぶのは「怒り」や「恥」かもしれません。そんなときは、以下のステップで感情を切り離します。

1. コメントをそのままコピーしてメモ帳に貼り付ける。
2. 感情語（例えば「馬鹿にされている」「嫌だ」）を削除し、事実だけを抜き出す。
3. 事実に基づいて「何を修正すればよいか」を箇条書きにする。

この作業により、感情に左右されずに行動計画を立てやすくなります。

### 2. 質問は「具体的かつオープン」に

レビューで疑問点があるときは、次のような質問の形が効果的です。

- **悪い例**：「これって大丈夫？」  
  → 答えが「はい／いいえ」に収まりやすく、議論が広がりにくい。

- **良い例**：「この関数の戻り値が `null` になるケースを想定していますが、ここで `null` チェックを入れるべきでしょうか？それとも呼び出し側で保証すべきでしょうか？」  
  → 目的と仮定を示し、複数の解決策を促すオープンな質問。

また、質問する前に自分で調べた結果を簡潔に添えると、レビュアーの負担が減り、スムーズに答えてもらえます。

```text
自分で調べたところ、同様のケースではライブラリXの`safeGet`関数が使われていることが多いです。ここでも同様のラッパーを作るのはありでしょうか？
```

### 3. フィードバックを「感謝」で締めくくる

指摘を受けた後は、必ず一言感謝を伝えましょう。たとえば：

> 「ご指摘ありがとうございます。早速修正してプッシュします。」

この一言が、レビューの雰囲気を良好に保ち、次回も積極的にフィードバックを求めやすくなります。

## レビューを成長の機会に変える具体的なステップ### ステップ1: レビュー前の自己チェックリストを作る

レビューを依頼する前に、自分でチェックできる項目をリスト化しておくと、指摘されるポイントを減らせます。例：

- [ ] 関数の責務が単一か？
- [ ] マジックナンバーやハードコードされた文字列はないか？
- [ ] エラーケースは適切にハンドリングしているか？
- [ ] テストは網羅しているか？（特に境界値）
- [ ] コメントはなぜそう書いたのかを説明しているか？

このチェックリストを毎回使うことで、指摘される「初歩的なミス」が減り、レビューの焦点がより高度な設計やアルゴリズムに移ります。

### ステップ2: 指摘を「タスク」に変換する

レビューで受けた各コメントを、タスク管理ツール（例：GitHub Issues、Jira、あるいは単なるTodoリスト）に変換します。例えば：

```
[ ] ネストが深い箇所を早期リターンでリファクタリング #12
[ ] 権限チェックをhasPermission関数に抽出 #13[ ] nullチェックの要否をチームで議論 #14
```

タスクとして可視化すると、達成感を得やすくなり、レビュー後のやる気低下を防げます。

### ステップ3: フィードバックループを閉じる修正が終わったら、再度レビューを依頼し、変更点が指摘通りに解決されたことを確認してもらいます。その際、以下のようにコメントを添えると良いでしょう。

> 「先ほどの指摘に基づき、ネストを早期リターンでリファクタリングしました。 diffはこちらです。他に気になる点があれば教えてください。」

この「指摘 → 修正 → フィードバック」のサイクルを繰り返すことで、少しずつ自分の盲点が見えてきますし、レビュアーからも「この人は真剣に取り組んでいる」という印象を得られます。

### ステップ4: 振り返りとナレッジ共有

週に一度、自分が受けたレビューのコメントをテーマ別にまとめます（例：命名、エラーハンドリング、テスト戦略）。まとめた内容は社内のWikiやブログに共有すると、自分の学びがチーム全体の知識ベースになり、さらに自身の理解が深まります。

## まとめ

コードレビューは単なるバグ取りではなく、エンジニアとしての思考法やコミュニケーションスキルを鍛える絶好の場です。指摘を感情的に受け止めず、「なぜ？」を追求し、具体的かつオープンな質問を心がけることで、レビューから得られる学びは格段に増えます。さらに、自己チェックリストやタスク化、フィードバックループの閉じ方、そして振り返りとナレッジ共有を習慣化すれば、レビューはストレスの源ではなく、着実な成長の機会になるでしょう。
