## 導入

ソフトウェア開発において、コードレビューは品質を向上させるだけでなく、エンジニア自身のスキルアップにも直結する重要なプロセスです。しかし、指摘を受ける側になると「自分はダメだ」と思い込んでしまいがちで、メンタルに悪影響を及ぼすケースも少なくありません。本記事では、私が実際に経験したレビューでの失敗と成功談を交えながら、指摘の受け止め方、質問の仕方、そしてレビューを成長の機会に変えるコツを具体的に紹介します。

## 背景・前提

私が所属しているチームでは、プルリクエスト（PR）をベースにしたレビューが標準です。レビューの流れは以下の通りです。

1. 開発者が機能修正や新機能のコードをブランチにプッシュし、PRを作成
2. レビュー担当者が変更点を確認し、コメントを付ける
3. 開発者がコメントに対して修正または回答を行う
4. 全ての議論が解決したらマージ

このプロセスにおいて、レビュー担当者からの指摘は「バグ防止」「可読性向上」「設計の改善」など多岐にわたります。一方で、受け取る側は「自分の実力が試されている」と感じやすく、特にネガティブなフィードバックが続くと自己効力感が低下しがちです。

## 指摘の受け止め方

### 1. ファクトと感情を分ける

レビューコメントには事実（バグ、スタイル違反）と感想（「ここはもっとシンプルにできるかも」）が混在しています。まずは事実のみを抽出し、感情的な反応は後で整理する習慣をつけると、冷静に対応できます。

```text
# コメント例
- ここでは `null` チェックが漏れています（事実）
- もっと関数名をわかりやすくした方がいいと思います（感想）
```

事実に対してはすぐに修正し、感想については「ご指摘ありがとうございます。別案としてこういう名前はどうでしょうか？」と返答すると、建設的な議論につながります。

### 2. 「失敗」ではなく「学び」と捉える

指摘を受けた瞬間に「自分はダメだ」と思わず、「今回の指摘から何を学べるか」に焦点を当てます。例えば、nullチェック漏れが指摘されたら、同様のパターンをプロジェクト全体で検索し、統一的なガード節を導入するきっかけにします。

```bash
# プロジェクト内で同様のパターンを検索
git grep -n "if.*== null" src/
```

このように、指摘を改善のトリガーとして活用すると、ネガティブな感情をポジティブな行動に変換しやすくなります。

## 質問の仕方

### 1. 具体的に、かつ簡潔に

質問が抽象的だと回答も曖昧になり、議論が長引きます。何を知りたいのか、どの部分に不安があるのかを一文で表現すると良いでしょう。

```
# 良くない例
この部分、何かおかしいですか？

# 良い例
ここでの変数 `userId` がゼロになるケースを想定していますが、実際にゼロが渡されるシナリオは存在しますか？
```

### 2. コードスニペットを添える

質問したいロジックが複雑な場合は、関連するコードを引用すると相手がすぐに状況を把握できます。

```java
// 質問したい箇所
if (user.getRole() == Role.ADMIN) {
    // ここでの権限チェックが十分か不安
}
```

これにより、レビュー担当者は「ここが不安なんだな」と即座に理解し、的確なフィードバックを返しやすくなります。

### 3. 感謝の気持ちを忘れない質問の終わりに「ご指摘ありがとうございます」や「助かります」と一言添えるだけで、相手も回答しやすくなり、チーム全心理的安全性が向上します。

## レビューを成長の機会にするコツ

### 1. レビュー後の振り返りを習慣化

PRがマージされたら、自分が受けたコメントを一覧にし、それぞれに対して「何を学んだか」「次回どのように改善するか」をメモします。たとえば、以下のようなテンプレートを使うと便利です。

| コメント内容 | 学んだこと | 次回の改善点 |
|--------------|------------|--------------|
| nullチェック漏れ | ガード節の重要性 | 同様のチェックはユーティリティメソッドに集約 |
| 変数名が不明瞭 | 名前付けのルール確認 | チームの命名規約ドキュメントを再読 |

この振り返りを週次で行うと、弱点が見える化され、計画的なスキル向上が可能になります。

### 2. メンターやペアプログラミングを活用

レビューで指摘された点について、すぐに同僚に相談できる環境があると、不安を抱え込まずに解決できます。たとえば、ペアプログラミングで実装中にレビュー担当者と一緒にコードを書くと、リアルタイムでフィードバックを受けながら学べます。

### 3. 小さな成功体験を積み重ねる

大きな指摘に落ち込むより、小さな指摘でも「直せた」という成功体験を積むことで自己効力感を高めます。例えば、インデントの修正や不要なインポートの削除など、すぐにできる改善から始めると、レビューへの抵抗感が減ります。

## まとめ

コードレビューは単なる品質チェックではなく、エンジニアとしての成長を加速させるフィードバックループです。指摘を事実と感情に分けて受け止め、具体的かつ感謝を伴う質問を心がけ、レビュー後に振り返りを行うことで、メンタルを保ちながらスキルアップを実現できます。

皆さんもぜひ、次のPRからこれらのポイントを試してみてください。レビューが恐ろしい場面ではなく、自分を伸ばすチャンスに変わるはずです。
