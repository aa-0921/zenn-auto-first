## はじめに

Rails は「設定より規約 (Convention over Configuration)」が大きな魅力ですが、規約に慣れていないと「なぜこんなエラーが出るの？」と戸惑う場面が多々あります。  
この記事では、**初心者から中級者が実務でよくハマる 3 つのエラー**（ルーティングエラー、マイグレーションエラー、N+1 クエリ）を取り上げ、  
* エラーが起きる典型的なシナリオ  
* デバッグ時に確認すべきポイント  
* 実際に動かせるコード例と対処法  

を具体例とともに解説します。実際に手を動かしながら読むことで、エラーの原因を「目視で把握」できるようになることを目指します。

## 1. ルーティングエラー – `No route matches` 系

### 1-1. 典型的なシナリオ

```bash
GET /users/123/edit
# => RoutingError (No route matches [GET] "/users/123/edit")
```

#### 何が起きているか？

* `config/routes.rb` に期待するパスが定義されていない、または HTTP メソッドが合っていない  
* `resources :users` と書いたつもりが、`only: [:index, :show]` で制限してしまっている  

### 1-2. デバッグの手順

| 手順 | 確認内容 | 補足 |
|------|----------|------|
| 1 | `rails routes | grep users` で実際に登録されているパスを一覧表示 | 期待通りの HTTP メソッド・URL が出ているか |
| 2 | `rake routes` の出力を確認し、`edit_user_path` が存在するか | `edit_user_path` が無ければ `resources` のオプションを見直す |
| 3 | ビューやコントローラで呼び出しているヘルパーが正しいか | `link_to 'Edit', edit_user_path(@user)` のようにシンボルが合っているか |
| 4 | `config/routes.rb` の記述順序 | 後に書いた `match '*path' => redirect('/')` などが先にマッチしてしまうことがある |

### 1-3. 具体例と修正

**問題のコード**

```ruby
# config/routes.rb
Rails.application.routes.draw do
  resources :users, only: [:index, :show]   # ← edit が抜けている
  root 'home#index'
end
```

**対処法**

```ruby
# 修正後
Rails.application.routes.draw do
  resources :users                         # 全 CRUD を有効化
  # もしくは edit のみ追加したい場合
  # resources :users, only: [:index, :show, :edit, :update]
  root 'home#index'
end
```

**デバッグ実行例**

```bash
$ rails routes | grep edit_user
edit_user GET /users/:id/edit(.:format) users#edit
```

これで `GET /users/123/edit` が正しくルーティングされます。

## 2. マイグレーションエラー – `ActiveRecord::StatementInvalid` 系

### 2-1. 典型的なシナリオ

```bash
rails db:migrate
# => ActiveRecord::StatementInvalid: SQLite3::SQLException: no such column: users.age: INSERT INTO "users" ("name", "age") VALUES (?, ?)
```

#### 何が起きているか？

* 既存テーブルに対して **カラム追加** のマイグレーションを実行したが、**以前のマイグレーションが未実行** か **ローカルと本番でスキーマがズレている**  
* `change_column` でデータ型を変えたが、既存データが変換できない  

### 2-2. デバッグの手順

1. **マイグレーションの実行順序を確認**  
   ```bash
   $ rails db:migrate:status
   ```
   `up` になっていないファイルがないか。

2. **スキーマの実態を確認**  
   ```bash
   $ rails dbconsole
   > .schema users
   ```
   期待通りのカラムがあるか。

3. **ロールバックして再実行**（安全な環境でのみ）  
   ```bash
   $ rails db:rollback STEP=1
   $ rails db:migrate
   ```

4. **バージョン管理でマイグレーションファイルが競合していないか**  
   `git diff db/migrate` で差分を確認。

### 2-3. 具体例と修正

**問題のコード**

```ruby
# db/migrate/20240301120000_add_age_to_users.rb
class AddAgeToUsers < ActiveRecord::Migration[7.0]
  def change
    add_column :users, :age, :integer, null: false, default: 0
  end
end
```

*しかし、`users` テーブルはすでに `age` カラムが `string` 型で存在している。*

**対処法①：カラムの型変更を正しく書く**

```ruby
# db/migrate/20240301121000_change_age_type_in_users.rb
class ChangeAgeTypeInUsers < ActiveRecord::Migration[7.0]
  def up
    # 文字列から整数へ安全に変換
    change_column :users, :age, :integer, using: 'age::integer', null: false, default: 0
  end

  def down
    change_column :users, :age, :string, null: false, default: '0'
  end
end
```

**対処法②：ローカルでスキーマがずれている場合はリセット**

```bash
$ rails db:drop db:create db:migrate
```

> **注意**: 本番環境で実行するとデータが失われます。必ずバックアップを取るか、データを保持したままマイグレーションを作成してください。

## 3. N+1 クエリ問題 – パフォーマンスの落とし穴

### 3-1. 典型的なシナリオ

```ruby
# app/controllers/articles_controller.rb
def index
  @articles = Article.all
end

# app/views/articles/index.html.erb
<% @articles.each do |article| %>
  <p><%= article.title %> - <%= article.author.name %></p>
<% end %>
```

**発生するクエリ（Rails ログ）**

```
SELECT "articles".* FROM "articles"
SELECT "users".* FROM "users" WHERE "users"."id" = 1 LIMIT 1
SELECT "users".* FROM "users" WHERE "users"."id" = 2 LIMIT 1
...
```

`articles` が 10 件でも、`users` への SELECT が 10 回（N+1）走ります。

### 3-2. デバッグの手順

| 手順 | 操作 | 目的 |
|------|------|------|
| 1 | `rails server` 起動後、開発環境で `config.log_level = :debug` を確認 | SQL が全て出力される |
| 2 | ブラウザで該当ページを表示し、サーバログに同じテーブルへの SELECT が多数出ていないか | N+1 の有無を視覚的に把握 |
| 3 | `bullet` gem を導入して自動検知 | 開発中に即座に警告が出る |
| 4 | `includes` / `preload` / `eager_load` の使い分けを確認 | 正しい eager loading 手法を選択 |

### 3-3. 具体例と修正

**問題のコード（再掲）**

```ruby
@articles = Article.all
```

**対処法①：`includes` を使う**（最もシンプル）

```ruby
@articles = Article.includes(:author).all
```

**対処法②：`eager_load` が必要なケース**（JOIN で取得したいとき）

```ruby
@articles = Article.eager_load(:author).all
```

**対処法③：`select` で必要なカラムだけ取得**（余計なデータ転送を削減）

```ruby
@articles = Article
             .includes(:author)
             .select(:id, :title, :author_id)
```

**Bullet の設定例**

```ruby
# Gemfile
group :development do
  gem 'bullet'
end

# config/environments/development.rb
Rails.application.configure do
  config.after_initialize do
    Bullet.enable = true
    Bullet.alert = true          # JavaScript アラートで通知
    Bullet.bullet_logger = true  # log/bullet.log に出力
    Bullet.console = true        # ブラウザのコンソールに出力
  end
end
```

Bullet が検知したら次のようにコンソールに表示されます。

```
N+1 Query detected
  Article => [:author]
  Add eager loading directives for :author.
```

この警告を見たら、すぐに `includes` へ置き換えるだけでパフォーマンスが大幅に改善します。

## まとめ

| エラー | 発生しやすい原因 | デバッグのキーポイント | 推奨解決策 |
|--------|------------------|------------------------|------------|
| `No route matches` | ルーティング定義漏れ、メソッド不一致 | `rails routes` で実際のパスを確認 | `resources` のオプション見直し、記述順序の調整 |
| `ActiveRecord::StatementInvalid`（マイグレーション） | スキーマのズレ、型変更時のデータ不整合 | `db:migrate:status`、`dbconsole` でスキーマ確認 | 正しい `change_column` の `using` オプション、必要ならロールバック・リセット |
| N+1 クエリ | 関連モデルを逐次取得 | ログで同一テーブルの SELECT が多数出ていないか、`bullet` で自動検知 | `includes` / `eager_load` で eager loading、必要に応じて `select` で絞り込み |

Rails のエラーは「**何が足りないか**」を把握すれば解決への道筋が見えてきます。  
- ルーティングは **ルートテーブル** を眺めて確認  
- マイグレーションは **スキーマと実行順序** を意識  
- N+1 は **SQL ログと Bullet** で検知  

本記事の手順を実際のプロジェクトで試すことで、エラーに対する「感覚」が身につき、実務でのトラブルシューティングが格段に速くなります。ぜひ、日々の開発フローに取り入れてみてください。
