## マイグレーションでデータ損失や失敗を防ぐテクニック  
マイグレーションはデータベース構造を変更する重要な手段ですが、初心者はカラムの重複や実行失敗につながるミスをよく犯します。  

```ruby  
# 例：誤ったマイグレーション  
class AddAgeColumn < ActiveRecord::Migration[6.1]  
  def change  
    add_column :users, :age, :integer  
    remove_column :users, :birthday  # このカラムが存在しない場合エラー  
  end  
end  
```  

### 失敗の原因と対処  
- **カラムが存在しない場合のエラー**  
  `remove_column`で削除しようとしたカラムが存在しないとマイグレーションが失敗します。  
- **スキーマの不整合**  
  既存のデータに適応しないカラム追加（例：デフォルト値の設定を忘れた場合）  

### 安全なマイグレーション設計  
1. **` reversible?`オプションを使う**  
   ```ruby  
   add_column :users, :age, :integer, reversible: true  # ロールバック可能  
   ```  
2. **データマイグレーションを分割**  
   ```ruby  
   # 1. カラム追加  
   add_column :users, :age, :integer  
   # 2. デフォルト値の設定（別マイグレーション）  
   User.update_all(age: 0)  
   ```  

## N+1クエリ問題を`includes`で最適化する  
ActiveRecordでは、関連テーブルに複数のクエリを発行する「N+1クエリ問題」がパフォーマンス低下の原因になります。初心者は「1つのユーザーを取得するために100件のクエリが走る」などの現象にハマりやすいです。  

```ruby  
# N+1クエリの例  
100.times do |i|  
  user = User.find(i+1)  
  user.posts  # 101件のクエリが発行される（100件のUser + 100件のPost）  
end  
```  

### `includes`で解決する方法  
```ruby  
# 1回のクエリで関連データを取得  
users = User.includes(:posts).all  # 2件のクエリ（UserとPost）  
```  

### 実際の測定方法  
1. **`ActiveRecord::LogSubscriber`でクエリを確認**  
   ```ruby  
   # config/initializers/active_record_log_subscriber.rb  
   ActiveRecord::Base.log_subscriber = ActiveRecord::LogSubscriber.new  
   ```  
2. **`bullet`GemでN+1を可視化**  
   `bullet`をGemfileに追加し、`rbenv`で有効化します。
