## 背景・前提
### Git の基本概念
| 用語 | 意味 |
|------|------|
| **リポジトリ** | コードと履歴を保持する場所。ローカルとリモートがある。 |
| **コミット** | 変更を保存するスナップショット。 |
| **ブランチ** | 独立した作業ライン。 |
| **マージ** | ブランチ間で変更を統合。 |
| **コンフリクト** | 同じ行を複数人が変更した場合に発生。 |

### 事前に必要なもの
- Git がインストールされた環境（Windows/Mac/Linux）  
- コマンドライン／ターミナルに慣れていること  
- GitHub/GitLab などのホスティングサービスにアカウントがあること

## 本文
### 1. リポジトリを取得（clone）
```bash
# GitHub のレポジトリをローカルにコピー
git clone https://github.com/your-org/sample-repo.git
cd sample-repo
```
`clone` すると、リモートの `origin` が自動で設定され、`master/main` がチェックアウトされます。

### 2. 最新状態を取得（pull）
```bash
# リモートの変更をローカルブランチへマージ
git pull origin main
```
- `git fetch` + `git merge` のショートカット  
- **注意**: 変更が多い場合は `git pull --rebase` でリベースすることで履歴を直線化できます。

### 3. 新しいブランチで作業
```bash
# ブランチ名は機能や課題に合わせて付ける
git checkout -b feature/login-form
```
ブランチを切るタイミングは **pull** 後がベストです。  
リモートにまだ存在しないローカルブランチは `git push -u origin feature/login-form` でリモートに作成します。

### 4. コミット
```bash
git add .
git commit -m "Add login form validation"
```
- `-m` で簡潔なメッセージを付ける  
- **ベストプラクティス**: 1コミットは1機能/バグ修正単位にまとめる

### 5. コミットをリモートへ（push）
```bash
git push origin feature/login-form
```
これでリモートに自分のブランチが反映されます。  
他のメンバーはこのブランチをベースに作業できます。

### 6. コンフリクトが発生したら
#### 6-1. コンフリクトの確認
```bash
git status
```
`both modified` と表示されるファイルがコンフリクト箇所です。

#### 6-2. コンフリクトの解消
```bash
git mergetool
```
- デフォルトは `vimdiff`/`meld` など。  
- 手動でファイルを編集し、`<<<<<<< HEAD` と `>>>>>>> branch-name` の区切りを削除します。

#### 6-3. 解消後のコミット
```bash
git add <conflicted-file>
git commit -m "Resolve merge conflict in <file>"
```
- ここでのコミットは **merge commit** になるため、履歴が分岐しやすい。  
- 必要に応じて `git commit --no-edit` で自動メッセージを使うことも。

#### 6-4. 再度 push
```bash
git push origin feature/login-form
```
コンフリクト解消後はリモートへ再度 push します。

### 7. Pull Request（Merge Request）でレビューを
```bash
# GitHub なら PR を作る
# GitLab なら MR を作る
```
- コミットメッセージに `<ISSUE>` を含めると自動で issue へリンクします。  
- コードレビューを受けることで品質が向上します。

## まとめ
| ステップ | 目的 | コマンド例 |
|----------|------|-----------|
| clone | リポジトリ取得 | `git clone <url>` |
| pull | 最新取得 | `git pull origin main` |
| branch | 作業分離 | `git checkout -b <name>` |
| commit | 変更保存 | `git commit -m ""` |
| push | 変更共有 | `git push origin <name>` |
| コンフリクト解消 | マージ失敗時 | `git mergetool` + `git commit` |

- **ポイント**  
  1. **pull** は必ずブランチを切る前に実行。  
  2. コミットは小さく、意味のある単位で。  
  3. コンフリクトは早めに解消し、レビューで防止を図る。  

これらを習慣化すれば、Git を使ったチーム開発がスムーズになります。  
ぜひ日々の開発に取り入れて、コード管理の「怖さ」を「楽しく」変えてみてください。
