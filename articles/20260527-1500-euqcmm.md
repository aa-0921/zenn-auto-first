## 1. REST API とは何か？：通信のルールを知る

REST API（Representational State Transfer API）とは、一言で言えば「Webの標準的な仕組み（HTTP）に乗っ取って、リソース（データ）を操作するための設計ルール」のことです。

APIを理解する上で欠かせないのが、**「HTTPメソッド」**と**「ステータスコード」**の概念です。

### 1.1 HTTPメソッド：何をするのか（動詞）

リソースに対して「何を行いたいか」を指示するのがHTTPメソッドです。よく使われるのは以下の4つ（CRUD操作に対応）です。

| メソッド | 操作 (CRUD) | 内容 | 例 |
| :--- | :--- | :--- | :--- |
| **GET** | Read (取得) | リソースの情報を取得する | ユーザー一覧を取得する |
| **POST** | Create (作成) | 新しいリソースを作成する | 新規ユーザーを登録する |
| **PUT** | Update (更新) | リソースを丸ごと置き換える | ユーザー情報を全て更新する |
| **PATCH** | Update (部分更新) | リソースの一部を更新する | ユーザーのメールアドレスだけ変える |
| **DELETE** | Delete (削除) | リソースを削除する | ユーザーを退会させる |

実務では、データの「一部だけ」を更新したい場合に `PUT` ではなく `PATCH` を使うといった使い分けが重要になります。

### 1.2 ステータスコード：結果はどうだったか（返事）

サーバーは処理が終わると、結果を「ステータスコード」という3桁の数字で返してきます。これを見るだけで、通信が成功したのか、エラーが起きたのかが瞬時に判断できます。

- **2xx (成功)**
  - `200 OK`: リクエストが成功した。
  - `201 Created`: リソースの作成に成功した（POST時によく使われる）。
- **4xx (クライアント側のエラー)**
  - `400 Bad Request`: リクエストの内容が不正（パラメータ不足など）。
  - `401 Unauthorized`: 認証が必要（ログインしていない）。
  - `403 Forbidden`: 権限がない（ログインはしているが、操作する権利がない）。
  - `404 Not Found`: 指定したリソースが存在しない。
- **5xx (サーバー側のエラー)**
  - `500 Internal Server Error`: サーバー側で予期せぬエラーが発生した。

フロントエンドエンジニアとしては、**「4xx系が返ってきたらフロント側のリクエストに問題がある可能性が高い」「5xx系が返ってきたらバックエンド側に調査を依頼する」**という切り分けができるようになると、デバッグが非常にスムーズになります。

## 2. JSON によるデータのやり取り

APIとの通信では、データの形式として **JSON (JavaScript Object Notation)** がデファクトスタンダードとして使われます。

JSONは、JavaScriptのオブジェクトに似たテキスト形式で、人間にもマシンにも読みやすいのが特徴です。

**リクエスト例（POSTでユーザーを作成する場合のBody）:**
```json
{
  "name": "田中 太郎",
  "email": "tanaka@example.com",
  "age": 25
}
```

**レスポンス例（GETでユーザーを取得した場合）:**
```json
{
  "id": 123,
  "name": "田中 太郎",
  "email": "tanaka@example.com",
  "created_at": "2023-10-27T10:00:00Z"
}
```

フロントエンドでAPIを叩く際は、このJSONを「文字列として送り」、受け取った「文字列をオブジェクトに変換（パース）して」扱うことになります。

## 3. フロントエンドからの実装例：fetch と axios

それでは、実際にJavaScriptを使ってAPIを呼び出してみましょう。標準的な `fetch` APIと、デファクトスタンダードであるライブラリ `axios` の2パターンを紹介します。

### 3.1 標準の `fetch` API を使う

`fetch` はブラウザに標準搭載されているため、ライブラリのインストールなしですぐに使えます。

```javascript
// データの取得 (GET)
async function getUser() {
  try {
    const response = await fetch('https://api.example.com/users/1');
    
    // ステータスコードが200-299以外の場合にエラーを投げる
    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }

    const data = await response.json(); // JSONをオブジェクトに変換
    console.log('取得成功:', data);
  } catch (error) {
    console.error('エラーが発生しました:', error);
  }
}

// データの作成 (POST)
async function createUser() {
  const newUser = { name: '佐藤', email: 'sato@example.com' };

  try {
    const response = await fetch('https://api.example.com/users', {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json', // 送信データがJSONであることを伝える
      },
      body: JSON.stringify(newUser), // オブジェクトをJSON文字列に変換
    });

    if (!response.ok) {
      throw new Error(`HTTP error! status: ${response.status}`);
    }

    const result = await response.json();
    console.log('作成成功:', result);
  } catch (error) {
    console.error('エラーが発生しました:', error);
  }
}
```

**`fetch` の注意点:**
`fetch` は、ネットワークエラー（通信断絶など）以外では `catch` に飛びません。例えば `404 Not Found` が返ってきても、それは「通信自体は成功（レスポンスが届いた）」とみなされるため、自分で `if (!response.ok)` のようにチェックする必要があります。

### 3.2 `axios` を使う

実務では `axios` がよく使われます。理由は、`fetch` よりも記述が簡潔で、便利な機能が豊富だからです。

```javascript
import axios from 'axios';

async function getUserWithAxios() {
  try {
    // axiosは自動でJSONをパースしてくれる
    const response = await axios.get('https://api.example.com/users/1');
    
    // response.data に直接オブジェクトが入っている
    console.log('取得成功:', response.data);
  } catch (error) {
    // 4xxや5xxのエラーは自動的にcatchブロックへ飛ぶ
    if (error.response) {
      // サーバーからレスポンスが返ってきたが、ステータスが2xx以外の場合
      console.error('ステータスコード:', error.response.status);
      console.error('エラー内容:', error.response.data);
    } else {
      // リクエスト送信自体に失敗した場合（ネットワークエラーなど）
      console.error('リクエストエラー:', error.message);
    }
  }
}
```

**`axios` のメリット:**
1. **自動JSON変換**: `JSON.stringify()` や `.json()` を書く必要がなく、`response.data` でデータにアクセスできます。
2. **エラーハンドリング**: ステータスコードが 2xx 以外の場合、自動的に `catch` に飛ばしてくれるため、エラー処理が書きやすいです。
3. **タイムアウト設定**: 通信がタイムアウトした時の設定などが容易です。

## まとめ

この記事では、REST APIの基本から実装方法までを解説してきました。重要なポイントをおさらいしましょう。

1. **HTTPメソッドを使い分ける**: `GET` (取得), `POST` (作成), `PUT/PATCH` (更新), `DELETE` (削除) の役割を理解する。
2. **ステータスコードで状況を判断する**: `2xx` は成功、`4xx` はクライアント側のミス、`5xx` はサーバー側の問題。
3. **JSON形式でやり取りする**: データの受け渡しは、構造化されたテキスト形式であるJSONが主流。
4. **実装ツールを選ぶ**: 標準の `fetch` は手軽だが、実務で複雑なエラーハンドリングや設定が必要なら `axios` が強力。

APIの仕組みを理解することは、フロントエンドエンジニアとして「データがどこから来て、どこでエラーが起きているのか」を把握するための第一歩です。まずは、ブラウザのデベロッパーツール（Networkタブ）を開いて、自分が作っているアプリがどんなリクエストを送り、どんなレスポンスを受け取っているか観察することから始めてみてください！
