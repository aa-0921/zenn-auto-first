## 背景と前提条件
- **useEffect の依存配列**：副作用（API 呼び出し、タイマー、DOM 操作など）を宣言する際に、依存する変数や函数を配列に列挙します。依存が足りないと古い値が参照され、意図しない結果が出ます。逆に不要な要素を列挙すると無駄な再実行が起き、パフォーマンスが低下します。  
- **再レンダー**：state または prop が変わるとコンポーネントは再レンダーされます。予期しない再レンダーは UI のフリッカーや不必要な副作用のトリガーになることがあります。  
- **キーの付け忘れ**：リストを描画する際に `key` プロパスを指定しないと、React が要素のアイデンティティを正しく管理できず、状態のズレやアニメーションの乱れが生じます。

## 代表的なエラーと対処法

### 1. useEffect の依存配列が不完全 or 過剰
#### 1‑1. 典型的なミス例
```jsx
function Counter() {
  const [count, setCount] = useState(0);
  const [data, setData] = useState([]);

  useEffect(() => {
    // データ取得は count に依存すると期待されるが、count が変わらないとfetch がonceだけ走る
    fetch(`/api/item/${count}`)
      .then(r => r.json())
      .then(setData);
  }, []); // ← 依存配列が空 → count の変化を検知できない
}
```
**問題点**  
- `count` が変わっても `useEffect` は一度だけ実行され、最新の `count` が反映されない。  
- 依存配列が空のままでは、意図しない stale data が永遠に残る。

#### 1‑2. 正しい対処法
```jsx
useEffect(() => {
  fetch(`/api/item/${count}`)
    .then(r => r.json())
    .then(setData);
}, [count]); // count を明示的に依存させる
```
**ポイント**  
- 依存配列に `count` を入れると、`count` が変わるたびに副作用が再実行され最新の値が取得できる。  
- ただし、`count` が頻繁に変わる場合はパフォーマンス上の懸念があるため、`useCallback` で安定した関数を作る、または `useMemo` で計算結果をキャッシュする手法を併用すると良い。

### 2. 無駄な再レンダー
#### 2‑1. 典型的なミス例
```jsx
function Parent() {
  const [flag, setFlag] = useState(false);
  const [value, setValue] = useState(0);

  // Parent が再レンダーされると、Child も再レンダーされる
  return (
    <div>
      <Child value={value} />
      <button onClick={() => setFlag(!flag)}>toggle</button>
    </div>
  );
}

function Child({ value }) {
  // value が変わらなくても再レンダーされる
  console.log('Child rendered');
  return <div>{value}</div>;
}
```
**問題点**  
- `Parent` の state が変わると `Child` も再レンダーされ、子コンポーネント内のロジックや副作用が余計に走る。

#### 2‑2. 正しい対処法
```jsx
import React, { memo } from 'react';

const Child = memo(({ value }) => {
  console.log('Child rendered'); // ここで memo が効いていることを確認できる
  return <div>{value}</div>;
});
```
- `React.memo` を使うことで props が同一であれば再レンダーを防げる。  
- さらに `useCallback` や `useMemo` で子コンポーネントに渡す関数やオブジェクトを安定させると、余計な再レンダーを減らせる。

### 3. キーの付け忘れ
#### 3‑1. 典型的なミス例
```jsx
function List({ items }) {
  return (
    <ul>
      {items.map((item) => (
        <li>{item.name}</li> // key が無いため React が要素の再利用を誤る
      ))}
    </ul>
  );
}
```
**問題点**  
- `items` が入れ替わったり新規追加されたときに、React は同じ DOM ノードを再利用し、状態やイベントハンドラが正しく保たれなくなる。

#### 3‑2. 正しい対処法
```jsx
function List({ items }) {
  return (
    <ul>
      {items.map((item) => (
        <li key={item.id}>{item.name}</li> // ユニークな key を指定
      ))}
    </ul>
  );
}
```
- `key` には **安定したユニークな値**（ID やハッシュ）を使用し、リストが再描画された際に React が要素の識別子として正しく機能するようにする。

## まとめ
- **useEffect の依存配列**は「何が変わるべきか」を明示的に列挙し、過剰な依存は `useCallback` や `useMemo` で最適化する。  
- **再レンダー**はコンポーネントの構造上避けられないが、`React.memo` や `useCallback` で子コンポーネントの再レンダー回数を最小化できる。  
- **リストの key**は必ずユニークで安定した値を設定し、リストの再描画時の状態ズレを防止する。  

これらの対処法を実践し、React DevTools や ESLint ルール（例：`react-hooks/exhaustive-deps`）を併用すれば、現場で起こる典型的なエラーを未然に防げます。エラーに悩まされる時間を減らし、より快適に React アプリを構築していきましょう。
