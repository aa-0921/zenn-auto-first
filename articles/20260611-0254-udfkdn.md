## 背景・前提

- **React 18** を前提に、**関数コンポーネント**と**Hooks**を使用します。
- **TypeScript**は必須ではありませんが、型安全の観点から型推論の活用例も紹介します。
- 本記事では「エラーの発生原因」→「対処法」→「チェックリスト」の流れで進めます。

### 1. useEffect の依存配列が不適切なとき

`useEffect` の第2引数に渡す依存配列は、**副作用が再実行されるべき**変数だけを列挙する必要があります。漏れや余計な値を入れると、意図しない再実行や infinite loop が発生します。

#### 典型的なミス例

```tsx
import React, { useState, useEffect } from 'react';

const Counter = () => {
  const [count, setCount] = useState(0);
  const [message, setMessage] = useState('');

  useEffect(() => {
    const timer = setInterval(() => {
      setCount(prev => prev + 1);
    }, 1000);
    return () => clearInterval(timer);
  }, []); // ← 依存配列が空だが、setCount は依存しない

  return <div>{count}</div>;
};
```

上記では `setCount` を呼び出す副作用に **count** を依存させていませんが、実際に副作用が **count** に依存します。結果として **count** が変わっても `useEffect` は再実行されず、意図しない挙動になります。

#### 正しい書き方

```tsx
useEffect(() => {
  const timer = setInterval(() => {
    setCount(prev => prev + 1);
  }, 1000);
  return () => clearInterval(timer);
}, []); // ここは空でも OK。setCount は React が内部で管理しているため依存不要
```

**ポイント**: `setState` 関数自体は変更されないので依存に入れなくて大丈夫。副作用が **外部値**（props, context, state）に依存する場合はそれを列挙する。

### 2. 再レンダーの原因と無駄なレンダリングを防ぐテクニック

#### 典型的な無駄再レンダー

```tsx
const Parent = () => {
  const [value, setValue] = useState(0);
  const obj = { val: value };

  return <Child obj={obj} />;
};

const Child = React.memo(({ obj }) => {
  return <div>{obj.val}</div>;
});
```

`Parent` で `obj` を毎回新しく作成しているため、`React.memo` が効果を発揮せず、毎回 `Child` が再レンダーします。

#### 解決策

1. **useMemo**でオブジェクトをメモ化

```tsx
const obj = useMemo(() => ({ val: value }), [value]);
```

2. **useCallback**で関数をメモ化（関数が渡る場合）

3. **PureComponent**（クラス）や**React.memo**（関数）を併用する

#### チェックリスト

- 依存配列に余計な変数が入っていないか
- オブジェクト・関数を毎回再生成していないか
- `React.memo`/`PureComponent` の効果を確認したか

### 3. リスト項目にキーが付け忘れたときの挙動

```tsx
const List = ({ items }) => (
  <ul>
    {items.map(item => (
      <li>{item.name}</li> // ← key がない
    ))}
  </ul>
);
```

キーがないと、React は **DOM の差分比較**で不具合が起き、パフォーマンス低下やバグ（入力フォームの値が保持されない）につながります。

#### 正しい書き方

```tsx
{items.map(item => (
  <li key={item.id}>{item.name}</li>
))}
```

- **id** がユニークで不変であれば最適
- 数値配列の場合は `index` は避ける（リストが入れ替わると不具合）

#### キーのベストプラクティス

1. **ユニークで不変な ID**を必ず使用
2. 生成済み ID が無い時は **UUID**を入手
3. 配列の順序が変わる可能性がある場合は **index** を使わない

## 本文（ステップ・コード例）

### ステップ 1: useEffect の依存配列を正しく定義する

```tsx
const DataFetcher = ({ url }) => {
  const [data, setData] = useState(null);

  useEffect(() => {
    let isMounted = true;
    fetch(url).then(res => res.json()).then(json => {
      if (isMounted) setData(json);
    });
    return () => { isMounted = false; };
  }, [url]); // url の変更にだけ再実行

  return data ? <pre>{JSON.stringify(data, null, 2)}</pre> : <span>Loading...</span>;
};
```

- `url` が変わると再フェッチ
- `isMounted` でアンマウント時の状態更新を防止

### ステップ 2: 無駄な再レンダーを防ぐメモ化

```tsx
const CounterList = ({ counters }) => {
  const incrementAll = useCallback(() => {
    counters.forEach(c => c.setCount(c.count + 1));
  }, [counters]);

  return (
    <div>
      <button onClick={incrementAll}>Increment All</button>
      {counters.map(c => (
        <Counter key={c.id} counter={c} />
      ))}
    </div>
  );
};
```

- `incrementAll` は `counters` が変わらない限り再生成されない
- `Counter` は `React.memo` でラップして再レンダーを抑制

### ステップ 3: キーを忘れずに付ける

```tsx
const TodoList = ({ todos }) => {
  return (
    <ul>
      {todos.map(todo => (
        <TodoItem
          key={todo.id} // ← 重要
          todo={todo}
        />
      ))}
    </ul>
  );
};
```

- `key` は **React の内部**でリストアイテムを識別するために必須
- `key` が重複すると `<li>` が置き換えられ、入力値が消える

## まとめ

- **useEffect の依存配列**は副作用が実際に依存する値だけを列挙し、無駄な再実行を防ぐ。
- **再レンダーの抑制**は `useMemo`・`useCallback`・`React.memo` を適切に使い、オブジェクト・関数の再生成を防ぐ。
- **リストにキーを付け忘れると** DOM 差分比較が誤動作し、パフォーマンス低下やバグを招く。必ずユニークで不変な ID をキーにする。

これらを意識して開発することで、React アプリケーションの安定性とパフォーマンスを向上させることができます。次回は「Context API と useReducer の組み合わせ」で状態管理の高度なパターンに挑戦してみましょう。
